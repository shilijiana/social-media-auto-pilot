```
M-09 位置说明:
  上游: M-07（素材匹配与调用模块）→ 传入视觉素材（底图候选材料）
        M-08（文案差异化生成模块）→ 传入标题文案（用于封面文字叠加）
  下游: M-11（内容精加工与定稿模块）→ 使用生成的封面图进行精加工
        M-12（合规性自检模块）→ 对封面图进行合规检测（水印、人脸、商标）
  自学习: M-25/M-26 收集封面图的用户反馈数据（点击率、停留时长），优化视觉偏好库
```

### 1.2 核心职责

根据内容主题和标题文案，生成多平台差异化封面图。核心流程包括：通过图片生成模型生成 3 张基础底图候选，选取最佳底图后叠加标题文字、账号 Logo、装饰元素，按 4 个平台尺寸规格（抖音 9:16、小红书 3:4、B站 16:9、X/Twitter 1:1）裁剪输出。应用视觉偏好库（色彩方案、字体排版、装饰风格）中的用户偏好，确保封面风格与账号历史调性一致。输出初稿版本和压缩版本，写入 `cover_images` 表，文件存储至 `data/files/outputs/images/`。

### 1.3 触发时机

- **自动触发**: M-08（文案差异化生成模块）完成后，由流水线引擎自动调用。传入内容草稿中的标题和话题信息
- **人工触发**: 运营人员在内容管理后台对某个内容草稿手动点击「生成封面」或对已有封面点击「重新生成」
- **批量触发**: 支持一次性为多个内容草稿批量生成封面图（如批量为本周所有内容生成封面）

### 2.3 数据来源

| 字段 | 来源 |
|------|------|
| `task_id` | 流水线引擎生成 |
| `content_draft_id` | M-08（文案差异化生成模块）输出的 draft_id |
| `title` | M-08 输出的 content.title（选取视觉表现力最强的平台标题） |
| `topic_info` | M-08 透传（来自 topics 表） |
| `target_platforms` | 分发计划配置 `config/distribution_plan.yaml`，可运行时覆盖 |
| `visual_preferences` | M-26（偏好提炼与规则生成模块）管理的视觉偏好库，通过 `preference_api.get_visual(topic_tags, topic_type)` 查询 |
| `generation_options` | 配置文件 `config/cover_generation.yaml` 加载，支持运行时覆盖 |
| `branding_assets` | 配置文件 `config/branding.yaml`，包含账号 Logo 文件路径和水印模板 |
| `material_images` | M-07 输出的 matched_materials 中 material_type='image' 的素材（可选，作为底图风格参考） |

## 4. 核心处理逻辑

### 4.1 处理步骤

```
Step 1: 输入校验与资源加载
  - 做什么: 校验输入格式，加载品牌资源和视觉偏好
  - 怎么判断:
      - 使用 jsonschema 库校验输入格式
      - 读取 config/branding.yaml 加载 Logo、水印模板文件路径
      - 校验品牌资源文件存在: logo_path, watermark_template_path
      - 如果 visual_preferences 缺失或 confidence < 0.7，调用 M-26 API 获取默认视觉偏好
      - 校验 title 非空且长度 <= 100 字符
  - 输出: 品牌资源路径（branding_assets）、视觉偏好配置（visual_preferences）

Step 2: 底图 Prompt 构建
  - 做什么: 根据话题信息和视觉偏好构建图片生成 Prompt
  - 怎么判断:
      - 使用 Jinja2 模板 prompts/cover_image/base_image_{topic_type}.j2 渲染 Prompt
      - 注入上下文:
        - topic_tags: 话题标签 → 影响图片主题元素
        - color_scheme: 色彩方案 → 影响图片色调
        - decoration.style: 装饰风格 → 影响视觉元素
        - composition: 构图要求（留出文字叠加区域）
      - 生成 3 个变体 Prompt（使用不同的种子词/风格描述，增加多样性）
      - 每个 Prompt 包含: 主题描述 + 风格引导 + 构图要求 + 质量要求（4K, high quality）
  - 输出: 3 个底图生成 Prompt（base_prompts）

Step 3: 并行生成底图候选
  - 做什么: 并行调用图片生成 API 生成 3 张底图候选
  - 怎么判断:
      - 对 3 个 Prompt 使用 asyncio.gather() 并行生成
      - 模型: generation_options.image_model（默认 gemini-2.5-flash-image）
      - 输出尺寸: 按最大平台尺寸生成（1920x1920），后续按平台裁剪
      - 每张底图最多重试 max_retries 次
      - 重试时微调 Prompt（添加 "different composition", "alternative style"）
  - 输出: 3 张底图文件（base_images）

Step 4: 底图质量评估与选择
  - 做什么: 评估 3 张底图的质量，选择最优的一张
  - 怎么判断:
      - 调用视觉评估 LLM（gemini-2.5-flash，支持图片输入）对每张底图评分
      - 评估维度:
        - aesthetic（美学评分）: 视觉吸引力、色彩搭配
        - relevance（主题相关性）: 与话题的关联度
        - composition（构图评分）: 画面平衡、视觉层次
        - text_space（文字区域可用性）: 中央/关键区域是否有足够空间叠加文字
        - overall（综合评分）: 加权平均
      - 权重: aesthetic 30%, relevance 25%, composition 20%, text_space 25%
      - 选择 overall 最高且 >= quality_threshold 的底图
      - 如果所有底图 overall < quality_threshold，选择最高分但添加 warning
  - 输出: 选中的底图（selected_base）及评分详情

Step 5: 标题文字叠加
  - 做什么: 在选中底图上叠加标题文字
  - 怎么判断:
      - 使用 Pillow 库进行图片合成
      - 根据 visual_preferences.typography 设置:
        - 字体: title_font（需确认字体文件存在，否则降级为系统默认字体）
        - 字号: 根据平台尺寸自适应（title_size_range.min 到 max 之间按比例计算）
        - 样式: title_style（纯色/描边/渐变/阴影/霓虹）
        - 颜色: text_color（来自 color_scheme）
        - 对齐: text_alignment
      - 文字位置: 根据 platform.preferred_layout 决定:
        - title_overlay: 居中叠加
        - split_screen: 左三分之一区域
        - minimal_text: 底部小字
        - full_image: 居中叠加
      - 文字换行: 根据底图宽度和字号自动计算每行最大字符数
      - 对于中文长标题，优先使用双行/三行布局
      - 添加文字阴影/描边确保可读性（对比度检查: 文字颜色 vs 背景色亮度差 >= 0.5）
  - 输出: 叠加文字后的图片（text_overlaid_images，按平台分）

Step 6: 账号品牌标识叠加
  - 做什么: 在封面上添加账号 Logo 和水印
  - 怎么判断:
      - Logo: 从 branding_assets 加载，按 branding.logo_position 定位
      - Logo 缩放: 按 branding.logo_size_ratio × 封面面积 计算像素尺寸
      - Logo 透明度: branding.logo_opacity
      - 水印: 如果 branding.watermark_enabled=true，叠加半透明水印模板
      - 水印使用 tile 模式或单次居中，由 config/branding.yaml 控制
      - 使用 Pillow 的 alpha_composite 进行透明叠加
  - 输出: 添加品牌标识后的图片（branded_images）

Step 7: 装饰元素添加
  - 做什么: 根据视觉偏好添加装饰性视觉元素
  - 怎么判断:
      - 根据 decoration.style 选择装饰模板:
        - geometric: 几何线条/形状
        - organic: 有机曲线/流体形状
        - gradient_overlay: 渐变遮罩（顶部或底部渐变）
        - grid: 网格线
        - tech_lines: 科技感线条/电路板纹路
        - particles: 粒子效果
        - minimal: 极简，不添加额外装饰
      - 装饰密度由 decoration.density 控制
      - 装饰颜色从 color_scheme.accent 和 color_scheme.secondary 选取
      - 装饰层置于文字层之下、底图之上
      - 圆角处理: 如果 decoration.corner_style='rounded'，添加圆角遮罩
  - 输出: 最终完整封面图（final_covers）

Step 8: 平台尺寸裁剪
  - 做什么: 将封面图按各平台尺寸规格裁剪
  - 怎么判断:
      - 平台尺寸映射:
        - douyin: 1080×1920（9:16，竖屏）
        - xiaohongshu: 1080×1440（3:4，竖屏）
        - bilibili: 1920×1080（16:9，横屏）
        - x: 1200×1200（1:1，正方形）
      - 裁剪策略: 从原始底图（1920×1920）按各平台比例居中裁剪
        - 竖屏（9:16, 3:4）: 取底图中央竖条，上下各留边
        - 横屏（16:9）: 取底图中央横条，左右各留边
        - 正方形（1:1）: 取底图中央正方形区域
      - 裁剪后重新调整文字位置和大小（如需要）
      - 使用 Lanczos 重采样保证裁剪质量
  - 输出: 各平台裁剪后的封面图（platform_covers）

Step 9: 压缩输出与缩略图生成
  - 做什么: 生成初稿 PNG 和压缩版 JPEG/WebP，生成缩略图
  - 怎么判断:
      - 初稿版: 保存为 PNG 格式（无损），路径: data/files/outputs/images/draft/
      - 压缩版: 按 compress_quality 保存为 JPEG（默认 85%）和/或 WebP，路径: compressed/
      - 缩略图: 生成 300×自适应 的缩略图，路径: thumbnails/
      - 文件命名: {cover_id}_{platform}_{version}.{ext}
      - 计算文件大小写入 metadata
  - 输出: 文件写入磁盘 + 写入 cover_images 表

Step 10: 结果汇总与写入
  - 做什么: 汇总各平台结果，写入数据库
  - 怎么判断:
      - 为每个平台生成唯一 cover_id（格式: COV-YYYY-NNNNNN）
      - 写入 cover_images 表
      - 写入 image_gen_logs 表（图片生成日志）
      - 写入 module_outputs 表
      - 计算整体 status:
        - 全部平台成功 → completed
        - 部分平台成功 → partial
        - 全部平台失败 → failed
  - 输出: 最终的 Output JSON
```

## 6. 错误处理策略

| 错误类型 | 处理方式 | 重试策略 | 告警级别 |
|---------|---------|---------|---------|
| 输入 JSON Schema 校验失败 | 返回 status=failed，错误详情写入 `errors` 表，终止执行 | 不重试 | ERROR |
| title 为空或超长 | 返回 status=failed，错误信息: "标题文字无效" | 不重试 | ERROR |
| 品牌资源文件缺失（Logo/水印） | 降级: 跳过品牌标识叠加，添加 warning: "品牌资源缺失，未添加 Logo" | 不重试 | WARN |
| 字体文件缺失 | 降级: 使用系统默认中文字体（NotoSansSC），添加 warning | 不重试 | WARN |
| 图片生成 API 返回 429 | 等待 Retry-After 头指定时间后重试 | 3 次，间隔递增 | WARN |
| 图片生成 API 返回 500/503 | 切换降级策略：使用纯色背景 + 装饰元素 + 文字合成（不依赖 AI 底图） | 主模型 2 次 → 降级方案 | WARN |
| 单张底图生成失败 | 继续生成其他候选，如果成功候选数 >= 1 继续流程；如果 0 张成功，启用降级方案 | 每张候选 3 次 | WARN |
| 底图质量评分全部 < quality_threshold | 选择最高分底图，添加 warning: "所有底图质量评分 < 阈值，建议人工审核" | 不重试 | WARN |
| 文字叠加后对比度不足 | 自动调整: 增加文字描边宽度，或切换文字颜色（黑→白 / 白→黑） | 自动修复 | INFO |
| 图片裁剪/处理内存不足 | 降低处理分辨率（从 1920 降至 1080），重试 | 1 次降分辨率重试 | WARN |
| 文件系统写入失败（磁盘满/权限） | 记录错误日志，通知运维；已生成的文件保留 | 不重试 | CRITICAL |
| 数据库写入失败 | 暂存结果到本地 JSON 文件 `data/m09_backup_{task_id}.json` | 5 次，间隔 2s | CRITICAL |
| 装饰素材目录为空 | 跳过装饰步骤，添加 warning: "装饰素材缺失，输出无装饰封面" | 不重试 | INFO |

### 降级策略

当图片生成 API 完全不可用时，启用以下降级方案：

1. **纯色背景 + 装饰 + 文字**: 使用 `color_scheme.background` 作为纯色背景，叠加装饰元素和标题文字。质量较低但保证有封面可用，标记 `source = "template_fallback"`
2. **素材库底图**: 从 M-07 的 matched_materials 中选取 relevance_score 最高的图片作为底图（需确认有版权使用授权）
3. **跳过本模块**: 在 `pipeline_config` 中设置 `cover_generator.skip_on_failure = true`，下游模块使用空白封面占位，待人工制作

| `decoration` | 使用的装饰风格 | 生成时写入（metadata JSONB） |
| `base_image_prompt` | 底图生成 Prompt | 生成时写入（image_gen_logs） |
| `quality_scores` | 底图质量评分（aesthetic/relevance/composition/text_space/overall） | 生成时写入（image_gen_logs） |
| `selected_candidate_index` | 被选中的底图候选编号（1-3） | 生成时写入 |
| `final_cover_used` | 精加工后最终使用的封面版本 | M-11 完成后回写 |
| `edit_details` | 人工修改详情（调整了哪些元素） | M-11 完成后回写 |
| `click_through_rate` | 封面点击率（发布后） | M-22 获取发布后 7 天数据 |
| `avg_view_duration` | 平均观看时长（视频封面关联） | M-22 获取发布后 7 天数据 |
| `ab_test_variant` | A/B 测试分组（如果生成了多个封面变体） | M-11 选择变体后回写 |

### 7.2 偏好库调用

- **调取偏好类型**: 视觉偏好（color_scheme、typography、decoration、branding、platform_layout）
- **置信度阈值**: >= 70%
- **无偏好时默认行为**: 使用 `config/cover_generation.yaml` 中的 `default_visual_by_topic_type` 映射，根据话题类型选择默认视觉方案

### 7.3 阶段演进行为

| 阶段 | 行为差异 |
|------|---------|
| P0 (人工主导) | 封面由设计师手动制作。模块仅负责将设计稿按平台尺寸裁剪和压缩，存储到 `cover_images` 表。图片生成 API 不启用。视觉偏好库为空。 |
| P1 (人机协同) | 模块自动生成初版封面，设计师在后台审核和修改。图片生成 API 启用。视觉偏好库开始积累数据（收集人工修改记录）。修改率高的色彩/字体/装饰方案会触发偏好更新。 |
| P2 (系统主导) | 模块全自动生成封面，仅当 quality_scores.overall < 60 或人工设置了强制审核标记时才通知人工。M-26 根据高点击率封面自动优化视觉偏好。支持 A/B 测试（生成 2 个变体封面，发布后自动选优）。 |
