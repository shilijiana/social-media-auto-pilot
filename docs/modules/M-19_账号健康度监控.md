# M-19 账号健康度监控

---

## 1. 模块定位

| 属性 | 值 |
|------|-----|
| **模块编号** | M-19 |
| **模块名称** | 账号健康度监控 |
| **所属分类** | 分发采集 |
| **优先级** | P1（核心模块，必须稳定运行） |
| **触发方式** | 定时任务（每日 08:00 UTC+8 自动执行）；支持手动触发 |
| **在系统中的位置** | 接收 M-18 产出的表现指标数据，对账号进行健康状态评估与异常告警，输出结果供 M-20 数据清洗与聚合使用 |

### 1.1 核心职责

1. **每日账号状态巡检**：检查各平台账号是否正常登录、是否收到平台警告信/限流通知、是否触发风控。
2. **流量异常监控**：追踪近 7 日平均浏览量，与历史平均值（30 日滑动窗口）进行对比，下降超过 30% 则触发告警。
3. **粉丝异动监控**：监测粉丝增减趋势，当出现加速掉粉（单日掉粉 > 日均掉粉量的 3 倍）时触发告警。
4. **周度健康报告生成**：每周一自动汇总过去一周的账号健康数据，生成结构化周报（JSON + Markdown 双格式），存储于 `data/files/reports/weekly/`。

### 1.2 上下游关系

```
M-17(发布执行) → M-18(表现数据采集) → M-19(本模块) → M-20(数据清洗与聚合)
```

- **上游**：M-18 提供 `performance_metrics` 数据（包含各平台各内容的浏览、互动、粉丝等指标）
- **下游**：M-20 消费 `account_health` 表作为数据清洗的过滤维度之一

## 4. 核心逻辑

### 4.1 主流程

```
开始
  │
  ├─ 1. 读取输入 → 解析 M-18 performance_metrics
  │
  ├─ 2. 账号状态检查
  │     ├─ 检查 login_status
  │     ├─ 检查 warning_letters 数量和严重程度
  │     ├─ 检查 content_violations
  │     └─ 检查 shadow_ban_indicator
  │     → 生成 account_status_score (0-100)
  │
  ├─ 3. 流量异常检测
  │     ├─ 计算 7 日平均浏览量 = avg(total_views over last 7 days)
  │     ├─ 获取 30 日历史平均浏览量 (historical_avg_views)
  │     ├─ deviation = (7day_avg - historical_avg) / historical_avg * 100
  │     ├─ IF deviation < -30% → 触发 traffic_drop 告警(severity=warning)
  │     ├─ IF deviation < -50% → 升级为 critical
  │     └─ 生成 traffic_score (0-100)
  │
  ├─ 4. 粉丝异动检测
  │     ├─ 获取当日 follower_change
  │     ├─ 获取 30 日日均粉丝变化量 (historical_avg_followers_delta)
  │     ├─ IF follower_change < 0 AND |follower_change| > 3 * |historical_avg_followers_delta|
  │     │   → 触发 follower_loss 告警(severity=warning)
  │     ├─ IF 连续 3 天加速掉粉 → 升级为 critical
  │     └─ 生成 follower_score (0-100)
  │
  ├─ 5. 互动率趋势检测
  │     ├─ 计算 7 日平均互动率 vs 30 日平均互动率
  │     ├─ IF 下降 > 20% → 触发 engagement_decline 告警
  │     └─ 生成 engagement_score (0-100)
  │
  ├─ 6. 综合健康评分计算
  │     ├─ health_score = weighted_avg(
  │     │     account_status_score × 0.30,
  │     │     traffic_score × 0.30,
  │     │     follower_score × 0.20,
  │     │     engagement_score × 0.20
  │     │   )
  │     ├─ health_status = classify(health_score):
  │     │     ≥ 80 → "healthy"
  │     │     60-79 → "attention"
  │     │     40-59 → "warning"
  │     │     < 40 → "critical"
  │     └─ 写入 account_health 表
  │
  ├─ 7. 告警通知
  │     ├─ IF health_status ∈ {warning, critical}
  │     │   → 通过通知渠道发送告警（钉钉/邮件/企业微信）
  │     └─ 记录告警日志
  │
  └─ 8. 周报检查
        ├─ IF 当前是周一 → 汇总上周 7 天 account_health 数据
        ├─ 生成周度健康报告（JSON + Markdown）
        ├─ 存储至 data/files/reports/weekly/weekly_health_YYYY-MM-DD.json
        └─ 存储至 data/files/reports/weekly/weekly_health_YYYY-MM-DD.md
```

### 4.2 评分算法详情

**account_status_score 计算规则**：
| 条件 | 扣分 |
|------|------|
| login_status = "expired" | -30 |
| login_status = "locked" | -60 |
| login_status = "verification_required" | -20 |
| 每条 warning_letter(severity=warning) | -10 |
| 每条 warning_letter(severity=critical) | -25 |
| 每条 content_violation | -15 |
| shadow_ban_indicator = true | -40 |
| 基础分 | 100 |

**traffic_score 计算规则**：
```
deviation_pct = (7day_avg_views - 30day_avg_views) / 30day_avg_views × 100
score = max(0, min(100, 100 + deviation_pct))
# 即：持平 = 100 分，下降 50% = 50 分，上升 50% = 100 分（上限）
```

**follower_score 计算规则**：
```
IF historical_avg_followers_delta >= 0:
    # 正常情况下粉丝增长
    IF follower_change >= 0:
        score = 100
    ELSE:
        score = max(0, 100 - abs(follower_change) * 0.5)
ELSE:
    # 正常情况下也在掉粉
    IF follower_change >= historical_avg_followers_delta:
        score = 100
    ELSE:
        ratio = follower_change / historical_avg_followers_delta
        score = max(0, 100 - (ratio - 1) * 50)
```

### 4.3 周报生成逻辑

1. 查询过去 7 天所有平台的 `account_health` 记录
2. 按平台分组，计算日均健康分、健康趋势
3. 汇总所有告警，标记已解决/未解决
4. 生成结构化 JSON 和人类可读 Markdown
5. 文件命名：`weekly_health_{week_end_date}.json` / `.md`

## 6. 错误处理

### 6.1 异常分类

| 异常类型 | 处理策略 | 重试次数 | 降级方案 |
|----------|----------|----------|----------|
| 上游 M-18 数据缺失 | 记录日志，跳过本次检查 | 3 次（间隔 5min） | 使用上一日数据作为近似值 |
| 账号凭证过期 | 触发 login_issue 告警 | 0 | 标记账号状态为 expired |
| 历史数据不足（新账号 < 7 天） | 使用已有天数的平均值 | - | health_score 标记为 "insufficient_data" |
| 数据库写入失败 | 记录错误日志 | 3 次（间隔 30s） | 暂存到本地文件，后续补录 |
| 通知发送失败 | 记录告警日志 | 3 次（间隔 1min） | 降级为仅数据库记录 |
| 周报生成失败 | 记录错误 | 2 次（间隔 10min） | 延迟到下一轮定时任务重试 |
| 除零异常（历史平均值为 0） | 设 deviation_pct = 0 | - | 跳过该维度的百分比计算 |

## 7. 自学习机制

### 7.1 动态基线调整

```
每 30 天自动重新计算：
- historical_avg_views：过去 30 天日均浏览量
- historical_avg_followers_delta：过去 30 天日均粉丝变化量
- 告警阈值可根据历史波动率自适应调整：
  volatility = std(过去 30 天浏览量) / mean(过去 30 天浏览量)
  IF volatility > 0.5 → 告警阈值放宽至 -40%（避免高波动账号频繁误报）
  IF volatility < 0.15 → 告警阈值收紧至 -20%（稳定账号更早发现问题）
```

### 7.2 告警疲劳控制

```
同一账号同一类型告警：
- 24 小时内最多触发 1 次（首次触发后，后续同类型不再重复发送）
- 连续 3 天触发同一告警 → 自动升级 severity 一级
- 连续 7 天未触发 → 自动降级 severity 一级
```

### 7.3 假阳性反馈

```
运营人员可标记告警为"误报"：
- 记录误报标记到 alerts_log
- 累计同一规则误报 > 5 次 → 自动调宽该规则的阈值 10%
- 累计同一规则误报 > 10 次 → 触发人工审核，建议调整规则参数
```

## 9. Hermes Agent 执行指令

### 9.1 每日健康检查指令

```
你作为 M-19 账号健康度监控模块的 Hermes Agent，请按照以下步骤执行每日健康检查：

1. 从 M-18 获取所有账号最新的 performance_metrics 数据
2. 对每个账号执行以下检查：
   a. 账号状态检查：
      - 读取 account_status.login_status
      - 统计 warning_letters 数量和严重等级
      - 统计 content_violations 数量
      - 检查 shadow_ban_indicator
      - 按评分规则计算 account_status_score
   b. 流量异常检测：
      - 从数据库查询该账号近 7 天的 total_views，计算 7 日平均值
      - 与 historical_avg_views 对比
      - 计算偏差百分比 deviation_pct = (7day_avg - historical_avg) / historical_avg * 100
      - 如果 deviation_pct < -30%，生成 traffic_drop 告警
      - 按评分规则计算 traffic_score
   c. 粉丝异动检测：
      - 读取当日 follower_change
      - 与 historical_avg_followers_delta 对比
      - 如果 |follower_change| > 3 * |historical_avg_followers_delta| 且为负值
        → 生成 follower_loss 告警
      - 查询过去 3 天粉丝变化趋势，如果连续加速掉粉 → 升级告警级别
      - 按评分规则计算 follower_score
   d. 互动率趋势检测：
      - 计算 7 日平均互动率 vs 30 日平均互动率
      - 如果下降超过 20%，生成 engagement_decline 告警
      - 按评分规则计算 engagement_score
3. 计算综合健康评分：
   health_score = account_status_score * 0.30 + traffic_score * 0.30 + follower_score * 0.20 + engagement_score * 0.20
4. 判定健康状态等级：
   - health_score >= 80 → "healthy"
   - 60 <= health_score < 80 → "attention"
   - 40 <= health_score < 60 → "warning"
   - health_score < 40 → "critical"
5. 将结果写入 account_health 表
6. 如果 health_status 为 "warning" 或 "critical"，通过通知渠道发送告警
7. 如果是周一，触发周度健康报告生成

注意事项：
- 新账号（数据不足 7 天）：跳过流量和粉丝异常检测，仅检查账号状态
- 告警去重：同一账号同一类型告警 24h 内不重复发送
- 所有计算保留 2 位小数精度
- 异常情况记录到错误日志并继续处理下一个账号
```

### 9.2 周报生成指令

```
你作为 M-19 账号健康度监控模块的 Hermes Agent，请生成周度健康报告：

1. 确定报告周期：本周一至本周日
2. 从 account_health 表查询该周期内所有记录
3. 按平台分组汇总：
   a. 计算每个平台的日均健康分
   b. 判定健康趋势：对比上周日均分
      - 本周 > 上周 × 1.05 → "improving"
      - 本周 < 上周 × 0.95 → "declining"
      - 其他 → "stable"
   c. 提取该平台本周的关键发现（TOP 3 告警类型）
   d. 基于告警类型生成改进建议
4. 计算整体摘要：
   - 所有平台平均分
   - 最佳/最差平台
   - 总告警数和严重告警数
5. 生成 JSON 报告 → data/files/reports/weekly/weekly_health_{end_date}.json
6. 生成 Markdown 报告 → data/files/reports/weekly/weekly_health_{end_date}.md
7. 将报告元数据写入 weekly_health_reports 表
8. 通过通知渠道发送报告摘要（整体健康评分 + 关键告警）
```
