  上游: M-05（素材去重与质量控制模块）→ 传入质检通过的素材列表及质量评分
  下游: M-07（选题与内容规划模块）→ 使用素材库中的版权确认素材进行选题匹配

  自学习: M-25 收集素材使用记录和表现数据，优化素材优先级排序

### 1.2 核心职责

将质检通过的素材正式归档到素材库，为每个素材生成全局唯一 ID（格式: `MAT-YYYY-NNNNNN`）。管理素材的版权状态——对已确认商业可用（免费授权或已购买许可）的素材标记「版权已确认」，追踪授权许可的到期时间并在到期前 7 天发出预警。构建素材使用记录，在内容生产中优先推荐版权已确认且历史表现优异的素材。确保素材库中所有素材均可追溯来源和版权状态，为合规运营提供保障。

### 1.3 触发时机

- **自动触发**: M-05（素材去重与质量控制模块）完成后，由流水线引擎自动调用。对 passed=true 的素材执行入库操作
- **人工触发**: 运营人员在素材管理后台手动上传素材后，触发「入库并标记版权」
- **定时触发**: 每日 08:00 UTC 扫描 `copyright_records` 表，检查即将到期（7 天内）的素材许可并发送预警通知；每月 1 日 00:00 UTC 执行素材使用统计汇总

### 2.3 数据来源

| 字段 | 来源 |
|------|------|
| `task_id` | 流水线引擎生成 |
| `curated_materials[].material_ref` | M-05 输出的 quality_results（passed=true） |
| `curated_materials[].storage_path` | M-05 移动到 curated 目录后的路径 |
| `curated_materials[].quality_score` | M-05 输出的 quality_score |
| `curated_materials[].metadata` | M-02/M-03 输出的素材元数据（通过 M-05 透传） |
| `curated_materials[].copyright_info` | M-02/M-03 预填的版权信息（如有）或运营人员手动填入 |
| `archiving_options` | 配置文件 `config/archiving_rules.yaml` 加载，支持运行时覆盖 |

## 4. 核心处理逻辑

### 4.1 处理步骤

```
Step 1: 输入校验
  - 做什么: 校验 task_id 格式、curated_materials 非空（1-500）、每个素材的必填字段完整
  - 怎么判断: 使用 jsonschema 库校验，对比 2.1 中的 Schema
  - 输出: 校验通过 → 进入 Step 2; 校验失败 → 返回 status=failed，记录错误详情

Step 2: 唯一 ID 生成
  - 做什么: 为每个待入库素材生成全局唯一 ID
  - 怎么判断:
      - ID 格式: MAT-YYYY-NNNNNN（前缀-年份-6位序号）
      - 从 material_id_sequence 表获取当前年份的下一个序号（原子操作: SELECT FOR UPDATE）
      - 使用 PostgreSQL 序列或 Redis INCR 保证并发安全
      - 序号从 000001 开始，每年 1 月 1 日重置
      - 生成 ID 后立即占用（写入 material_id_sequence 表），避免空洞
  - 输出: 每个素材的 material_id

Step 3: 版权状态分配
  - 做什么: 确定每个素材的版权状态
  - 怎么判断:
      - 如果输入包含 copyright_info.status 且不为 null，使用预填状态
      - 如果 status = "licensed" 且有 license_expiry_date:
        - 计算 days_until_expiry = license_expiry_date - 当前日期
        - days_until_expiry <= 0 → 自动更新 status 为 "expired"
        - days_until_expiry <= 7 → 自动更新 status 为 "expiring"
        - days_until_expiry > 7 → 保持 status 为 "licensed"
      - 如果 status = "unknown" 且 auto_detect_license = true:
        - 根据 source_platform 查询许可检测规则
        - unsplash/pexels/pixabay → 自动标记为 "free"，license_type 设为对应平台许可
        - 其他平台 → 保持 "unknown"
      - 如果没有 copyright_info → 使用 copyright_defaults.default_status
      - 判断 is_copyright_confirmed:
        - status = "free" → true
        - status = "licensed" 且 days_until_expiry > 0 → true
        - 其他 → false
  - 输出: 每个素材的 copyright_status、copyright_detail、is_copyright_confirmed

Step 4: 许可到期预警
  - 做什么: 检测即将到期或已过期的许可
  - 怎么判断:
      - 扫描所有 status = "licensed" 的素材
      - days_until_expiry <= 0 → 加入 expired_items 列表，更新 status 为 "expired"
      - 0 < days_until_expiry <= 7 → 加入 expiring_soon 列表，更新 status 为 "expiring"
      - 写入 copyright_alerts 表，触发后续通知流程
      - 对于已过期但仍在内容中被引用的素材（was_in_use=true），发送 CRITICAL 级别告警
  - 输出: copyright_summary 中的 expiring_soon 和 expired_items 列表

Step 5: 素材优先级计算
  - 做什么: 计算每个素材的 priority_flag，用于下游模块排序
  - 怎么判断:
      - priority_flag 基础分 = quality_score * 0.5
      - copyright 加分:
        - is_copyright_confirmed = true → +30
        - commercial_use_allowed = true → +10
        - modification_allowed = true → +10
      - 历史表现加分（如有）:
        - 该素材类型/标签历史上产生高互动的记录 → +10
      - priority_flag = min(基础分 + 加分, 100)
      - 权重可在 config/archiving_rules.yaml 中调整
  - 输出: 每个素材的 priority_flag（0-100）

Step 6: 缩略图生成
  - 做什么: 为图片和视频素材生成多尺寸缩略图
  - 怎么判断:
      - 如果 generate_thumbnails = false，跳过此步骤
      - 图片: 使用 Pillow 按 thumbnail_sizes 配置生成缩略图
        - 保持宽高比，使用 LANCZOS 重采样
        - 保存为 JPEG 格式，质量 85%
        - 文件名: {material_id}_{size_name}.jpg
      - 视频: 使用 ffmpeg 提取首帧作为封面图
        - 命令: `ffmpeg -i <input> -ss 00:00:01 -vframes 1 -q:v 2 <output>`
        - 对提取的首帧按 thumbnail_sizes 生成缩略图
      - 缩略图上传至 s3://materials/library/thumbnails/
  - 输出: 每个素材的 thumbnail_paths 对象

Step 7: 自动分类
  - 做什么: 根据 content_tags 自动为素材分配分类标签
  - 怎么判断:
      - 如果 auto_categorize = false，跳过此步骤
      - 从 metadata.content_tags 中提取 confidence >= 0.7 的标签
      - 将标签名作为 categories 列表（最多 5 个标签）
      - 如果 content_tags 为空，根据 type 和 source_platform 分配默认分类
      - 分类标签写入 materials 表的 categories 字段（数组类型）
  - 输出: 每个素材的 categories 列表

Step 8: 素材文件迁移与入库
  - 做什么: 将素材从 curated 目录迁移到正式素材库，写入数据库
  - 怎么判断:
      - 将素材从 s3://materials/curated/ 移动到 s3://materials/library/<type>/YYYY/MM/DD/
      - 文件命名: {material_id}.{format}
      - 使用 OSS copy + delete 操作确保原子性
      - 写入 materials 表（主表记录）
      - 写入 copyright_records 表（版权记录）
      - 写入 material_usage_log 表（初始使用记录: usage_count=0, last_used_at=null）
      - 写入 module_outputs 表
      - 生成 copyright_summary 汇总
  - 输出: 最终的 Output JSON
```

## 6. 错误处理策略

| 错误类型 | 处理方式 | 重试策略 | 告警级别 |
|---------|---------|---------|---------|
| 输入 JSON Schema 校验失败 | 返回 status=failed，错误详情写入 `errors` 表，终止执行 | 不重试 | ERROR |
| ID 序号分配冲突（并发写入 material_id_sequence） | 使用 PostgreSQL Advisory Lock 或 SELECT FOR UPDATE 重试 | 3次，间隔 100ms | WARN |
| 版权许可日期格式错误 | 记录警告日志，该字段设为 null，status 保持原值 | 不重试 | WARN |
| OSS 文件移动失败（源文件不存在） | 记录该素材为 failed，继续处理剩余素材 | 不重试（源文件缺失无法恢复） | ERROR |
| OSS 文件移动失败（目标路径权限不足） | 检查 OSS Bucket Policy，发送告警 | 3次，间隔 2s | CRITICAL |
| 缩略图生成失败（图片格式损坏） | 记录警告，thumbnail_paths 设为 null，素材仍正常入库 | 不重试 | WARN |
| 缩略图生成失败（ffmpeg 不可用） | 视频素材 thumbnail_paths 设为 null，记录警告 | 不重试 | WARN |
| 数据库写入失败（连接断开/锁超时） | 暂存结果到本地 JSON 文件 `data/m06_backup_{task_id}.json`，发送告警 | 5次，间隔 2s | CRITICAL |
| 数据库写入失败（唯一约束冲突 - material_id 重复） | 重新生成 ID（递增序号）后重试 | 3次 | WARN |
| copyright_records 表写入失败 | 素材仍正常入库到 materials 表，版权记录暂存到本地 JSON 文件 | 5次，间隔 2s | WARN |
| 全部素材入库失败（status=failed） | 通知流水线引擎终止下游模块，写入 `pipeline_errors` 表 | 不重试 | CRITICAL |

### 降级策略

当模块完全不可用时（如 OSS 不可达 + 数据库离线），启用以下降级方案：

1. **延迟入库**: 将 curated_materials 数据暂存到本地文件系统 `data/pending_archive/`，待基础设施恢复后由恢复任务自动重新执行入库
2. **人工介入**: 发送企业微信/钉钉通知给运营团队，附上 curated 目录下的素材列表，由人工手动录入
3. **跳过本模块**: 在 `pipeline_config` 中设置 `material_archiver.skip_on_failure = true`，流水线直接使用 curated 目录的素材路径继续后续流程（使用临时 ID 替代正式 material_id）

## 7. 自学习触发条件

### 7.1 学习数据收集

本模块向以下表写入数据，供 M-25 定期收集分析：

| 数据字段 | 说明 | 收集时机 |
|---------|------|---------|
| `material_id` | 素材唯一 ID | 入库时写入 |
| `usage_count` | 素材被使用的总次数 | M-10 每次使用后累加 |
| `last_used_at` | 最近一次被使用的时间 | M-10 每次使用后更新 |
| `content_performance` | 使用该素材的内容表现（互动率、转化率等） | 内容发布后 7 天内，从 M-22 获取 |
| `avg_engagement_by_type` | 按素材类型的平均互动数据 | M-22 定期汇总后回写 |
| `priority_effectiveness` | 高优先级素材是否确实产生了更好的内容表现 | M-25 分析后回写 |
| `copyright_risk_score` | 版权风险评估（基于过期次数、来源平台风险等级） | 每次版权状态变更时计算 |

### 7.2 偏好库调用

- **调取偏好类型**: 素材来源偏好（用户偏好使用哪些平台的素材）、素材类型偏好（图片 vs 视频的使用频率和表现差异）
- **置信度阈值**: >= 70%
- **无偏好时默认行为**: 使用 `config/archiving_rules.yaml` 中的默认优先级权重，按 quality_score 和 is_copyright_confirmed 排序

### 7.3 阶段演进行为

| 阶段 | 行为差异 |
|------|---------|
| P0 (人工主导) | 素材入库完全由人工完成。运营人员手动上传素材、填写元数据、确认版权状态。模块仅负责生成唯一 ID 和记录入库时间。版权预警由人工定期检查。 |
| P1 (人机协同) | 模块自动生成 ID、生成缩略图、计算优先级。版权状态仍需人工确认（特别是来源为社交媒体平台的素材）。版权预警自动发送通知。素材使用记录自动追踪。 |
| P2 (系统主导) | 模块全自动执行入库全流程。版权状态通过来源平台自动检测（Unsplash/Pexels 等标记为 free，其他平台根据历史确认模式推断）。仅当检测到许可即将过期或素材被多次使用的许可即将到期时通知人工。M-25 根据素材历史表现自动调整 priority_flag 权重。 |

### 注意事项
- dry-run 模式不会写入数据库或移动 OSS 文件，仅输出到本地 JSON 文件

- 版权预警仅对 status = "licensed" 且有 license_expiry_date 的素材生效
