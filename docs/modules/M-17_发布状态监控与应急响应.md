# M-17: 发布状态监控与应急响应

## 1. 模块概述

| 属性 | 值 |
|------|-----|
| 模块编号 | M-17 |
| 模块名称 | 发布状态监控与应急响应 |
| 所属类别 | 分发采集 |
| 优先级 | P0（最高优先级，核心模块） |
| 核心职责 | 发布后持续监控页面反馈，按状态分类处理：审核中（超时等待后复查）、发布失败（截图+标记人工处理）、网络故障（24h/48h 自动重试）、限流/降权（记录原因，评估是否可重发），确保每次发布都有明确终态 |
| 目标用户 | Hermes Agent（自动化监控）、运营人员（应急处理） |

## 2. 输入输出定义

### 2.1 输入

| 输入项 | 来源模块 | 数据格式 | 触发条件 |
|--------|----------|----------|----------|
| 发布任务 | M-16（多平台分发） | `PublishTask` | M-16 提交发布后触发监控注册 |
| 发布完成通知 | M-16 | `PublishCompletedEvent` | M-16 完成发布操作后发送 |
| 人工应急指令 | 前端/通知系统 | `EmergencyCommand` | 人工介入时的操作指令 |

### 2.2 输出

| 输出项 | 目标 | 数据格式 | 说明 |
|--------|------|----------|------|
| 发布状态更新 | `publish_tasks` 表 | `PublishStatusUpdate` | 更新发布任务为终态 |
| 告警通知 | 通知系统 | `Alert` | 关键异常告警 |
| 监控截图 | 对象存储 | `Screenshot` | 异常状态的页面截图 |
| 发布完成事件 | M-18（数据采集） | `PublishFinalizedEvent` | 发布确认完成后触发数据采集 |

## 4. 核心业务逻辑

### 4.1 状态分类与处理策略

```
发布完成 (M-16)
    │
    ▼
┌──────────────────────────────────────────────────────┐
│                    状态检测分类                         │
└──────────────────────────────────────────────────────┘
    │
    ├── 发布成功 (published)
    │   └── 确认URL可访问 → 记录成功 → 通知M-18开始数据采集
    │
    ├── 审核中 (under_review)
    │   └── 每30分钟复查 → 最多等待24小时
    │       ├── 审核通过 → 标记 published
    │       └── 超时 → 标记 review_timeout → 告警(medium)
    │
    ├── 审核未通过 (review_failed)
    │   └── 截图保存 → 记录拒绝原因 → 告警(high) → 人工处理
    │
    ├── 发布失败 (publish_failed)
    │   └── 截图保存 → 分析失败原因 → 告警(high) → 人工处理
    │       └── 可重发原因 → 建议重新发布
    │
    ├── 网络故障 (network_error)
    │   └── 记录故障时间 → 自动重试计划:
    │       第1次: 24小时后
    │       第2次: 48小时后
    │       └── 仍失败 → 告警(critical) → 人工处理
    │
    └── 疑似降权/限流 (shadow_banned)
        └── 记录检测依据 → 评估影响范围
            ├── 可重发 → 建议更换内容重新发布
            └── 不可重发 → 告警(medium) → 记录经验
```

### 4.2 监控轮询策略

```
初始间隔: 5 分钟（前 30 分钟高频检查）
    ↓
稳定间隔: 15 分钟（30 分钟 - 2 小时）
    ↓
低频间隔: 1 小时（2 小时 - 24 小时）
    ↓
最终检查: 24 小时后最终确认
    ↓
超时未决: 标记 human_required
```

### 4.3 审核超时处理

| 平台 | 典型审核时长 | 超时阈值 | 超时后处理 |
|------|-------------|----------|------------|
| 抖音 | 5-30 分钟 | 24 小时 | 告警 + 持续监控 |
| 小红书 | 10-60 分钟 | 24 小时 | 告警 + 持续监控 |
| B站 | 30 分钟-4 小时 | 48 小时 | 告警 + 持续监控 |
| Twitter | 即时 | 1 小时 | 直接标记失败 |

### 4.4 降权检测逻辑

检测指标（综合判断，至少满足 2 项触发告警）:
1. 内容在搜索结果中不可见（搜索标题前10个词 + 账号名，无结果）
2. 浏览量增长异常（发布 1 小时后浏览量 < 平台平均值的 10%）
3. 互动率为 0（有展示但无点赞/评论/收藏）
4. 平台返回限流提示或违规通知

### 4.5 网络故障重试机制

```
第 1 次故障 → 等待 5 分钟 → 重试
第 2 次故障 → 等待 30 分钟 → 重试
第 3 次故障 → 进入长期重试计划:
    ├── 24 小时后重试
    ├── 48 小时后重试
    └── 仍失败 → 标记 human_required → 发送 critical 告警
```

## 6. 数据库表设计

### 6.1 publish_monitors 表

```sql
CREATE TABLE publish_monitors (
    monitor_id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    task_id UUID NOT NULL REFERENCES publish_tasks(task_id) ON DELETE CASCADE,
    content_id UUID NOT NULL,
    platform VARCHAR(30) NOT NULL,
    publish_url TEXT,
    status VARCHAR(30) NOT NULL DEFAULT 'monitoring',
    status_detail TEXT,
    check_count INTEGER DEFAULT 0,
    max_checks INTEGER DEFAULT 20,
    last_check_at TIMESTAMPTZ,
    next_check_at TIMESTAMPTZ,
    check_interval_seconds INTEGER DEFAULT 300,
    retry_plan JSONB,
    created_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    resolved_at TIMESTAMPTZ,
    UNIQUE(task_id)
);

CREATE INDEX idx_pm_status ON publish_monitors(status);
CREATE INDEX idx_pm_next_check ON publish_monitors(next_check_at) WHERE status IN ('monitoring', 'under_review', 'retrying');
```

### 6.2 monitor_screenshots 表

```sql
CREATE TABLE monitor_screenshots (
    screenshot_id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    monitor_id UUID NOT NULL REFERENCES publish_monitors(monitor_id) ON DELETE CASCADE,
    task_id UUID NOT NULL,
    url TEXT NOT NULL,
    taken_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    description TEXT,
    page_status VARCHAR(50)
);

CREATE INDEX idx_ms_monitor ON monitor_screenshots(monitor_id);
```

### 6.3 publish_alerts 表

```sql
CREATE TABLE publish_alerts (
    alert_id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    task_id UUID NOT NULL REFERENCES publish_tasks(task_id),
    content_id UUID,
    alert_type VARCHAR(50) NOT NULL,
    severity VARCHAR(20) NOT NULL,
    message TEXT NOT NULL,
    detail JSONB,
    screenshot_url TEXT,
    created_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    acknowledged_at TIMESTAMPTZ,
    acknowledged_by VARCHAR(100),
    resolution TEXT
);

CREATE INDEX idx_pa_task ON publish_alerts(task_id);
CREATE INDEX idx_pa_severity ON publish_alerts(severity);
CREATE INDEX idx_pa_unacknowledged ON publish_alerts(created_at) WHERE acknowledged_at IS NULL;
```

## 7. 异常处理

| 异常场景 | 处理策略 |
|----------|----------|
| 监控页面加载失败 | 重试 3 次，间隔 30 秒，仍失败按网络故障处理 |
| 页面结构变化导致状态检测失败 | 截图保存 → 降级为人工判断 → 告警通知 |
| 监控任务队列积压 | 动态调整检查间隔，优先处理高优先级任务 |
| 监控服务宕机 | 重启后扫描所有 `monitoring` 状态记录，恢复监控 |
| 告警风暴（短时间大量告警） | 同类告警合并，5 分钟内同类型只发送 1 次 |

## 8. Hermes Agent 执行指令

```yaml
hermes_agent:
  module: M-17
  triggers:
    - event: publish.completed
      source: M-16
      action: register_and_start_monitor
    - schedule: "*/5 * * * *"
      action: process_monitoring_queue
    - command: emergency_action
      source: [user, admin]
      action: execute_emergency_command
    - schedule: "0 */1 * * *"
      action: check_retry_schedule

  workflows:
    register_and_start_monitor:
      description: 注册发布后监控并开始首次检查
      steps:
        - step: 创建监控记录
          function: create_monitor_record(task_id, content_id, platform, publish_url)
          config:
            initial_interval: 300
            max_checks: 20
        - step: 执行首次检查
          function: check_publish_status(monitor_id)
          delay: 60s
        - step: 调度后续检查
          function: schedule_next_check(monitor_id, interval=300)

    process_monitoring_queue:
      schedule: every 5 minutes
      steps:
        - step: 查询到期检查
          function: get_due_monitors()
          query: SELECT * FROM publish_monitors WHERE next_check_at <= NOW() AND status IN ('monitoring', 'under_review', 'retrying')
        - step: 并行执行检查（每个监控独立）
          loop: for monitor in due_monitors
          function: check_publish_status(monitor.monitor_id)

    check_publish_status:
      description: 检查单次发布状态
      steps:
        - step: 更新检查计数
          function: increment_check_count(monitor_id)
        - step: 打开发布页面
          function: open_publish_page(publish_url, platform)
          on_error:
            if: network_error
            then: handle_network_error(monitor_id)
            else: retry(3, interval=30s) >> handle_persistent_error
        - step: 检测页面状态
          function: detect_page_status(page, platform)
          returns: status_classification
        - step: 按分类处理
          function: handle_status_by_classification(monitor_id, status_classification)
          branches:
            published:
              - function: mark_monitor_resolved(monitor_id, "published")
              - function: notify_m18_publish_finalized(task_id, "published")
              - function: update_task_status(task_id, "published")
            under_review:
              - function: update_monitor_status(monitor_id, "under_review")
              - function: check_review_timeout(monitor_id)
                if: exceeded_24h
                then:
                  - function: send_alert("review_timeout", severity="medium")
                  - function: mark_monitor_resolved(monitor_id, "review_timeout")
                else:
                  - function: schedule_next_check(monitor_id, interval=900)
            review_failed:
              - function: take_screenshot(monitor_id, "review_rejected")
              - function: extract_rejection_reason(page)
              - function: send_alert("review_rejected", severity="high")
              - function: mark_monitor_resolved(monitor_id, "review_failed")
              - function: update_task_status(task_id, "failed")
            publish_failed:
              - function: take_screenshot(monitor_id, "publish_failed")
              - function: analyze_failure_reason(page)
              - function: send_alert("publish_failed", severity="high")
              - function: mark_monitor_resolved(monitor_id, "publish_failed")
              - function: update_task_status(task_id, "failed")
            shadow_banned:
              - function: take_screenshot(monitor_id, "shadow_ban_suspected")
              - function: collect_ban_evidence(page)
              - function: assess_re_publish_feasibility(evidence)
              - function: send_alert("shadow_ban_detected", severity="medium")
              - function: mark_monitor_resolved(monitor_id, "shadow_banned")
            network_error:
              - function: handle_network_error(monitor_id)

    handle_network_error:
      description: 处理网络故障，设置重试计划
      steps:
        - step: 检查当前重试次数
          function: get_retry_count(monitor_id)
        - step: 按重试阶段处理
          switch: retry_count
          cases:
            0-2:
              - function: schedule_retry(monitor_id, delay_minutes=[5, 30, 60][retry_count])
              - function: update_monitor_status(monitor_id, "retrying")
            3:
              - function: setup_long_retry_plan(monitor_id, [24h, 48h])
              - function: update_monitor_status(monitor_id, "retrying")
            4:
              - function: setup_long_retry_plan(monitor_id, [48h])
              - function: update_monitor_status(monitor_id, "retrying")
            default:
              - function: send_alert("network_persistent_failure", severity="critical")
              - function: mark_monitor_resolved(monitor_id, "network_error")
              - function: request_human_intervention(monitor_id, "network_exhausted")

    check_review_timeout:
      description: 检查审核是否超时
      steps:
        - step: 计算审核等待时长
          function: calculate_wait_duration(monitor_id)
        - step: 获取平台超时阈值
          function: get_platform_timeout_threshold(platform)
          default: 86400
        - step: 比较并处理
          if: wait_duration > threshold
          then:
            - function: take_screenshot(monitor_id, "review_timeout")
            - function: send_alert("review_timeout", severity="medium")
            - function: mark_monitor_resolved(monitor_id, "review_timeout")

    assess_re_publish_feasibility:
      description: 评估降权内容是否可以重发
      steps:
        - step: 分析降权原因
          function: analyze_ban_reason(evidence)
        - step: 检查内容合规性
          function: check_content_compliance(content_id)
        - step: 生成重发建议
          function: generate_re_publish_advice(ban_reason, compliance_result)
          possible_outputs:
            - "可原内容重发（修改标题/标签）"
            - "需修改内容后重发"
            - "不建议重发（记录经验）"

    execute_emergency_command:
      description: 执行人工应急指令
      steps:
        - step: 验证指令权限
          function: verify_command_permission(command.issued_by, command.action)
          on_error: abort("权限不足")
        - step: 按操作类型执行
          switch: command.action
          cases:
            retry_now:
              - function: reset_monitor_for_retry(command.task_id)
              - function: check_publish_status(monitor_id)
            force_retry:
              - function: update_task_status(command.task_id, "pending")
              - function: notify_m16_republish(command.task_id)
            cancel_publish:
              - function: mark_monitor_resolved(monitor_id, "cancelled")
              - function: update_task_status(command.task_id, "cancelled")
            mark_resolved:
              - function: mark_monitor_resolved(monitor_id, "resolved", command.reason)
            switch_account:
              - function: trigger_account_failover(command.task_id, command.params.account_id)
            ignore:
              - function: mark_monitor_resolved(monitor_id, "resolved", "ignored_by_human")
        - step: 记录操作日志
          function: log_emergency_action(command)

    schedule_next_check:
      description: 调度下一次状态检查
      steps:
        - step: 计算检查间隔
          function: calculate_interval(check_count)
          logic: |
            if check_count <= 6: interval = 300 (前30分钟)
            elif check_count <= 12: interval = 900 (30分钟-2小时)
            elif check_count <= 24: interval = 3600 (2小时-24小时)
            else: interval = 7200 (24小时后)
        - step: 更新下次检查时间
          function: set_next_check_at(monitor_id, NOW() + interval)
```

## 9. 非功能性需求

| 类别 | 要求 |
|------|------|
| 性能 | 单次状态检查 < 10 秒；监控轮询不阻塞发布流程 |
| 可靠性 | 监控覆盖率 100%（每个发布任务均有监控记录）；告警送达率 > 99% |
| 可扩展性 | 支持同时监控 500+ 个活跃发布任务 |
| 安全 | 告警信息不泄露敏感数据；截图存储加密 |
| 可用性 | 监控服务宕机后 5 分钟内自动恢复 |
