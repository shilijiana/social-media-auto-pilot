# M-25 学习数据收集

---

## 一、模块定位

| 属性 | 值 |
|------|-----|
| 模块编号 | M-25 |
| 模块名称 | 学习数据收集 |
| 所属分类 | 自学习 |
| 优先级 | P1 |
| 上游依赖 | M-13（人工审阅与反馈） |
| 下游依赖 | M-26（偏好提炼与规则生成） |
| 触发方式 | 事件驱动（审阅完成后自动触发） |

### 功能概述

本模块是自学习系统的**数据采集层**。在每次人工审阅完成后，自动收集所有审阅节点记录（通过/修改/拒绝/补充），按照文案、封面、视频、素材、话题、策略六大类别进行分类归档，为每条记录打上标签，记录上下文信息，形成结构化的学习数据记录。首次运行时可导入初始偏好模板。

### 设计目标

1. **零遗漏**：所有审阅操作均被记录，不可跳过
2. **细粒度标签**：每个修改/拒绝操作需标注具体原因标签
3. **上下文完整**：记录操作发生时的完整环境信息
4. **分类清晰**：六大类别覆盖内容生产全链路

### 4.2 分类逻辑

| content_type | category | 说明 |
|-------------|----------|------|
| copy | copy | 文案类内容 |
| cover | cover | 封面图/缩略图 |
| video | video | 视频内容 |

| topic | topic | 话题/选题 |

### 4.3 标签提取算法

```python
# 伪代码
def extract_tags(comment_text, action_type, content_type):
    """
    从评论中提取特征标签
    """
    tags = []
    
    # 1. 关键词匹配
    keyword_patterns = {
        "tone": ["语气", "口吻", "态度", "调性"],
        "color": ["颜色", "色调", "配色", "色系"],
        "duration": ["时长", "长度", "时间", "秒数"],
        "composition": ["构图", "布局", "排版", "结构"],
        "typography": ["字体", "字号", "字型", "文字"],
        "audio": ["BGM", "背景音乐", "音效", "配音"],
        "topic": ["选题", "话题", "方向", "角度"],
        "style": ["风格", "画风", "调性"],
        "format": ["格式", "尺寸", "比例"],
        "platform": ["平台", "渠道"]
    }
    
    for tag_category, keywords in keyword_patterns.items():
        for kw in keywords:
            if kw in comment_text:
                tags.append({
                    "tag_name": f"{kw}调整",
                    "tag_category": tag_category,
                    "tag_weight": calculate_weight(action_type, severity)
                })
    
    # 2. 操作类型推导标签
    if action_type == "modify":
        tags.append(derive_modification_tag(comment_text, content_type))
    elif action_type == "reject":
        tags.append({"tag_name": "内容拒绝", "tag_category": "format", "tag_weight": 1.0})
    elif action_type == "pass":
        tags.append({"tag_name": "一次通过", "tag_category": "format", "tag_weight": 0.1})
    
    return tags
```

### 4.4 去重与合并

- 同一 task_id 只创建一条 learning_record
- 如果 task 已有记录，仅更新 tags 和 context
- 同一内容多次审阅：保留所有版本记录，用 `production_round` 区分

### 4.5 初始模板导入

首次运行时检测 `learning_records` 表是否为空：
- 若为空 + 提供了 template → 调用 M-28 导入模板
- 若为空 + 无 template → 跳过，等待首次审阅数据
- 若非空 → 正常增量收集

## 六、错误处理

### 6.1 输入校验错误

| 错误码 | 场景 | 处理方式 |
|--------|------|---------|
| E2501 | review_tasks 为空数组 | 跳过处理，返回成功（无数据可收集） |
| E2502 | task_id 格式无效 | 记录错误日志，跳过该条记录 |
| E2503 | content_type 不在枚举范围内 | 归入 "other" 分类，标记需人工审核 |
| E2504 | 必填字段缺失 | 返回错误，不写入数据库 |

### 6.2 数据库写入错误

| 错误码 | 场景 | 处理方式 |
|--------|------|---------|
| E2510 | 数据库连接失败 | 重试3次，间隔1s/3s/5s，仍失败则告警 |
| E2511 | 写入冲突（重复record_id） | 使用 UPSERT 策略，更新已有记录 |
| E2512 | 批量写入部分失败 | 回滚已写入记录，记录失败详情 |

### 6.3 业务异常

| 错误码 | 场景 | 处理方式 |
|--------|------|---------|
| E2520 | 标签提取失败 | 使用默认标签 ["人工修改"]，标记 need_review |
| E2521 | 内容快照获取失败 | 仅记录 context，跳过快照字段 |
| E2522 | 上下文信息不完整 | 用默认值填充缺失字段，标记 partial_context=true |

## 七、自学习

### 7.1 标签库自进化

- 每收集 100 条记录，分析新增标签使用频率
- 高频新标签（>5次）自动加入预定义标签库
- 低频标签（30天内<2次）自动归档

### 7.2 分类准确性优化

- 记录分类修正历史
- 如果某 content_type 被手动修改分类 >3次，更新分类映射规则

### 7.3 采集质量监控

- 统计每条记录的标签数量分布
- 标签为空的记录占比 >20% 时告警
- 记录处理延迟（从审阅完成到记录创建的时间差）

### 8.3 边界条件检查

- [ ] 空审阅列表处理是否正确
- [ ] 超长评论文本（>2000字）是否被截断或分段处理
- [ ] 并发写入时是否有重复记录
- [ ] 同一 task 多次触发是否幂等

### 8.4 性能检查

- [ ] 单次处理 100 条记录耗时 < 5秒
- [ ] 数据库写入使用批量操作
- [ ] 内容快照大小是否合理（<10KB/条）

### 8.5 日志检查

```sql
-- 查询最近的学习记录
SELECT category, action_type, COUNT(*) 
FROM learning_records 
WHERE created_at > NOW() - INTERVAL '1 hour'
GROUP BY category, action_type;

-- 检查未打标签的记录
SELECT record_id, task_id, category 
FROM learning_records 
WHERE tags IS NULL OR jsonb_array_length(tags) = 0;

-- 检查处理延迟
SELECT AVG(EXTRACT(EPOCH FROM (created_at - (SELECT reviewed_at FROM review_tasks WHERE task_id = learning_records.task_id)))) 
AS avg_delay_seconds 
FROM learning_records 
WHERE created_at > NOW() - INTERVAL '1 day';
```

## 九、Hermes Agent 指令

### 9.1 角色定义

```
你是 M-25 学习数据收集模块的 Hermes Agent。
你的职责是：在每次人工审阅完成后，自动收集、分类、标注所有审阅操作，
将非结构化的审阅反馈转化为结构化的学习数据。
```

### 9.2 执行指令

```
当收到来自 M-13 的审阅完成事件时，执行以下步骤：

1. 【数据接收】解析事件中的 review_tasks 和 review_comments 数据。
   验证数据格式完整性，对于格式无效的记录，记录错误日志后跳过。

2. 【分类归档】根据每条审阅任务的 content_type 字段，将记录归入以下类别：
   - copy（文案）
   - cover（封面）
   - video（视频）
   - material（素材）
   - topic（话题）
   - strategy（策略）

3. 【标签提取】对每条审阅评论执行标签提取：
   a. 分析 comment_text，使用关键词匹配识别修改涉及的特征维度
   b. 根据 action_type 自动添加基础标签：
      - "pass" → "一次通过"
      - "modify" → 从评论内容提取具体修改标签
      - "reject" → "内容拒绝" + 拒绝原因标签
      - "supplement" → "补充内容"
   c. 根据 severity 设置标签权重：
      - "minor" → 0.3
      - "moderate" → 0.6
      - "major" → 1.0

4. 【上下文记录】构建 context 对象：
   a. 必填字段：content_type, target_platform, topic_area, production_round
   b. 扩展字段：reviewer_id, auto_generated, time_of_day, day_of_week
   c. 如某字段缺失，使用默认值并标记 partial_context=true

5. 【内容快照】保存原始内容和修改后内容的摘要快照：
   a. text_preview：截取前500字符
   b. key_features：提取3-5个关键特征
   c. diff_summary：简要描述修改差异（仅 modify 操作）

6. 【置信度影响评估】评估本次操作对偏好置信度的影响：
   a. is_positive：pass 或 supplement 为 true，modify/reject 为 false
   b. impact_score：根据 severity 和 action_type 计算
   c. affected_preferences：列出可能受影响的偏好维度

7. 【数据持久化】将处理结果写入 learning_records 表：
   a. 使用 UPSERT 策略避免重复
   b. 批量写入提高性能
   c. 设置 processed_by_m26 = false

8. 【异常处理】遇到以下情况时：
   - 标签提取失败 → 使用 ["人工修改"] 作为默认标签，标记 need_review=true
   - 内容快照获取失败 → 仅记录 context，跳过快照字段
   - 数据库写入失败 → 重试3次后告警

9. 【完成确认】返回处理结果摘要：
   {
     "total_processed": <处理总数>,
     "by_category": { "copy": N, "cover": N, ... },
     "by_action": { "pass": N, "modify": N, "reject": N, "supplement": N },
     "new_tags_discovered": [...],
     "errors": [...]
   }
```

### 9.3 质量约束

```
- 标签提取准确率目标 >90%
- 分类准确率目标 >95%
- 记录创建延迟 <10秒（从收到事件到写入完成）
- 不允许跳过任何有效审阅记录
- 不允许在标签为空的情况下写入记录（pass 操作除外）
```

### 9.4 自检指令

```
每次执行完成后，自检以下项目：
1. 检查 learning_records 表是否有新增记录（对比执行前后 COUNT）
2. 检查是否有 tags 为空的非 pass 记录
3. 检查是否有 context 缺失必填字段的记录
4. 计算本次处理的平均延迟
5. 如发现异常，主动上报并在返回结果中注明
```
