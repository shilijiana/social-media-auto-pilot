### 1.2 核心职责

对从临时素材库（M-02/M-03 产出）中获取的原始素材进行全量质量检测和去重处理。具体包括：图片相似度去重（pHash 算法）、视频清晰度检测（拒绝低于阈值的素材）、水印/版权标识检测（模板匹配）。按类型（图片/视频/音频/文本）进行分类过滤和归档，最终生成「素材采集报告」。确保只有高质量、无重复、无水印问题的素材进入下游内容生产流程。

### 1.3 触发时机

- **自动触发**: M-02（图文素材采集）或 M-03（视频素材采集）完成后，由流水线引擎自动调用。当临时素材库有新增素材时也会触发增量质检
- **人工触发**: 运营人员在素材管理后台上传自定义素材后，手动触发「质量检测」
- **定时触发**: 每日 02:00 UTC 对临时素材库中未质检的素材执行批量质检；每周一 04:00 UTC 执行全量素材库 pHash 重计算（清理历史重复积累）

### 2.3 数据来源

| 字段 | 来源 |
|------|------|
| `task_id` | 流水线引擎生成 |
| `source_materials[].material_ref` | M-02/M-03 输出的临时素材 ID |
| `source_materials[].storage_path` | 临时素材库 OSS 路径（M-02/M-03 上传后返回） |
| `source_materials[].metadata` | M-02/M-03 输出的素材元数据 |
| `source_materials[].existing_hash` | M-02/M-03 预计算的素材哈希（可选） |
| `quality_config` | 配置文件 `config/quality_rules.yaml` 加载，支持运行时覆盖 |
| `quality_config.watermark.watermark_templates_path` | 配置文件路径，存放常见平台水印模板图片 |

## 4. 核心处理逻辑

### 4.1 处理步骤

```
Step 1: 输入校验
  - 做什么: 校验 task_id 格式、source_materials 非空（1-500）、quality_config 参数合法性
  - 怎么判断: 使用 jsonschema 库校验，对比 2.1 中的 Schema
  - 输出: 校验通过 → 进入 Step 2; 校验失败 → 返回 status=failed，记录错误详情

Step 2: 素材分类与格式校验
  - 做什么: 按 type 字段将素材分为 image/video/audio/text 四类，校验文件格式是否在允许列表中
  - 怎么判断:
      - 从 storage_path 中提取文件扩展名
      - 检查扩展名是否在 quality_config.<type>.allowed_formats 中
      - 文件格式不支持 → 标记为 failed，rejection_reasons 添加「格式不支持」
      - 同时校验文件大小是否在 max_file_size_mb 限制内
  - 输出: 分类后的素材列表（含格式校验结果）

Step 3: 分辨率检测
  - 做什么: 对图片和视频素材进行分辨率检测
  - 怎么判断:
      - 图片: 从 metadata.width/height 读取（如无则使用 Pillow 读取实际尺寸）
      - 视频: 从 metadata.width/height 读取（如无则使用 ffprobe 读取实际尺寸）
      - 对比 quality_config.<type>.min_width 和 min_height
      - 分辨率不达标 → 标记为 failed，rejection_reasons 添加「分辨率不达标」
  - 输出: 每个素材的分辨率检测结果

Step 4: 清晰度检测
  - 做什么: 检测图片和视频的清晰度/画质
  - 怎么判断:
      - 图片: 使用 OpenCV 计算 Laplacian 方差（cv2.Laplacian + cv2.meanStdDev）
        - Laplacian 方差 < min_laplacian_variance → 判定为模糊
        - 方差在 [threshold, threshold*1.5] 范围内 → 标记为「边缘质量」，需人工复核
      - 视频: 检查 bitrate（从 metadata.bitrate_bps 读取或使用 ffprobe 获取）
        - bitrate < min_bitrate_mbps * 1000000 → 判定为低画质
        - 同时检查帧率，fps < min_fps → 判定为不流畅
      - 音频: 检查采样率和比特率（如适用）
      - 文本: 检查内容长度（最短 10 字符，最长 10000 字符）
  - 输出: 每个素材的清晰度检测结果

Step 5: 水印与版权标识检测
  - 做什么: 检测图片和视频帧中是否包含水印或版权标识
  - 怎么判断:
      - 如果 watermark.enable_detection = false，跳过此步骤
      - 加载 watermark_templates_path 下的水印模板图片
      - 图片: 使用 OpenCV 模板匹配（cv2.matchTemplate）扫描图片四角和中心区域
        - 匹配分数 > detection_confidence → 判定存在水印
      - 视频: 提取首帧、中间帧、尾帧各 1 张，对每帧执行模板匹配
      - 版权文字检测: 使用 OCR（如 Tesseract 或 PaddleOCR）扫描图片边缘区域
        - 检测到 "copyright"、"版权"、"©"、"TM"、"®" 等文字 → 标记
      - 检测到水印或版权标识 → 标记为 failed
  - 输出: 每个素材的水印检测结果

Step 6: 相似度计算与去重
  - 做什么: 计算素材指纹并检测重复
  - 怎么判断:
      - 图片:
        - 使用 imagehash 库计算 pHash（感知哈希）和 dHash（差异哈希）
        - 将图片缩放至 256x256 后再计算哈希（统一分辨率避免干扰）
        - 计算 Hamming 距离，距离 < (1 - image_similarity_threshold) * 64 → 判定为重复
        - 对于重复组，保留 quality_score 最高的一张
      - 视频:
        - 提取视频的前、中、后 3 个关键帧
        - 对每帧计算 pHash，取平均值作为视频指纹
        - Hamming 距离比较，阈值 video_similarity_threshold
        - 同时比较 duration_seconds 差异（差异 > 10% 则判定为不同视频）
      - 文本:
        - 计算文本的 SimHash 或 MinHash
        - 比较相似度，阈值 text_similarity_threshold
      - 跨类型不去重（图片不与视频比较）
  - 输出: 去重结果，每个素材的 is_duplicate 状态

Step 7: 综合质量评分
  - 做什么: 为每个素材计算综合质量评分
  - 怎么判断:
      - 图片质量分 = 分辨率分(0-30) + 清晰度分(0-30) + 无水印分(0-20) + 格式分(0-10) + 文件大小合理分(0-10)
      - 视频质量分 = 分辨率分(0-25) + 清晰度分(0-25) + 帧率分(0-10) + 无水印分(0-20) + 格式分(0-10) + 时长合理分(0-10)
      - 各项得分基于实际值在阈值范围内的位置进行线性插值计算
      - quality_score >= 60 为通过
  - 输出: 每个素材的 quality_score 和 passed 状态

Step 8: 生成报告与结果写入
  - 做什么: 汇总所有质检结果，生成去重报告和质量报告，写入数据库
  - 怎么判断:
      - 生成 dedup_report: 统计输入总数、发现重复数、保留唯一数、重复组详情
      - 生成 quality_report: 统计通过/失败数量、按类型分组、按失败原因分组
      - 将 passed=true 的素材路径写入 curated 目录映射表
      - 将 passed=false 的素材路径写入 rejected 目录（30 天自动清理）
      - 写入 module_outputs 表、dedup_reports 表、quality_reports 表、material_hashes 表
  - 输出: 最终的 Output JSON
```

## 6. 错误处理策略

| 错误类型 | 处理方式 | 重试策略 | 告警级别 |
|---------|---------|---------|---------|
| 输入 JSON Schema 校验失败 | 返回 status=failed，错误详情写入 `errors` 表，终止执行 | 不重试 | ERROR |
| OSS 素材下载超时（> 60s） | 记录该素材为 failed，继续处理剩余素材 | 3次，指数退避 2s/4s/8s | WARN |
| 图片格式损坏/无法解码 | 记录错误日志，标记该素材为 failed，rejection_reasons 添加「文件损坏」 | 不重试 | ERROR |
| 视频文件损坏/无法解析 | 使用 ffprobe 重试解析，失败则标记为 failed | 重试 1 次（使用不同解码参数） | WARN |
| pHash 计算内存溢出（单张图片 > 100MP） | 自动缩放至最大 4096px 后重试 | 缩放后重试 1 次 | WARN |
| 水印模板文件缺失 | 跳过水印检测步骤，在报告中标注「水印检测未执行」 | 不重试 | WARN |
| OCR 服务不可用（版权文字检测失败） | 降级：跳过版权文字检测，仅进行模板匹配水印检测 | 不重试 | WARN |
| 数据库写入失败（连接断开/锁超时） | 暂存结果到本地 JSON 文件 `data/m05_backup_{task_id}.json`，发送告警 | 5次，间隔 2s | CRITICAL |
| OSS 写入失败（存储空间不足） | 清理 `s3://materials/rejected/` 中超过 30 天的文件，发送告警 | 清理后重试 1 次 | CRITICAL |
| 全部素材质检失败（status=failed） | 通知流水线引擎终止下游模块，写入 `pipeline_errors` 表 | 不重试 | CRITICAL |

### 降级策略

当模块完全不可用时（如 OpenCV 库损坏 + OSS 不可达 + 数据库离线），启用以下降级方案：

1. **跳过质检**: 在 `pipeline_config` 中设置 `quality_controller.skip_on_failure = true`，流水线直接将临时素材库中的素材传递给 M-06，不做去重和质量过滤
2. **人工介入**: 发送企业微信/钉钉通知给运营团队，附上临时素材库路径，由人工手动筛选
3. **使用缓存质检结果**: 如果同一批素材之前执行过质检（基于 material_ref 查询 material_hashes 表），使用缓存的质检结果继续流程

## 7. 自学习触发条件

### 7.1 学习数据收集

本模块向以下表写入数据，供 M-25 定期收集分析：

| 数据字段 | 说明 | 收集时机 |
|---------|------|---------|
| `material_ref` | 素材引用 ID | 每次质检完成后立即写入 |
| `phash` / `dhash` | 素材感知哈希指纹 | pHash 计算完成后写入 material_hashes 表 |
| `quality_score` | 综合质量评分 | 质检完成后写入 |
| `clarity_score` | 清晰度评分 | 清晰度检测完成后写入 |
| `watermark_detected` | 是否检测到水印 | 水印检测完成后写入 |
| `content_performance` | 该素材制成内容后的表现（互动率等） | 内容发布后 7 天内，从 M-22 获取 |
| `effective_threshold` | 该素材类型实际产生高互动的最低质量分数 | M-22 分析后回写 |
| `threshold_adjustment` | M-25 建议的质量阈值调整值 | M-25 分析后回写 |

### 7.2 偏好库调用

- **调取偏好类型**: 质量偏好（用户对素材清晰度、风格、来源平台的偏好）
- **置信度阈值**: >= 70%
- **无偏好时默认行为**: 使用 `config/quality_rules.yaml` 中的默认阈值，不做个性化调整

### 7.3 阶段演进行为

| 阶段 | 行为差异 |
|------|---------|
| P0 (人工主导) | 质检完全由人工完成。模块仅提供素材列表展示和手动标记界面（通过/拒绝/待定）。质量阈值和去重规则由运营人员在后台手动配置。 |
| P1 (人机协同) | 模块自动执行所有质检项并生成质量评分，但最终通过/拒绝由人工确认。特别是边缘质量素材（quality_score 在 50-70 之间）需要人工判断。人工可覆盖自动判定结果。 |
| P2 (系统主导) | 模块全自动执行质检和决策。仅当质量评分处于边缘区间（55-65）或检测到新型水印（不在模板库中）时通知人工审核。去重自动执行，不通知。M-25 根据历史表现数据自动调整质量阈值。 |

### 验收标准
- [ ] 输入校验通过: 空 source_materials 或非法 quality_config 应返回 status=failed
- [ ] 核心逻辑无异常: dry-run 模式下所有步骤执行无 Python Exception
- [ ] 输出格式匹配 Schema: 包含 quality_results、dedup_report、quality_report、execution_metadata
- [ ] 去重逻辑正确: 相似图片（pHash Hamming 距离 < 阈值）被正确识别并去重
- [ ] 清晰度检测准确: 模糊图片 Laplacian 方差 < 100，清晰图片 > 100
- [ ] 视频质量检测正确: bitrate < 2Mbps 的视频被拒绝
- [ ] 水印检测生效: 带水印图片被正确识别（detected=true）
- [ ] 格式校验正确: 不在 allowed_formats 中的文件被拒绝
- [ ] 质量评分合理: quality_score 在 0-100 范围内，60 分以上为通过
- [ ] 错误场景有降级处理: 模拟 OSS 不可用，确认模块返回 status=partial 并记录错误
- [ ] 日志输出完整: 检查日志包含 Step 1-8 的执行记录和每步耗时统计
- [ ] 内存使用正常: 处理 500 张 4K 图片时内存峰值 < 4GB

### 注意事项
- dry-run 模式不会移动 OSS 文件或写入数据库，仅输出到本地 JSON 文件，适用于开发调试
- pHash 计算对大图片（> 4096px）会自动缩放，确保内存使用可控
- 水印模板匹配可能误报，建议 detection_confidence 设为 0.85 以上
- 视频清晰度检测需要 ffprobe 可用，确认 `which ffprobe` 返回有效路径
- OCR 版权文字检测是可选的，如果 Tesseract/PaddleOCR 未安装，设置 `watermark.detect_copyright_text = false`
- 去重仅在相同类型（image/image、video/video）内执行，不会跨类型去重
- 首次运行前确保水印模板目录已创建: `mkdir -p config/watermark_templates/`
- 各平台水印模板可从公开资源获取，或手动截图平台 logo 作为模板
- 若遇到 `ModuleNotFoundError: No module named 'cv2'`，执行 `pip install opencv-python-headless>=4.8`
- rejected 目录下的素材会在 30 天后自动清理（通过定时任务 `cleanup_rejected_materials` 执行）
