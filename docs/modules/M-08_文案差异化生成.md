```
  上游: M-07（素材匹配与调用模块）→ 传入匹配完成的素材列表和话题信息
  下游: M-11（内容精加工与定稿模块）→ 使用生成的文案进行精加工
        M-12（合规性自检模块）→ 对生成的文案进行合规检查
  自学习: M-25 收集各平台文案的表现数据，优化风格偏好库
```

### 1.2 核心职责

根据匹配到的素材和话题信息，为 4 个目标平台（抖音、小红书、B站、X/Twitter）生成差异化文案。每个平台的文案需严格遵循其平台调性、用户阅读习惯和字数限制。调用风格偏好库（M-26 管理）中的用户偏好数据，确保文案风格与账号历史调性一致。生成的四套文案写入 `content_drafts` 表，供后续精加工和审阅使用。

### 1.3 触发时机

- **自动触发**: M-07（素材匹配与调用模块）完成后，由流水线引擎自动调用。传入匹配结果和话题信息
- **人工触发**: 运营人员在内容管理后台对某个话题手动点击「生成文案」或对已有文案点击「重新生成」
- **批量触发**: 支持一次性为多个话题生成文案（如批量生成本周所有话题的初版文案）

### 2.3 数据来源

| 字段 | 来源 |
|------|------|
| `task_id` | 流水线引擎生成 |
| `topic_plan_id` | M-07 透传（来自 topics 表） |
| `matched_materials` | M-07（素材匹配与调用模块）输出中的 matched_materials 数组 |
| `target_platforms` | 分发计划配置 `config/distribution_plan.yaml`，可运行时覆盖 |
| `style_preferences` | M-26（偏好提炼与规则生成模块）管理的风格偏好库，通过 `preference_api.get_style(topic_tags, platforms)` 查询 |
| `generation_options` | 配置文件 `config/copy_generation.yaml` 加载，支持运行时覆盖 |

## 4. 核心处理逻辑

### 4.1 处理步骤

```
Step 1: 输入校验与话题加载
  - 做什么: 校验输入格式，加载话题完整信息
  - 怎么判断:
      - 使用 jsonschema 库校验输入格式
      - 查询 topics 表获取话题完整信息: title, description, tags, topic_type
      - 校验 matched_materials 非空
      - 校验 target_platforms 在 [douyin, xiaohongshu, bilibili, x] 范围内
  - 输出: 话题完整信息（topic_info）、素材上下文（material_context）

Step 2: 风格偏好库查询
  - 做什么: 从 M-26 偏好库加载各平台的风格配置
  - 怎么判断:
      - 如果输入已包含 style_preferences 且 confidence >= 0.7，直接使用
      - 如果未提供或 confidence < 0.7，调用 M-26 API: preference_api.get_style(topic_tags, platforms)
      - 如果偏好库无匹配结果（新账号或无历史数据），使用 config/copy_templates.yaml 中的默认风格模板
      - 风格配置包含: tone, title_patterns, hook_patterns, hashtag_habits, cta_patterns, emoji_usage
  - 输出: 每个平台的风格配置（platform_styles）

Step 3: Prompt 模板选择与上下文注入
  - 做什么: 为每个平台选择对应的 Prompt 模板，注入素材和风格上下文
  - 怎么判断:
      - 根据平台 + tone 组合选择 Prompt 模板:
        - douyin: prompts/copywriting/douyin_{tone}.j2
        - xiaohongshu: prompts/copywriting/xiaohongshu_{tone}.j2
        - bilibili: prompts/copywriting/bilibili_{tone}.j2
        - x: prompts/copywriting/x_{tone}.j2
      - 模板不存在时，使用对应平台的 default.j2 兜底模板
      - 使用 Jinja2 渲染模板，注入以下上下文:
        - topic_info: 话题标题、描述、标签
        - material_context: 素材列表摘要（material_type, content_tags, relevance_score）
        - style_config: 该平台的风格配置
        - platform_rules: 平台规则（字数限制、格式要求等）
  - 输出: 每个平台的渲染后的完整 Prompt（rendered_prompts）

Step 4: 各平台文案生成（并行执行）
  - 做什么: 为每个平台并行调用 LLM 生成文案
  - 怎么判断:
      - 对 target_platforms 中的每个平台，启动异步 LLM 调用
      - 使用 asyncio.gather() 并行执行，提高效率
      - LLM 参数:
        - model: gemini-2.5-flash（默认），支持配置覆盖
        - temperature: generation_options.temperature
        - max_output_tokens: 根据平台限制动态设置（douyin=800, xiaohongshu=1500, bilibili=3000, x=500）
      - 每个平台最多重试 max_retries_per_platform 次
      - 重试时使用递增 temperature（+0.1/次）增加多样性
  - 输出: 每个平台的 LLM 原始响应（llm_responses）

Step 5: 响应解析与结构提取
  - 做什么: 解析 LLM 返回的 JSON 响应，提取文案结构
  - 怎么判断:
      - LLM 响应格式要求: JSON，包含 title, body, hashtags, cta（字段名因平台而异）
      - 使用正则表达式从响应中提取 JSON 块
      - JSON 解析失败时:
        - 第 1 次: 返回错误信息给 LLM，要求重新输出合法 JSON
        - 第 2 次: 使用启发式规则从文本中提取 title/body/hashtags
        - 第 3 次: 标记该平台为 failed
      - 提取成功后，补充平台特定字段:
        - x 平台: 自动生成 alt_text（图片描述，用于无障碍访问）
        - xiaohongshu: 确保 body 中有 emoji 分隔符
        - bilibili: 确保有分节标题（## 格式）
  - 输出: 每个平台的解析后的文案结构（parsed_drafts）

Step 6: 长度与格式校验
  - 做什么: 校验每个平台的文案是否符合字数限制和格式要求
  - 怎么判断:
      - 计算 char_count = len(title) + len(body) + sum(len(h) for h in hashtags)
      - 平台限制对照:
        - douyin: char_count <= 500
        - xiaohongshu: char_count <= 1000
        - bilibili: char_count <= 2000
        - x: char_count <= 280
      - 如果 char_count > platform_limit:
        - auto_truncate=true: 自动截断 body 至限制长度，添加 warning
        - auto_truncate=false: 标记 length_valid=false，添加 warning: "文案超出平台限制 X 字符"
      - Emoji 数量校验（仅警告，不阻塞）:
        - xiaohongshu: 建议 >= 5 个 emoji
        - douyin: 建议 1-5 个 emoji
        - bilibili: 建议 0-3 个 emoji
        - x: 建议 0 个 emoji
      - 格式校验:
        - xiaohongshu: 检查是否有 emoji 分隔符和分段结构
        - bilibili: 检查是否有 ## 分节标题
  - 输出: 每个平台的 metadata（含校验结果和警告）

Step 7: 内容质量评估
  - 做什么: 对生成的文案进行基础质量评估
  - 怎么判断:
      - 调用 LLM 进行自评（使用 prompts/copywriting/quality_check.j2）
      - 评估维度:
        - 与话题的关联度（relevance_score, 0-100）
        - 平台调性匹配度（platform_fit_score, 0-100）
        - 可读性（readability_score, 0-100）
        - 原创性（originality_score, 0-100）
      - 任一维度 < 60 分: 标记 warning，提示人工审核
      - 多维度 < 50 分: 触发重新生成（最多 1 次额外重试）
  - 输出: 每个平台的质量评估结果（quality_scores），写入 metadata

Step 8: 结果写入与去重
  - 做什么: 将生成的文案写入 content_drafts 表，检查去重
  - 怎么判断:
      - 为每个平台生成唯一 draft_id（格式: DFT-YYYY-NNNNNN）
      - 检查 content_drafts 表中是否有相似文案（body 相似度 > 80%）:
        - 使用 pgvector 向量相似度查询
        - 如有相似文案，添加 warning: "检测到与历史文案 DFT-XXXX-XXXXXX 相似度 XX%"
      - 写入 content_drafts 表
      - 写入 llm_call_logs 表（LLM 调用日志）
      - 写入 module_outputs 表
      - 计算整体 status:
        - 全部平台成功 → completed
        - 部分平台成功 → partial
        - 全部平台失败 → failed
  - 输出: 最终的 Output JSON
```

### 4.2 平台差异化规则

| 维度 | 抖音 | 小红书 | B站 | X (Twitter) |
|------|------|--------|-----|-------------|

| **标题风格** | 强钩子 + 口语化 | 干货标题 + Emoji 前缀 | 深度导语 + 结论前置 | 极简观点 |
| **正文结构** | 短句分行 + 数字列表 | Emoji 分段 + 干货列表 | 分节标题(##) + 论证展开 | 单段观点 + 短标签 |

| **话题标签** | 热搜标签 + 垂类标签 | 长尾关键词标签 | 内容分类标签 | 简短标签(1-2个) |

| **特有要求** | 视频口播感 | 分段 + 干货标记 | 深度论证 + 引用 | alt_text（图片描述） |

## 6. 错误处理策略

| 错误类型 | 处理方式 | 重试策略 | 告警级别 |
|---------|---------|---------|---------|
| 输入 JSON Schema 校验失败 | 返回 status=failed，错误详情写入 `errors` 表，终止执行 | 不重试 | ERROR |
| matched_materials 为空 | 返回 status=failed，错误信息: "素材列表为空，无法生成文案" | 不重试 | ERROR |
| 风格偏好库查询超时（> 5s） | 降级：使用 `config/copy_templates.yaml` 中的默认风格模板 | 不重试 | WARN |
| LLM API 返回 429 (Rate Limit) | 等待 Retry-After 头指定的时间后重试 | 3次，间隔递增 | WARN |
| LLM API 返回 500/503 | 切换至降级模型 `gemini-2.5-flash-lite` 重试 | 主模型 2 次 → 降级模型 2 次 | WARN |
| LLM 响应 JSON 解析失败 | 第1次: 反馈错误要求重新输出；第2次: 启发式文本提取；第3次: 标记失败 | 最多 3 次 | WARN |
| 某平台生成失败（重试耗尽） | 该平台标记为 failed，其他平台继续生成。status=partial | 不重试（该平台） | WARN |
| 全部平台生成失败 | 返回 status=failed，通知运营人员手动撰写 | 不重试 | CRITICAL |
| 文案超长（validate_length=true） | 标记 length_valid=false，添加 warning，不阻塞流程 | 不重试（除非 auto_truncate=true） | INFO |
| 数据库写入失败（连接断开/锁超时） | 暂存结果到本地 JSON 文件 `data/m08_backup_{task_id}.json` | 5次，间隔 2s | CRITICAL |
| pgvector 相似度检测超时 | 跳过去重检测，添加 warning: "未执行去重检测" | 不重试 | WARN |
| LLM Token 超限 | 自动压缩 material_context（取前 5 个素材的 content_tags 摘要）后重试 | 1 次 | WARN |

### 降级策略

当模块完全不可用时（如 LLM 服务中断），启用以下降级方案：

1. **模板填充**: 使用预定义的文案模板 + 素材标签进行简单填充生成。文案质量较低，仅作为应急使用，标记 `source = "template_fallback"`
2. **人工撰写**: 发送企业微信/钉钉通知给运营团队，附上话题信息和素材列表，由人工撰写初版文案
3. **跳过本模块**: 在 `pipeline_config` 中设置 `copy_generator.skip_on_failure = true`，下游模块使用空文案占位，待人工补充

## 7. 自学习触发条件

### 7.1 学习数据收集

本模块向以下表和字段写入数据，供 M-25/M-26 定期收集分析：

| 数据字段 | 说明 | 收集时机 |
|---------|------|---------|
| `draft_id` | 文案草稿 ID | 生成时写入 |
| `platform` | 目标平台 | 生成时写入 |
| `tone` | 实际使用的语气风格 | 生成时写入 |
| `prompt_template_used` | 使用的 Prompt 模板 | 生成时写入 |
| `style_confidence` | 风格偏好库匹配置信度 | 生成时写入 |
| `quality_scores` | 内容质量评估得分（relevance, platform_fit, readability, originality） | 生成时写入（metadata JSONB） |
| `final_content_used` | 精加工后最终使用的文案版本 | M-11 完成后回写 |
| `edit_distance` | 初版到最终版的编辑距离（衡量人工修改程度） | M-11 完成后计算并回写 |
| `content_performance` | 发布后互动数据（点赞、评论、转发、收藏等） | 内容发布后 7 天内，从 M-22 获取 |
| `platform_engagement_rate` | 各平台该文案的互动率 | M-22 计算后回写 |
| `ab_test_variant` | A/B 测试分组（如果生成了多个文案变体） | M-11 选择变体后回写 |

### 7.2 偏好库调用

- **调取偏好类型**: 风格偏好（tone、title_patterns、hashtag_habits、hook_patterns、cta_patterns、emoji_usage）
- **置信度阈值**: >= 70%
- **无偏好时默认行为**: 使用 `config/copy_templates.yaml` 中的 `default_style_by_topic_type` 映射，根据话题类型（技术/生活/娱乐等）选择默认风格模板

### 7.3 阶段演进行为

| 阶段 | 行为差异 |
|------|---------|
| P0 (人工主导) | 文案完全由运营人员手动撰写。模块仅负责将手动输入的文案格式化存储到 content_drafts 表。Prompt 模板不启用。风格偏好库为空。 |
| P1 (人机协同) | 模块自动生成初版文案，运营人员在后台审核和修改。LLM 调用启用。风格偏好库开始积累数据（收集人工修改记录）。修改率高的平台/风格会触发偏好更新。 |
| P2 (系统主导) | 模块全自动生成文案，仅当 quality_scores 中任一维度 < 60 或人工设置了强制审核标记时才通知人工。M-26 根据历史高表现文案自动优化 Prompt 模板。支持 A/B 测试（生成 2 个变体，发布后自动选优）。 |
