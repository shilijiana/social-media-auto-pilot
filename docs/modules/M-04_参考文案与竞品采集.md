### 1.2 核心职责

采集同领域优质账号的高互动标题和文案内容（每平台 5-10 个账号），收集竞品爆款内容（标题、话题标签、互动数据）。对采集到的文案进行去重、主题分类、可复用句式提取，最终生成结构化的「参考文案库」。该文案库为 M-08 内容排版模块提供标题风格参考和句式模板，帮助生成符合平台调性和竞品水平的文案内容。

### 1.3 触发时机

- **自动触发**: M-01（关键词智能扩展）完成后，由流水线引擎自动调用。若关键词池有新增热门关键词，也会触发增量采集
- **人工触发**: 运营人员在竞品管理后台添加新竞品账号后，手动触发「采集该账号历史爆款文案」
- **定时触发**: 每日 06:00 UTC 执行全量竞品爆款文案采集（覆盖过去 24 小时新增内容）；每周一 03:00 UTC 执行句式模式重新提取（配合 M-25 偏好更新）

## 4. 核心处理逻辑

### 4.1 处理步骤

```
Step 1: 输入校验
  - 做什么: 校验 task_id 格式、keywords 数组非空（1-50）、competitor_accounts 非空（1-50）、target_platforms 合法
  - 怎么判断: 使用 jsonschema 库校验，对比 2.1 中的 Schema
  - 输出: 校验通过 → 进入 Step 2; 校验失败 → 返回 status=failed，记录错误详情

Step 2: 平台搜索接口调用
  - 做什么: 根据关键词在每个目标平台上搜索热门内容
  - 怎么判断:
      - 读取 config/platform_search_config.yaml 获取各平台搜索 API 端点
      - 对每个关键词按 weight 降序排列后依次搜索
      - 每个关键词最多返回 max_items_per_keyword 条结果
      - 搜索结果按互动量（点赞+评论+分享）降序排列
      - 仅采集 interaction > engagement_threshold 的内容
  - 输出: 搜索结果列表（含标题、正文、互动数据、话题标签、发布日期、原始链接）

Step 3: 竞品账号内容拉取
  - 做什么: 拉取竞品账号在 date_range_days 范围内的历史高互动内容
  - 怎么判断:
      - 按 priority 降序排列竞品账号，优先处理高优先级账号
      - 调用各平台用户内容 API，拉取最近 date_range_days 天的内容
      - 每个账号最多拉取 max_items_per_account 条
      - 按互动量排序，仅保留互动量 > engagement_threshold 的内容
      - 对已拉取的内容检查是否与 Step 2 搜索结果重复（基于 URL 去重）
  - 输出: 竞品文案列表（含账号信息、内容详情、互动数据）

Step 4: 内容去重
  - 做什么: 对采集到的所有文案进行去重处理
  - 怎么判断:
      - 计算每条文案的 SHA-256 哈希值（基于 content_text 去除空白和 emoji 后的标准化文本）
      - 相同哈希值的文案仅保留互动量最高的一条
      - 对于高度相似但非完全一致的文案（编辑距离 < 5 且长度差异 < 10%），调用 LLM 判断是否为同义内容
      - 相似文案保留互动量较高的版本
  - 输出: 去重后的文案列表，记录去重统计（total_items, unique_items, dedup_removed）

Step 5: 主题分类
  - 做什么: 对去重后的文案进行主题分类
  - 怎么判断:
      - 调用 LLM 对每条文案进行分类，分类维度包括:
        - 主题分类: 「标题模板」「互动话术」「话题标签组合」「开头钩子」「结尾引导」「痛点描述」「利益点突出」
        - 内容领域: 基于 content_tags 和关键词推断
        - 情绪基调: 正面/中性/负面/好奇/紧迫
      - 使用 prompt: `prompts/copy_classification.md`
      - LLM 分类置信度 < 0.7 时标记为「未分类」，待人工审核
  - 输出: 每条文案的分类标签

Step 6: 句式模式提取（LLM）
  - 做什么: 从采集的文案中提取可复用的句式模板
  - 怎么判断:
      - 如果 enable_sentence_extraction = false，跳过此步骤
      - 将同主题分类下的文案批量发送给 LLM
      - LLM 识别重复出现的句式结构，将具体内容替换为 [变量] 占位符
      - 句式模式分为: title_hook（标题钩子）、cta_ending（结尾行动号召）、pain_point（痛点描述）、benefit_highlight（利益点突出）、question_opening（疑问开头）、number_list（数字列表）、emotional_trigger（情绪触发）
      - 统计每个句式模板的使用频次和平均互动量
      - 使用 prompt: `prompts/sentence_pattern_extraction.md`
  - 输出: 句式模板列表（含分类、频次、互动数据、使用示例）

Step 7: 话题标签组合分析
  - 做什么: 提取高频且高互动的话题标签组合
  - 怎么判断:
      - 统计所有采集文案中出现的话题标签组合（2-5 个标签为一组）
      - 按出现频次和平均互动量综合排序
      - 筛选出频次 >= 3 且平均互动量超过平台中位数的组合
      - 生成按平台分类的话题标签推荐列表
  - 输出: 话题标签组合推荐列表

Step 8: 参考文案库生成与写入
  - 做什么: 将所有采集和分析结果写入数据库
  - 怎么判断:
      - 将去重后的文案写入 materials 表（type=text），按 platform 和 topic_category 组织
      - 将句式模板写入 sentence_patterns 表
      - 将话题标签组合写入 hashtag_library 表
      - 将标题模板写入 title_templates 表
      - 写入 module_outputs 表
      - 为每条文案生成 reusable_score（基于互动量、句式复用度、新鲜度综合计算）
  - 输出: 最终的 Output JSON
```

## 6. 错误处理策略

| 错误类型 | 处理方式 | 重试策略 | 告警级别 |
|---------|---------|---------|---------|
| 输入 JSON Schema 校验失败 | 返回 status=failed，错误详情写入 `errors` 表，终止执行 | 不重试 | ERROR |
| 平台搜索 API 超时（> 30s） | 记录该关键词搜索失败，继续处理剩余关键词 | 3次，指数退避 2s/4s/8s | WARN |
| 平台搜索 API 返回 429（频率限制） | 等待 Retry-After 头指定的时间，暂停该平台后续请求 60s | 按 Retry-After 重试，最多 5 次 | WARN |
| 竞品账号内容拉取失败（账号不存在/被封禁） | 记录该账号为 failed，在 competitor_accounts 表中标记状态为 inactive | 不重试 | WARN |
| 竞品账号内容拉取 403（权限不足） | 记录日志，标记该账号状态为 restricted | 不重试 | WARN |
| LLM 文案分类超时（> 10s/条） | 降级：使用规则匹配（基于关键词和 hashtag 分类） | 重试 1 次 | WARN |
| LLM 句式提取返回异常（500/429/超时） | 降级：跳过句式提取步骤，仅保留原始文案采集结果。status 标记为 partial | 重试 1 次 | WARN |
| 数据库写入失败（连接断开/锁超时） | 暂存结果到本地 JSON 文件 `data/m04_backup_{task_id}.json`，发送告警 | 5次，间隔 2s | CRITICAL |
| 去重哈希计算内存溢出（单条文案 > 100KB） | 截断文案至前 10000 字符后计算哈希，记录截断警告 | 不重试 | WARN |
| 全部平台采集失败（status=failed） | 通知流水线引擎终止下游模块，写入 `pipeline_errors` 表 | 不重试 | CRITICAL |

### 降级策略

当模块完全不可用时（如所有平台 API 不可达 + LLM 服务中断），启用以下降级方案：

1. **静态文案库兜底**: 使用 `config/static_reference_copy.yaml` 中预置的通用文案模板和句式，确保 M-08 有基础参考数据可用
2. **人工介入**: 发送企业微信/钉钉通知给运营团队，附上失败详情，由人工手动导入最近一周的竞品文案数据
3. **跳过本模块**: 在 `pipeline_config` 中设置 `reference_copy_collector.skip_on_failure = true`，流水线使用上一次成功的文案库缓存继续后续流程（缓存有效期 7 天）
