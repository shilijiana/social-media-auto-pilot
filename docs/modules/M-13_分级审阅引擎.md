### 1.2 核心职责

分级审阅引擎是整个系统渐进式自动化的**核心组件**。它不直接生产内容，而是作为一个决策中枢，动态决定内容生产流程中 5 个审阅节点的执行模式（A/B/C）。通过持续收集审阅数据和运营人员反馈，配合 M-25 自学习模块的成熟度评估，逐步从「人工全量审核」（模式 A）过渡到「系统自动审批 + 人工抽检」（模式 C）。**关键原则：人工始终可以越级干预任何节点，无论当前处于何种模式。**

### 1.3 触发时机

- **自动触发**: 当内容生产流程到达任意审阅节点时，由流水线引擎自动调用 M-13 决定该节点的审阅模式并执行审阅
- **人工触发**: 运营人员在审阅管理后台手动对某个内容发起审阅（不受当前模式限制）
- **定时触发**: 每周一 00:00 UTC 执行全量审阅模式重评估——根据最新的自学习数据，重新计算各节点的模式升级条件
- **紧急触发**: 当检测到内容异常（如合规风险、发布失败等），自动触发对应节点的紧急审阅（强制模式 A）

### 2.3 数据来源

| 字段 | 来源 |
|------|------|
| `task_id` | 流水线引擎生成 |
| `review_node` | 流水线引擎根据当前生产阶段确定 |
| `content_ref` | 上游模块输出（M-24 → 审阅①; M-07 → 审阅②-A; M-08 → 审阅②-B; M-11 → 审阅③; M-22 → 审阅④; M-24 → 审阅⑤） |
| `current_phase` | 系统配置表 `system_phase`，由上线日期和当前日期计算 |
| `preference_library_stats` | M-25（学习数据收集）+ M-26（偏好提炼与规则生成）定期汇总 |
| `review_history` | `review_tasks` 表聚合查询 |
| `manual_override` | 运营人员在审阅后台手动设置 |
| `review_options` | 配置文件 `config/tiered_review.yaml` |

| 下游消费 | 审阅②-A pass → 触发 M-08（文案差异化生成） |
| 下游消费 | 审阅②-B pass → 触发 M-11（内容精加工与定稿） |
| 下游消费 | 审阅③ pass → 触发 M-16（多平台分发） |
| 下游消费 | 审阅④ pass → 触发 M-24（下期计划草案生成） |
| 下游消费 | 审阅⑤ pass → 确认下期计划并排期 |

### 4.3 处理步骤（状态机逻辑）

```
Step 1: 输入校验与上下文加载
  - 做什么: 校验输入格式，加载审阅所需的完整上下文
  - 怎么判断:
      - 使用 jsonschema 库校验输入
      - 根据 review_node 加载对应内容:
        - review_1_plan → 查询 topics 表
        - review_2a_material → 查询 material_matches 表
        - review_2b_draft → 查询 content_drafts 表
        - review_3_final → 查询 final_content 表
        - review_4_report → 查询 data_reports 表
        - review_5_next_plan → 查询 next_plans 表
      - 加载该节点的 review_history（最近 50 条）
      - 加载 preference_library_stats（如果输入未提供，调用 M-25 API 获取）
  - 输出: 完整审阅上下文（review_context）

Step 2: 人工越级检查
  - 做什么: 检查是否有人工越级指令
  - 怎么判断:
      - 如果 manual_override.enabled = true:
        - 强制使用 manual_override.force_mode
        - mode_source = "manual_override"
        - 记录越级日志: override_log 表
        - 跳过 Step 3（模式判定），直接进入 Step 4（审阅执行）
      - 如果 manual_override.enabled = false:
        - 继续 Step 3
  - 输出: 是否被越级（is_overridden）+ 强制模式

Step 3: 模式判定（核心状态机）
  - 做什么: 根据当前阶段、偏好库统计、历史数据，判定本次审阅模式
  - 怎么判断:
      3.1 基于时间的基础模式判定:
          - system_week <= 4 → 基础模式 = A
          - 5 <= system_week <= 12 → 基础模式 = B
          - system_week >= 13 → 基础模式 = C
      
      3.2 各节点的升级条件评估:
          
          审阅① (review_1_plan):
            - B 升级条件: preference_library_stats.style_confidence > 0.80 
                          AND preference_library_stats.consecutive_correct_rounds >= 3
            - C 升级条件: 不适用（计划表始终不可全自动）
            - 注意: 审阅① 有特殊性——即使在 Week 13+，最高只能到模式 B
          
          审阅②-A (review_2a_material):
            - B 升级条件: preference_library_stats.material_filter_confidence > 0.85
            - C 升级条件: preference_library_stats.material_filter_confidence > 0.95
                          AND review_history 中连续 5 轮 decision='pass' 且无人工纠正
            - 注意: 素材审阅涉及版权风险，升级条件比文案审阅更严格
          
          审阅②-B (review_2b_draft):
            - B 升级条件: preference_library_stats.style_coverage > 0.70
            - C 升级条件: preference_library_stats.style_coverage > 0.90
            - 注意: 覆盖率不足时不应自动审阅，避免风格偏离
          
          审阅③ (review_3_final):
            - B 升级条件: system_week >= 5（时间驱动）
            - C 升级条件: preference_library_stats.first_pass_rate > 0.80
                          AND preference_library_stats.first_pass_consecutive_rounds >= 5
            - 注意: 成品审阅是最关键的节点，C 模式条件最严格
          
          审阅④ (review_4_report):
            - B 升级条件: system_week >= 5（时间驱动）
            - C 升级条件: system_week >= 9（时间驱动，约 8 周后）
            - 注意: 数据报告审阅主要靠时间驱动，数据质量由 M-22 保证
          
          审阅⑤ (review_5_next_plan):
            - B 升级条件: system_week >= 5（时间驱动）
            - C 升级条件: preference_library_stats.adoption_rate > 0.75
                          AND preference_library_stats.adoption_consecutive_rounds >= 4
            - 注意: 计划采纳率是核心指标
      
      3.3 模式 = min(基础模式, 升级条件允许的最高模式)
          - 例如: system_week=8 (基础=B), 审阅②-B style_coverage=0.72 (>0.70), 最终=B
          - 例如: system_week=14 (基础=C), 审阅③ first_pass_rate=0.75 (<0.80), 最终=B
      
      3.4 降级检查:
          - 审阅②-A: 最近 10 条中有版权风险素材被系统误判通过 → 强制降级到 B
          - 审阅②-B: 最近 10 条中人工修改率 > 30% → 强制降级到 B
          - 审阅③: 最近 10 条发布后负面反馈率 > 阈值 → 强制降级到 B
          - 审阅⑤: 最近 4 轮 adoption_rate < 60% → 强制降级到 B
      
      3.5 风险升级检查:
          - 如果 review_options.escalation_on_risk = true:
            - 内容包含 critical/high 风险标记 → 强制升级到 A
            - 内容包含 medium 风险标记 → 最高 B
      
      3.6 记录模式判定过程:
          - 写入 mode_escalation_log 表
          - 如果模式发生变更（与 previous_mode 不同），发送通知
  - 输出: determined_mode, mode_source, escalation_conditions_met

Step 4: 审阅执行
  - 做什么: 根据判定模式执行审阅
  - 怎么判断:
      
      4.1 模式 A (人工全量审核):
          - reviewer_type = "human"
          - 系统不参与审阅决策
          - 生成审阅任务: 写入 review_tasks 表，status='pending_human_review'
          - 发送通知给运营人员（企业微信/钉钉）
          - 审阅结果由运营人员在后台手动提交
          - 等待人工操作（不阻塞流水线，但下游模块等待此审阅通过后才执行）
          - confidence = 1.0（人工决策）
      
      4.2 模式 B (系统建议 + 人工确认):
          - reviewer_type = "hybrid"
          - 系统执行自动审阅:
            a. 调用 LLM (gemini-2.5-flash) 进行内容评估
            b. 使用 prompts/review/{review_node}_review.j2 模板
            c. 评估维度因节点而异（见 4.4 各节点评估维度）
            d. 生成系统建议: result + confidence + comments
          - 写入 review_tasks 表，status='pending_human_confirmation'
          - 发送通知给运营人员: "系统建议 [pass/reject/modify]（置信度 XX%），请在 {auto_approve_timeout_hours} 小时内确认"
          - 等待人工确认:
            - 人工确认通过 → 以人工确认为准
            - 人工纠正 → 记录纠正数据到 learning_feedback
            - 超时未确认 → 自动按系统建议执行，标记 auto_confirmed=true
          - 记录 learning_feedback.was_correct（人工确认后回填）
      
      4.3 模式 C (系统自动审批 + 人工抽检):
          - reviewer_type = "system"
          - 系统执行自动审阅（同模式 B 的审阅逻辑）
          - 直接写入 review_tasks 表，status='auto_approved' 或 'auto_rejected'
          - 抽检逻辑:
            - 按 spot_check_rate_c 比例随机抽取
            - 抽检算法: 加权随机（近期有纠正记录的节点权重更高）
            - 被抽中: status='pending_spot_check', 通知运营人员抽检
            - 未被抽中: 自动生效，通知运营人员（信息性通知，无需操作）
          - 抽检结果处理:
            - 抽检通过 → 维持原决定
            - 抽检不通过 → 升级到模式 B 重新审阅，并触发降级评估
          - confidence = 系统算法给出（通常 0.80-0.95）
      
      4.4 风险标记检测:
          - 在模式 B 和 C 中，系统自动检测风险标记:
            - 合规风险: 敏感词检测（调用 M-12 接口）
            - 质量风险: 内容质量评分 < 阈值
            - 品牌安全: 负面/争议内容检测
            - 版权风险: 素材版权即将到期或未知
            - 敏感话题: 涉及政治/社会敏感话题
          - 检测到 high/critical 风险 → 触发风险升级（强制模式 A）
  - 输出: decision 对象

Step 5: 结果持久化与通知
  - 做什么: 将审阅结果写入数据库，发送通知
  - 怎么判断:
      - 生成 review_id（格式: REV-YYYY-NNNNNN）
      - 写入 review_tasks 表
      - 写入 review_comments 表（审阅意见）
      - 如有风险标记，写入 review_risk_flags 表
      - 写入 review_learning_data 表（供 M-25 消费）
      - 写入 module_outputs 表
      - 根据决策结果触发下游模块:
        - pass → 通知流水线引擎继续下一阶段
        - reject → 通知上游模块重新生成，记录拒绝原因
        - modify → 通知上游模块按修改指示调整
      - 发送通知:
        - 模式变更通知（模式 A→B, B→C 等）
        - 待审阅通知（模式 A/B 需要人工操作时）
        - 审阅完成通知（模式 C 自动审批后）
  - 输出: 最终的 Output JSON
```

### 4.4 各节点系统审阅评估维度（模式 B/C 使用）

## 6. 错误处理策略

| 错误类型 | 处理方式 | 重试策略 | 告警级别 |
|---------|---------|---------|---------|
| 输入 JSON Schema 校验失败 | 返回 status=failed，终止执行 | 不重试 | ERROR |
| 无效的 review_node | 返回 status=failed，错误信息: "未知的审阅节点" | 不重试 | ERROR |
| content_ref 对应的内容不存在 | 返回 status=failed，错误信息: "待审阅内容不存在" | 不重试 | ERROR |
| 偏好库统计查询超时（> 5s） | 降级：使用 Redis 缓存的最新统计；缓存也无则按基础时间模式判定 | 2次，间隔 1s | WARN |
| LLM 自动审阅返回异常（500/429/超时） | 降级：切换为规则引擎审阅（纯规则匹配，无 AI 评估） | 主模型 2 次 → 规则引擎 | WARN |
| 模式判定出现逻辑冲突 | 降级：强制使用模式 A（保守策略），发送告警 | 不重试 | ERROR |
| 审阅任务写入数据库失败 | 暂存到本地 JSON `data/m13_backup_{task_id}.json` | 5次，间隔 2s | CRITICAL |
| 通知发送失败 | 写入 `notification_retry_queue` 表 | 3次，间隔 5min | WARN |
| 并发审阅冲突（同一内容被多次提交审阅） | 使用 Redis 分布式锁，后来的请求返回 "审阅已在进行中" | 不重试 | INFO |
| 抽检算法异常 | 降级：使用简单随机抽样 | 不重试 | WARN |
| M-12 合规检查不可用 | 跳过合规检查，标记 risk_flags 中合规维度为 "unchecked" | 不重试 | WARN |

### 降级策略

当模块完全不可用时（数据库离线 + LLM 不可达），启用以下降级方案：

1. **全人工模式**: 所有审阅节点强制模式 A，运营人员在后台手动完成所有审阅。系统仅记录审阅结果（通过备份文件恢复后批量导入）
2. **流水线暂停**: 如果 M-13 完全不可用超过 1 小时，流水线引擎自动暂停所有待审阅节点的后续流程
3. **跳过本模块**: 在 `pipeline_config` 中设置 `tiered_review_engine.skip_on_failure = true`，所有审阅默认通过，下游模块继续执行。**仅限紧急情况使用，需人工事后补审**

## 7. 自学习触发条件

### 7.1 学习数据收集

本模块向 `review_learning_data` 表写入以下数据，供 M-25 定期收集分析：

| 数据字段 | 说明 | 收集时机 |
|---------|------|---------|
| `review_id` | 审阅记录 ID | 每次审阅完成后写入 |
| `review_node` | 审阅节点 | 每次审阅完成后写入 |
| `mode` | 审阅模式 (A/B/C) | 每次审阅完成后写入 |
| `decision` | 审阅决定 (pass/reject/modify) | 每次审阅完成后写入 |
| `system_confidence` | 系统置信度 | 模式 B/C 时写入 |
| `was_correct` | 系统审阅决策是否正确 | 人工确认/纠正后回填 |
| `human_correction` | 人工纠正内容 | 有人工纠正时写入 |
| `correction_type` | 纠正类型 | 有人工纠正时写入 |
| `review_duration_ms` | 审阅耗时 | 每次审阅完成后写入 |
| `time_to_human_response_ms` | 人工响应耗时 | 模式 B 人工确认后写入 |
| `content_performance` | 审阅通过的内容最终表现 | 内容发布后 7 天内从 M-22 获取 |
| `false_pass_rate` | 误通过率（审阅通过但内容表现差） | M-22 分析后回写 |
| `false_reject_rate` | 误拒绝率（审阅拒绝但同类内容人工通过后表现好） | M-22 分析后回写 |

### 7.2 偏好库调用

- **调取偏好类型**: 审阅决策偏好（运营人员在什么条件下倾向于 pass/reject/modify）、模式切换偏好（运营人员何时选择越级）
- **置信度阈值**: >= 80%（审阅①升级条件）；>= 70%（审阅②-B 升级条件）
- **无偏好时默认行为**: 按时间基础模式执行（Week 1-4 → A, Week 5-12 → B, Week 13+ → C）

### 7.3 阶段演进行为

| 阶段 | 行为差异 |
|------|---------|
| P0 (人工主导) | 所有审阅节点强制模式 A。系统仅记录审阅数据，不参与任何决策。`review_tasks` 表记录人工的每一次审阅操作。`review_learning_data` 表开始积累基线数据。 |
| P1 (人机协同) | 系统根据时间阶段和升级条件自动判定各节点模式。满足升级条件的节点进入模式 B（系统建议+人工确认）。不满足的保持模式 A。M-25 开始分析人工纠正模式，优化升级条件阈值。 |
| P2 (系统主导) | 多数节点进入模式 C（自动审批+抽检）。审阅①始终最高为模式 B（不可全自动）。系统持续监控误通过率和误拒绝率，自动调整升级/降级条件。抽检比例从 10% 可动态降至 5%。 |

### 注意事项
- M-13 是整个系统最复杂的模块，调试时请逐个节点验证
- 模式判定是纯逻辑计算，不依赖 LLM（仅审阅执行步骤使用 LLM）
- 审阅①（计划表）在任何阶段最高只能到模式 B，这是设计约束，不是 bug
- 升级条件评估依赖 M-25 的偏好库统计数据，如果 M-25 尚未实现，可使用模拟数据
- 人工越级（override）是最高优先级，任何模式判定结果都会被越级覆盖
- 风险升级（risk escalation）优先级次之，但低于人工越级
- 模式降级是自动触发的，不需要人工干预，但会发送通知告知运营人员
- 抽检比例在模式 C 下默认为 10%，可通过配置调整为 5%（需运营主管审批）
- 所有审阅决策都会被记录到 review_learning_data 表，这是 M-25 自学习的数据源
- 审阅节点之间是有顺序依赖的，不能跳过中间节点直接审阅后续节点
- 如果内容被 reject，需要上游模块重新生成后再提交审阅（新 review_id）
- 如果内容被标记 modify，修改后重新提交审阅时使用相同的 content_ref_id 但新的 review_id
