  上游: M-11（内容精加工与定稿模块）→ 传入精加工后的内容（文案/封面/视频）
  下游: M-11（内容精加工与定稿模块）→ 返回合规报告（反馈循环）

### 1.2 核心职责

在内容发布前对文案、封面图、视频进行全面的合规性自检。扫描文案中的竞品名称、绝对化用语（"第一""最好""唯一"等）、敏感词汇；检测图片/视频中的未授权商标、人脸、水印。生成内容质量预评分（标题吸引力、封面视觉冲击力），输出合规报告（通过/风险列表/修改建议）。作为发布前的最后一道防线，确保内容符合平台审核规范和法律法规要求。

### 1.3 触发时机

- **自动触发**: M-11（内容精加工与定稿模块）完成精加工后，由流水线引擎自动调用
- **人工触发**: 运营人员在内容管理后台手动点击「合规检查」对特定内容进行合规扫描
- **定时触发**: 定时任务对 `publish_queue` 中待发布内容进行发布前最后一次合规扫描（发布前 30 分钟）

### 2.3 数据来源

| 字段 | 来源 |
|------|------|
| `task_id` | 流水线引擎生成 |
| `content_draft_id` | M-11 透传 |
| `check_targets.copy_texts` | M-11 精加工后的最终文案 |
| `check_targets.cover_images` | M-11 精加工后的最终封面图路径 |
| `check_targets.video_paths` | M-11 精加工后的最终视频路径 |

| `competitor_blacklist` | 配置文件 `config/competitor_blacklist.yaml`（由运营团队维护） |
| `absolute_claim_patterns` | 配置文件 `config/compliance_patterns.yaml` |
| `sensitive_word_list` | 配置文件 `config/sensitive_words.yaml`（定期更新） |
| `trademark_database` | `data/trademark_db/` 目录下的商标图片特征库 |
| `context` | M-11 透传（来自 topics 表和 account 配置） |

## 4. 核心处理逻辑

### 4.1 处理步骤

```
Step 1: 输入校验与规则加载
  - 做什么: 校验输入格式，加载合规检查规则
  - 怎么判断:
      - 使用 jsonschema 库校验输入格式
      - 加载规则配置: 根据 check_rules.rule_set 加载对应严格程度的规则
        - strict: 全部规则启用，低阈值，高灵敏度
        - standard: 核心规则启用，中等阈值
        - relaxed: 仅关键规则启用，高阈值
      - 加载黑名单/白名单:
        - 竞品名称黑名单: config/competitor_blacklist.yaml
        - 绝对化用语正则: config/compliance_patterns.yaml
        - 敏感词列表: config/sensitive_words.yaml
      - 如果 check_rules 中指定了覆盖值，使用覆盖值
      - 如果 context.previous_violations 非空，自动提升相关检测灵敏度
  - 输出: 最终生效的规则集（effective_rules）

Step 2: 文案合规扫描
  - 做什么: 扫描所有平台文案中的违规内容
  - 怎么判断:
      - 竞品名称检测:
        - 使用 Aho-Corasick 多模式匹配算法在文案中搜索竞品名称
        - 对匹配结果调用 LLM 进行上下文判断（区分提及 vs 推广/诋毁）
        - 上下文判断 Prompt: "以下文本中提到了竞品「{name}」，判断是否为客观提及（如行业分析、对比评测），还是不当推广/诋毁。仅回复 'neutral' 或 'violation'。"
      - 绝对化用语检测:
        - 使用正则表达式匹配: /第一|最好|唯一|绝对|100%|99%|全网|顶级|无敌|最强|独家/g
        - 对匹配结果调用 LLM 进行上下文判断（区分广告用语 vs 客观陈述）
        - 如「全网首发」为绝对化用语，「全网数据显示」为客观引用
      - 敏感词检测:
        - 使用 Aho-Corasick 匹配敏感词列表
        - 敏感词分级: 政治敏感 > 色情暴力 > 虚假信息 > 一般敏感
        - 对匹配结果进行上下文判断
      - 每个发现记录: type, severity, platform, matched_text, position, suggestion
  - 输出: 文案扫描结果（text_scan）

Step 3: 封面图合规扫描
  - 做什么: 扫描所有平台封面图中的违规元素
  - 怎么判断:
      - 水印检测:
        - 使用 OpenCV 模板匹配或深度学习模型检测图片中的水印
        - 排除本账号水印（白名单匹配）
        - 检测到外部水印 → severity=high
      - 人脸检测:
        - 使用 MTCNN 或 RetinaFace 检测图片中的人脸
        - 如果检测到人脸，检查是否有肖像权授权记录
        - 无授权记录 → severity=critical（需确认授权或模糊处理）
        - 已授权 → severity=low（标记供审阅参考）
      - 商标检测:
        - 使用图像特征匹配（SIFT/ORB）与商标库比对
        - 商标库: data/trademark_db/ 目录下的商标图片特征文件
        - 检测到未授权商标 → severity=high
      - 不当内容检测:
        - 使用 NSFW 模型检测不当图像内容
        - 使用 OCR 提取图片中的文字，再进行文案合规扫描
      - 每个发现记录: type, severity, platform, detected_object, confidence, bounding_box, suggestion
  - 输出: 封面图扫描结果（image_scan）

Step 4: 视频合规扫描
  - 做什么: 提取视频关键帧并进行合规扫描
  - 怎么判断:
      - 帧提取策略:
        - 每隔 N 秒提取一帧（N = max(1, duration_sec / 30)，即至少 30 帧）
        - 场景变化检测: 使用帧间差异（histogram difference）检测场景切换点，额外提取切换点帧
        - 关键时间点: 开头 0-3 秒、结尾最后 3 秒各提取密集帧（每 0.5s 一帧）
      - 对每帧执行与封面图相同的合规扫描:
        - 水印检测
        - 人脸检测
        - 商标检测
        - 不当内容检测
      - 字幕违规检测:
        - 如果有 SRT 字幕文件，提取字幕文本
        - 对字幕文本执行文案合规扫描
      - 音频检测（可选）:
        - 提取音频进行语音识别（ASR）
        - 对识别文本执行文案合规扫描
        - 检测未授权 BGM（音频指纹比对）
      - 每个发现记录: type, severity, platform, timestamp_sec, detected_object, confidence, frame_path, suggestion
  - 输出: 视频扫描结果（video_scan）

Step 5: 平台规则校验
  - 做什么: 校验各平台内容的格式和规则合规性
  - 怎么判断:
      - 抖音规则:
        - 文案字符数 <= 500
        - 视频分辨率 >= 720p
        - 视频时长 15s-60min
        - 禁止引流到其他平台（如微信号、QQ号）
      - 小红书规则:
        - 文案字符数 <= 1000
        - 图片分辨率 >= 1080×1080
        - 禁止虚假种草、夸大宣传
        - 医疗/金融类内容需资质
      - B站规则:
        - 文案字符数 <= 2000
        - 视频分辨率 >= 720p
        - 禁止低俗内容
      - X (Twitter) 规则:
        - 文案字符数 <= 280（Premium 可更长）
        - 禁止仇恨言论
        - 敏感内容需标记
      - 每个平台规则从 config/platform_rules/{platform}.yaml 加载
  - 输出: 平台规则校验结果（platform_rules）

Step 6: 内容质量预评分
  - 做什么: 对内容进行发布前质量评分
  - 怎么判断:
      - 标题吸引力评分（title_appeal）:
        - 长度评分: 最优长度 15-25 字（抖音/小红书），偏离越多扣分越多
        - 钩子评分: 检测问句、数字、感叹、悬念等钩子元素
        - 使用 LLM 评估: "请评估以下标题的吸引力（0-100）: {title}"
      - 封面视觉评分（cover_visual）:
        - 构图评分: 使用美学评估模型（如 NIMA）评分
        - 色彩评分: 检测色彩对比度、饱和度是否适合平台
        - 文字可读性: OCR 检测文字区域对比度
      - 综合评分（overall）:
        - overall = title_appeal × 0.4 + cover_visual × 0.6
      - 如果 overall < quality_threshold.overall_min，生成低质量警告
  - 输出: 质量评分（quality_scores）

Step 7: 综合判定与建议生成
  - 做什么: 汇总所有检测结果，生成综合判定和建议
  - 怎么判断:
      - 汇总所有 findings，按 severity 分类统计
      - 判定逻辑:
        - 存在 critical 级别 finding → overall_status = "fail"
        - 存在 high 级别 finding → overall_status = "risk_detected"
        - 仅有 medium/low 级别 finding → overall_status = "risk_detected"
        - 无任何 finding → overall_status = "pass"
      - 建议生成:
        - fail → recommendation = "blocked"（有 critical 违规）或 "revision_required"（有 high 违规）
        - risk_detected 且 quality_scores.overall < 60 → "revision_recommended"
        - risk_detected 且 quality_scores.overall >= 60 → "publish_with_warnings"
        - pass → "ready_to_publish"
      - 为每个 finding 生成修改建议（suggestion）
  - 输出: 综合摘要（summary）

Step 8: 结果写入
  - 做什么: 写入合规检查记录
  - 怎么判断:
      - 生成 check_id（格式: CMP-YYYY-NNNNNN）
      - 写入 compliance_checks 表
      - 写入 module_outputs 表
      - 如果 overall_status='fail'，同时写入 errors 表
  - 输出: 最终的 Output JSON
```

## 6. 错误处理策略

| 错误类型 | 处理方式 | 重试策略 | 告警级别 |
|---------|---------|---------|---------|
| 输入 JSON Schema 校验失败 | 返回 overall_status=fail，终止执行 | 不重试 | ERROR |
| 文案为空 | 跳过文案扫描，标记 text_scan.status='skipped' | 不重试 | INFO |
| 封面图文件不存在 | 跳过该平台封面扫描，标记 image_scan 中该平台为 'skipped' | 不重试 | WARN |
| 视频文件不存在 | 跳过该平台视频扫描，标记 video_scan 中该平台为 'skipped' | 不重试 | WARN |
| 竞品黑名单/敏感词列表加载失败 | 使用上次缓存的列表（Redis）；无缓存时跳过对应检测 | 不重试 | CRITICAL |
| 商标库加载失败 | 跳过商标检测，添加 warning: "商标库不可用" | 不重试 | WARN |
| 人脸检测模型加载失败 | 跳过人脸检测，添加 warning: "人脸检测模型不可用" | 不重试 | WARN |
| 视频帧提取失败（FFmpeg 错误） | 跳过该视频的视频帧检测，仅进行字幕文本检测 | 不重试 | WARN |
| LLM 上下文判断调用超时（> 5s） | 降级: 使用纯规则匹配结果（不进行上下文判断） | 不重试 | WARN |
| LLM 质量评分调用失败 | 使用规则评分替代（仅基于长度、钩子元素等规则） | 不重试 | INFO |
| OCR 识别失败 | 跳过图片文字提取步骤 | 不重试 | INFO |
| 数据库写入失败 | 暂存到本地 JSON 文件 `data/m12_backup_{task_id}.json` | 5 次，间隔 2s | CRITICAL |
| 全部检测项失败 | 返回 overall_status='fail'，要求人工进行合规检查 | 不重试 | CRITICAL |

### 降级策略

1. **LLM 不可用**: 所有上下文判断降级为纯规则匹配（可能增加误报率），质量评分使用规则计算
2. **图像/视频检测模型不可用**: 跳过对应检测，在报告中标注检测项状态为 'skipped'
3. **商标库不可用**: 仅检测文字层面的商标名称（从文案中匹配），跳过图像商标检测
4. **全部检测不可用**: 返回 fail，要求人工合规审核

### 7.3 阶段演进行为

| 阶段 | 行为差异 |
|------|---------|
| P0 (人工主导) | 合规检查完全由人工完成。模块仅提供检测结果展示界面。LLM 上下文判断不启用。 |
| P1 (人机协同) | 模块自动执行全部检测，人工审核检测结果并标记误报。LLM 上下文判断启用（减少误报）。检测规则开始根据误报反馈优化。 |
| P2 (系统主导) | 模块全自动检测，仅当 overall_status='fail' 或存在 critical 级别发现时才通知人工。检测规则根据平台驳回数据自动更新。新增检测规则自动从驳回原因中提取。 |
