# M-20 数据清洗与聚合

---

## 1. 模块定位

| 属性 | 值 |
|------|-----|
| **模块编号** | M-20 |
| **模块名称** | 数据清洗与聚合 |
| **所属分类** | 数据分析 |
| **优先级** | P2（重要模块，数据分析链路的入口） |
| **触发方式** | 定时任务（每日 09:00 UTC+8，在 M-19 之后执行）；支持手动触发 |
| **在系统中的位置** | 接收 M-18 的原始表现数据，结合 M-19 的账号健康状态，进行数据清洗与聚合计算，为 M-21 多维度效果分析提供清洁数据基础 |

### 1.1 核心职责

1. **数据清洗**：读取原始表现数据，识别并处理异常值（突发峰值、缺失项），确保数据质量。
2. **异常值检测**：使用 IQR（四分位距）方法检测离群点，使用 3-Sigma 原则标记极端异常。
3. **互动率计算**：统一计算各内容/账号的互动率 = (点赞数 + 评论数 + 分享数) / 浏览数 × 100%。
4. **多维度聚合**：按平台、内容类型、发布时间等维度聚合数据，生成聚合视图。
5. **基准对比**：将当前数据与偏好库中的历史基准进行对比，标记表现异常维度的数据。

### 1.2 上下游关系

```
M-18(表现数据采集) ──┐
                      ├──→ M-20(本模块) → M-21(多维度效果分析)
M-19(账号健康监控) ──┘
```

- **上游**：M-18 提供 `performance_metrics` 原始数据；M-19 提供 `account_health` 作为数据过滤维度
- **下游**：M-21 消费清洗后的 `analytics_reports` 表数据进行多维分析

## 4. 核心逻辑

### 4.1 主流程

```
开始
  │
  ├─ 1. 加载输入数据
  │     ├─ 从 M-18 读取 performance_metrics records
  │     ├─ 验证数据格式完整性
  │     └─ 生成 batch_id 标记本批次
  │
  ├─ 2. 数据预处理
  │     ├─ 检测缺失字段，标记 data_completeness
  │     ├─ 填充可推导的缺失值（如 views=0 时 engagement_rate=NULL）
  │     └─ 数据类型标准化与范围校验
  │
  ├─ 3. 异常值检测（IQR 方法）
  │     ├─ 按平台 + 内容类型分组
  │     ├─ 对每组内的 views、likes、comments、shares：
  │     │   Q1 = 第 25 百分位数
  │     │   Q3 = 第 75 百分位数
  │     │   IQR = Q3 - Q1
  │     │   lower_bound = Q1 - 1.5 × IQR
  │     │   upper_bound = Q3 + 1.5 × IQR
  │     │   IF value < lower_bound OR value > upper_bound
  │     │     → 标记为 mild_outlier
  │     └─ 记录 outlier_dimension 列表
  │
  ├─ 4. 极端异常标记（3-Sigma 方法）
  │     ├─ 对每组计算 mean 和 std
  │     ├─ z_score = (value - mean) / std
  │     ├─ IF |z_score| > 3.0 → 标记为 extreme_outlier
  │     └─ 记录 sigma_score
  │
  ├─ 5. 数据清洗处理
  │     ├─ mild_outlier：使用 Winsorize 方法（缩尾处理）
  │     │   → 将超出边界的值替换为边界值
  │     ├─ extreme_outlier：标记为异常，不参与后续聚合计算
  │     │   → 保留原始值但设置 is_cleaned = true
  │     └─ missing_data：使用同组中位数插值（仅对非关键字段）
  │
  ├─ 6. 互动率计算
  │     ├─ IF views > 0:
  │     │   engagement_rate = (likes + comments + shares) / views × 100
  │     ├─ ELSE:
  │     │   engagement_rate = NULL
  │     ├─ like_rate = likes / views × 100
  │     ├─ comment_rate = comments / views × 100
  │     ├─ share_rate = shares / views × 100
  │     └─ IF reach > 0: reach_efficiency = views / reach × 100
  │
  ├─ 7. 多维度聚合
  │     ├─ 按 platform 聚合：各平台平均互动率、总浏览量
  │     ├─ 按 content_type 聚合：各内容类型的表现对比
  │     ├─ 按 publish_hour 聚合：最佳发布时间分析
  │     ├─ 按 account_id 聚合：各账号表现排名
  │     └─ 按 content_topic 聚合：话题表现分析
  │
  ├─ 8. 历史基准对比
  │     ├─ 从偏好库加载历史基准数据
  │     ├─ 按 platform + content_type 匹配基准值
  │     ├─ 计算偏差：deviation = (current - benchmark) / benchmark × 100
  │     ├─ IF |deviation| > 20% → 标记为显著偏差
  │     └─ 计算百分位排名 performance_percentile
  │
  ├─ 9. 写入数据库
  │     ├─ 逐条写入 analytics_reports 表
  │     ├─ 写入聚合汇总记录
  │     └─ 更新 batch 处理状态
  │
  └─ 10. 输出处理摘要
        ├─ 总记录数、清洗数、异常数
        ├─ 各维度聚合统计
        └─ 通知下游 M-21 数据已就绪
```

### 4.2 IQR 异常值检测算法

```
Function detect_outliers_iqr(group_data, field_name):
    values = extract_field(group_data, field_name)
    values = [v for v in values if v is not None]

    IF len(values) < min_records_for_stats:
        RETURN []  # 样本量不足，不做检测

    sorted_values = sort(values)
    n = len(sorted_values)
    Q1 = sorted_values[int(n * 0.25)]
    Q3 = sorted_values[int(n * 0.75)]
    IQR = Q3 - Q1

    lower_fence = Q1 - iqr_multiplier * IQR
    upper_fence = Q3 + iqr_multiplier * IQR

    outliers = []
    FOR each record in group_data:
        value = record[field_name]
        IF value < lower_fence OR value > upper_fence:
            outliers.append({
                "content_id": record.content_id,
                "field": field_name,
                "value": value,
                "bound": "lower" if value < lower_fence else "upper",
                "fence_value": lower_fence if value < lower_fence else upper_fence
            })
    RETURN outliers
```

### 4.3 3-Sigma 异常标记算法

```
Function detect_anomalies_3sigma(group_data, field_name):
    values = [r[field_name] for r in group_data if r[field_name] is not None]
    IF len(values) < 3:
        RETURN []

    mean = sum(values) / len(values)
    variance = sum((v - mean)^2 for v in values) / len(values)
    std = sqrt(variance)

    IF std == 0:
        RETURN []  # 无变异，无法检测

    anomalies = []
    FOR each record in group_data:
        z = (record[field_name] - mean) / std
        IF abs(z) > sigma_threshold:
            anomalies.append({
                "content_id": record.content_id,
                "field": field_name,
                "z_score": z,
                "sigma_level": abs(z)
            })
    RETURN anomalies
```

## 6. 错误处理

### 6.1 异常分类

| 异常类型 | 处理策略 | 重试次数 | 降级方案 |
|----------|----------|----------|----------|
| 输入数据为空 | 跳过本次处理，记录日志 | 0 | 等待下一批次数据 |
| 单条记录格式错误 | 跳过该记录，继续处理其他 | 0 | 记录到 error_records 表 |
| IQR 检测样本不足 | 标记为 insufficient_sample | - | 跳过该分组的异常检测 |
| 历史基准数据缺失 | 仅计算当前指标，不进行对比 | - | benchmark_comparison 字段置 null |
| 数据库写入超时 | 分批重试 | 3 次 | 降级为批量写入（每 100 条提交一次） |
| 标准差为零（所有值相同） | 跳过 3-Sigma 检测 | - | 标记该分组为 low_variance |
| 内存溢出（大数据量） | 分页处理 | - | 每批处理 1000 条记录 |

## 7. 自学习机制

### 7.1 动态阈值调整

```
每 30 天分析异常值检测结果：
- 统计 IQR 检出率（被标记为异常的比例）
- IF 检出率 > 10%：
  → 可能 IQR 乘数过小或数据分布特殊
  → 自动调整 iqr_multiplier 从 1.5 到 2.0
- IF 检出率 < 1%：
  → 可能阈值过宽，漏检异常
  → 自动调整 iqr_multiplier 从 1.5 到 1.2
```

### 7.2 基准库更新

```
每周自动更新历史基准：
- 将本周清洗后的数据（anomaly_level = "normal"）纳入基准库
- 重新计算各 platform + content_type 组合的：
  - 平均互动率
  - 中位数互动率
  - P25/P75 互动率
  - 平均浏览量
- 版本化管理：每次更新生成新的 benchmark_version
```

### 7.3 异常模式学习

```
记录运营人员的手动修正：
- 如果某条被标记为 extreme_outlier 的数据被手动恢复
  → 记录该内容特征（platform、type、topic）
  → 当类似特征内容再次出现相似指标时，降低异常标记的置信度
- 累计被手动恢复的同特征异常 > 3 次
  → 生成规则：该特征组合的 3-Sigma 阈值放宽至 4.0
```

## 9. Hermes Agent 执行指令

### 9.1 标准数据清洗指令

```
你作为 M-20 数据清洗与聚合模块的 Hermes Agent，请按照以下步骤执行数据清洗任务：

1. 确认输入数据：
   - 从 M-18 获取最新的 performance_metrics batch
   - 验证 batch_id 和 records 数量
   - 检查是否有 M-19 的 account_health 过滤条件需要应用

2. 数据预处理：
   - 遍历所有 records，检查必填字段完整性
   - 记录缺失字段到 quality_flags.missing_fields
   - 计算 data_completeness = 已填充必填字段数 / 总必填字段数 × 100
   - 跳过完全无效的记录（content_id 缺失或 views/likes/comments/shares 全部缺失）

3. 异常值检测（按 platform + content_type 分组）：
   a. IQR 检测：
      - 对每组内的 views、likes、comments、shares 分别计算 Q1、Q3、IQR
      - 判定边界：lower = Q1 - 1.5×IQR, upper = Q3 + 1.5×IQR
      - 超出边界的值标记为 mild_outlier
      - 记录 outlier_dimension
   b. 3-Sigma 标记：
      - 对每组计算 mean 和 std
      - 计算每条记录的 z_score
      - |z_score| > 3.0 → 标记为 extreme_outlier
      - 记录 sigma_score
   c. 注意：样本量 < 10 的分组跳过统计检测

4. 数据清洗：
   - mild_outlier → Winsorize 缩尾：将值替换为对应的 fence_value
   - extreme_outlier → 保留原始值，设置 is_cleaned=true，标记 anomaly_level
   - 缺失值 → 使用同组中位数插值（仅对非核心指标）
   - 记录 cleaning_method

5. 互动率计算：
   - engagement_rate = (likes + comments + shares) / views × 100
   - like_rate = likes / views × 100
   - comment_rate = comments / views × 100
   - share_rate = shares / views × 100
   - 注意：views=0 时以上指标均设为 NULL

6. 基准对比：
   - 从 benchmark_library 加载对应 platform + content_type 的历史基准
   - 计算 benchmark_deviation = (current - benchmark) / benchmark × 100
   - 计算 performance_percentile
   - 判定 performance_label

7. 聚合汇总：
   - 按 platform 聚合：各平台平均互动率、总浏览量、异常率
   - 按 content_type 聚合：各内容类型表现对比
   - 按 publish_hour 聚合：识别最佳发布时段
   - 写入 aggregation_summary 表

8. 写入结果：
   - 逐条写入 analytics_reports 表
   - 写入 outlier_log（所有被标记为异常或离群值的记录）
   - 更新批次处理状态为 completed

9. 输出处理摘要：
   - 总记录数、清洗数、异常数、缺失数据数
   - 各平台平均互动率排名
   - 通知 M-21 数据已就绪
```
