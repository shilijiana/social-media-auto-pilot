# M-27 规则时效性管理

---

## 一、模块定位

| 属性 | 值 |
|------|-----|
| 模块编号 | M-27 |
| 模块名称 | 规则时效性管理 |
| 所属分类 | 自学习 |
| 优先级 | P3 |
| 上游依赖 | M-26（偏好提炼与规则生成） |
| 下游依赖 | M-29（学习成果应用）、M-30（学习成效量化与报告） |
| 触发方式 | 每日定时任务（03:00）+ 手动触发 |

### 功能概述

本模块是自学习系统的**规则保鲜层**。偏好规则会随着时间推移和用户需求变化而逐渐过时。本模块通过记录每条规则的"最后验证时间"，对超过60天未验证的规则自动降级，超过90天的标记"待确认"并暂停应用，确保偏好库中的规则始终反映用户最新的偏好。

### 设计目标

1. **自动过期**：过期规则自动降级，无需人工干预
2. **平滑过渡**：降级是渐进式的（置信度减半），而非直接删除
3. **人工可控**：提供人工标记接口，允许手动处理过期规则
4. **批量重置**：重大策略变更时支持按类别批量重置规则

### 4.2 时效性分级处理逻辑

```python
def check_rule_expiry(rule, current_time):
    """
    检查单条规则的时效性并执行相应操作
    """
    days_since_verified = (current_time - rule["statistics"]["last_verified_at"]).days
    
    if days_since_verified < 60:
        # 60天以内：无需处理
        return {"action": "no_change", "rule_id": rule["rule_id"]}
    
    elif 60 <= days_since_verified < 75:
        # 60-75天：轻度降级，置信度减半
        new_confidence = round(rule["confidence"] * 0.5, 1)
        new_status = "reference_only" if new_confidence < 90 else rule["status"]
        return {
            "action": "degrade_confidence",
            "rule_id": rule["rule_id"],
            "previous_confidence": rule["confidence"],
            "new_confidence": new_confidence,
            "previous_status": rule["status"],
            "new_status": new_status,
            "days_since_verified": days_since_verified,
            "reason": f"规则超过{days_since_verified}天未验证，置信度减半"
        }
    
    elif 75 <= days_since_verified < 90:
        # 75-90天：中度降级，置信度降至25%
        new_confidence = round(rule["confidence"] * 0.25, 1)
        new_status = "reference_only"
        return {
            "action": "degrade_confidence",
            "rule_id": rule["rule_id"],
            "previous_confidence": rule["confidence"],
            "new_confidence": new_confidence,
            "previous_status": rule["status"],
            "new_status": new_status,
            "days_since_verified": days_since_verified,
            "reason": f"规则超过{days_since_verified}天未验证，置信度降至25%"
        }
    
    else:  # days_since_verified >= 90
        # 90天以上：标记待确认，暂停应用
        return {
            "action": "pause_application",
            "rule_id": rule["rule_id"],
            "previous_confidence": rule["confidence"],
            "new_confidence": rule["confidence"],  # 保留原置信度，但暂停
            "previous_status": rule["status"],
            "new_status": "pending_review",
            "days_since_verified": days_since_verified,
            "reason": f"规则超过{days_since_verified}天未验证，标记为'待确认'并暂停应用"
        }
```

### 4.3 重新验证逻辑

当规则被 M-26 或 M-29 重新引用/验证时，自动更新 `last_verified_at`：

```python
def revalidate_rule(rule_id, verification_result):
    """
    规则被重新验证时的处理
    """
    rule = get_rule(rule_id)
    
    if verification_result == "confirmed":
        # 验证通过：恢复置信度
        if rule["confidence"] < 50:
            rule["confidence"] = 50  # 最低恢复到50%
        else:
            rule["confidence"] = min(rule["confidence"] * 1.2, 100)
        rule["status"] = "reference_only" if rule["confidence"] < 90 else "active"
    
    elif verification_result == "overturned":
        # 验证推翻：大幅降低置信度
        rule["confidence"] = max(rule["confidence"] * 0.3, 10)
        rule["status"] = "pending_review"
    
    # 更新最后验证时间
    rule["statistics"]["last_verified_at"] = datetime.now().isoformat()
    
    return rule
```

### 4.4 批量重置

```python
def batch_reset_category(library, rule_category, reason, operator_id):
    """
    重大策略变更时批量重置指定类别规则
    
    操作：
    1. 将该类别所有规则的置信度重置为 50%
    2. 状态重置为 reference_only
    3. 记录操作日志
    """
    rules = get_rules_by_category(library, rule_category)
    
    results = []
    for rule in rules:
        previous = {
            "confidence": rule["confidence"],
            "status": rule["status"]
        }
        
        rule["confidence"] = 50.0
        rule["status"] = "reference_only"
        rule["execution_mode"] = "suggest_reference"
        rule["statistics"]["last_verified_at"] = datetime.now().isoformat()
        rule["updated_at"] = datetime.now().isoformat()
        
        # 添加重置日志
        add_reset_log(rule["rule_id"], previous, reason, operator_id)
        
        results.append({
            "rule_id": rule["rule_id"],
            "previous_confidence": previous["confidence"],
            "new_confidence": 50.0,
            "action": "batch_reset"
        })
    
    return results
```

## 七、自学习

### 7.1 降级阈值自适应

- 统计被降级后又重新验证的规则比例
- 如果 >30% 的降级规则后续被重新确认有效，说明降级阈值可能过于激进
- 自动将 60 天阈值延长至 75 天

### 7.2 规则存活周期分析

- 统计不同类型规则的平均存活周期
- 策略类规则通常存活更久，素材/风格类规则变化更快
- 根据存活周期动态调整不同类别规则的降级阈值

### 7.3 废弃原因归类

- 收集所有被标记 deprecated 的规则
- 分析废弃原因分布（时效性过期 vs 策略变更 vs 用户偏好改变）
- 反馈给 M-26 优化规则生成算法

### 8.2 降级逻辑检查

```sql
-- 检查未验证天数分布
SELECT 
  CASE 
    WHEN days_since < 60 THEN 'normal'
    WHEN days_since < 75 THEN 'warning_60_75'
    WHEN days_since < 90 THEN 'warning_75_90'
    ELSE 'critical_90plus'
  END as status_level,
  COUNT(*) 
FROM (
  SELECT 
    rule_id,
    EXTRACT(DAY FROM (NOW() - statistics->>'last_verified_at')::timestamp) as days_since
  FROM preference_rules 
  WHERE status != 'deprecated'
) sub
GROUP BY status_level;

-- 检查是否有规则 last_verified_at 为 NULL
SELECT COUNT(*) FROM preference_rules 
WHERE statistics->>'last_verified_at' IS NULL;
```

## 九、Hermes Agent 指令

### 9.1 角色定义

```
你是 M-27 规则时效性管理模块的 Hermes Agent。
你的职责是：每日定时检查偏好规则库中所有规则的时效性，
对过期规则自动降级或暂停，确保偏好库中的规则始终保持新鲜有效。
```

### 9.2 执行指令

```
触发条件：
- 定时触发：每日 03:00 自动执行
- 手动触发：通过 API 调用
- 事件触发：重大策略变更时

执行步骤：

1. 【规则加载】从 preference_rules 表加载数据：
   a. 筛选条件：status != 'deprecated'
   b. 加载字段：rule_id, library, rule_name, confidence, status, execution_mode, 
      statistics.last_verified_at, created_at, updated_at
   c. 按 library 分组：material, style, strategy

2. 【时效性计算】对每条规则：
   a. 获取 statistics.last_verified_at
   b. 如果 last_verified_at 为 NULL，使用 created_at 作为替代
   c. 计算 days_since_verified = NOW() - last_verified_at
   d. 记录计算结果

3. 【分级处理】根据 days_since_verified 执行操作：

   Level 0（< 60天）：正常
   - action: "no_change"
   - 不做任何修改

   Level 1（60-75天）：轻度降级
   - action: "degrade_confidence"
   - new_confidence = round(confidence * 0.5, 1)
   - 如果 new_confidence < 90，则 status → "reference_only"
   - 记录降级日志

   Level 2（75-90天）：中度降级
   - action: "degrade_confidence"
   - new_confidence = round(confidence * 0.25, 1)
   - status → "reference_only"
   - execution_mode → "suggest_reference"

   Level 3（≥ 90天）：暂停应用
   - action: "pause_application"
   - status → "pending_review"
   - execution_mode → null
   - 保留原 confidence 值不变
   - 添加到通知列表，提示人工确认

4. 【规则更新】批量写入更新：
   a. 收集所有需要修改的规则
   b. 使用乐观锁更新（检查 version 字段）
   c. 批量 UPSERT 到 preference_rules
   d. 记录每次更新的前后状态

5. 【通知生成】对于 Level 3 规则：
   a. 生成通知条目
   b. 包含：rule_id, rule_name, days_since_verified, suggested_action
   c. suggested_action 可选：
      - "请确认此规则是否仍然适用"
      - "建议标记为已过时并删除"
      - "建议重新验证此规则"

6. 【日志记录】写入 expiry_check_log 表：
   {
     "check_id": "EXP-{timestamp_hex}",
     "checked_at": "ISO datetime",
     "total_rules_checked": N,
     "rules_degraded": N,
     "rules_paused": N,
     "rules_unchanged": N
   }

7. 【手动操作处理】如果收到手动操作请求：
   a. 验证操作类型和目标有效性
   b. 执行对应操作：
      - mark_deprecated: status → "deprecated"
      - reset_confidence: 更新指定规则的 confidence
      - batch_reset_category: 批量重置指定类别
      - extend_validity: 更新 last_verified_at 为当前时间
   c. 记录操作日志（操作人、时间、原因）

8. 【结果返回】返回处理摘要：
   {
     "check_id": "EXP-{hex}",
     "checked_at": "ISO datetime",
     "summary": {
       "total_rules_checked": N,
       "rules_unchanged": N,
       "rules_degraded": N,
       "rules_paused": N,
       "rules_manually_deprecated": N,
       "by_library": { "material": {...}, "style": {...}, "strategy": {...} },
       "degradation_details": {
         "level_1_days_60_75": N,
         "level_2_days_75_90": N,
         "level_3_days_90_plus": N
       }
     },
     "notifications": [...]
   }
```

### 9.3 质量约束

```
- 降级操作准确率 100%（不允许误降级）
- 定时任务执行时间 < 5分钟（假设规则数 < 10000）
- 通知覆盖率 100%（所有 >90天规则必须通知）
- 不允许在 60 天内降级任何规则
- 不允许将 confidence 降至负数
```

### 9.4 自检指令

```
每次执行完成后，自检以下项目：
1. 检查是否有规则被错误降级（days_since < 60 但被降级）
2. 检查降级后 confidence 是否在合理范围（0-100）
3. 检查 status 转换是否合法
4. 检查 notification 列表是否完整
5. 检查日志是否写入成功
6. 如发现异常，立即告警并记录
```

### 9.5 异常恢复指令

```
如果在执行过程中发生异常中断：
1. 检查 expiry_check_log 中的最后一条记录
2. 找到 check_id 对应的 rules_processed 进度
3. 从下一个未处理的规则继续
4. 使用 dry_run=true 先验证再实际执行
5. 完成后合并两次执行的结果
```
