```
M-02 位置说明:
  上游: M-01（关键词智能扩展） → 传入扩展关键词用于搜索
  下游: M-05（素材去重与质量控制）、M-06（素材入库与版权追踪）、M-07（素材匹配与调用）
  自学习: M-25 收集素材使用偏好数据（色调/构图/风格偏好）
```

### 1.2 核心职责

使用扩展关键词从免费图库（Unsplash、Pexels）搜索并下载高清图片素材。对图片进行分辨率过滤、色调分析、相关性评分，确保素材质量满足内容生产需求。输出素材记录写入 `materials` 表，文件存入本地素材库。

### 1.3 触发时机

- **自动触发**: M-01（关键词扩展）完成后，由流水线引擎自动调用
- **人工触发**: 运营人员在素材管理后台手动输入关键词搜索并采集图片
- **定时触发**: 每日 06:00 UTC 执行热门关键词的图片素材预采集（配合 M-23 热点数据）

## 4. 核心处理逻辑

### 4.1 处理步骤

```
Step 1: 输入校验
  - 做什么: 校验 task_id 格式、topic_id 格式、search_queries 非空且不超过 30 个、platform 合法
  - 怎么判断: 使用 jsonschema 库校验，对比 2.1 中的 Schema
  - 输出: 校验通过 → 进入 Step 2; 校验失败 → 返回 status=failed，记录错误详情

Step 2: 加载偏好规则
  - 做什么: 如果输入包含 style_preference_id，从 M-25 偏好库加载素材风格偏好
  - 怎么判断:
      - 调用 M-25 偏好查询接口，传入 style_preference_id
      - 若命中偏好（置信度 >= 70%），将偏好注入筛选参数
      - 偏好包含：偏好的色调类型、构图风格（居中/三分法/留白等）、亮度偏好
  - 输出: 筛选偏好对象 `style_filter_params`

Step 3: API 搜索图片
  - 做什么: 使用搜索关键词，调用 Unsplash 和 Pexels API 搜索图片
  - 怎么判断:
      - 对每个 search_query.keyword，分别调用 Unsplash 和 Pexels API
      - 每次请求最多获取 30 条结果（API 分页限制）
      - 按 relevance_score 排序请求优先级（高分关键词优先）
      - 记录每次 API 调用的响应状态和速率限制头
      - 若 API 返回空结果，记录并跳过该关键词
  - 输出: 原始搜索结果列表（URL、作者、尺寸等元数据）

Step 4: 分辨率过滤
  - 做什么: 过滤掉不满足最低分辨率要求的图片
  - 怎么判断:
      - 检查图片的 width >= min_resolution.width 且 height >= min_resolution.height
      - 对于 portrait 竖版素材，额外要求 height/width >= 1.2
      - 对于 landscape 横版素材，额外要求 width/height >= 1.2
      - 分辨率不达标的图片直接丢弃，记录过滤原因
  - 输出: 满足分辨率要求的图片列表

Step 5: 色调分析
  - 做什么: 对通过分辨率过滤的图片进行色调分析
  - 怎么判断:
      - 下载图片缩略图（最大 800px）进行色调分析
      - 使用 OpenCV 提取主色调（K-Means 聚类，k=5）
      - 计算饱和度水平和亮度水平
      - 与 color_tone_filters 进行匹配度评分
      - 若用户指定了 color_tone_filters，色调不匹配的图片降低 overall 评分
  - 输出: 每张图片的 `color_analysis` 对象和 `color_tone_match` 评分

Step 6: 相关性评分
  - 做什么: 评估图片内容与搜索关键词的相关性
  - 怎么判断:
      - 调用 Gemini 视觉模型分析图片内容，生成 content_tags
      - 将 content_tags 与搜索关键词进行语义相似度计算
      - 结合关键词的 relevance_score 进行加权评分
      - 综合评分 = content_tag_match * 0.5 + keyword_relevance * 0.3 + visual_quality * 0.2
  - 输出: 每张图片的 `relevance` 评分和 `content_tags`

Step 7: 下载与保存
  - 做什么: 下载满足所有质量要求的图片到本地素材库
  - 怎么判断:
      - 按 overall 评分降序排列，下载前 max_images_per_query 张图片
      - 文件命名遵循规范: `[选题编号]_[平台]_[日期]_[类型]_[素材ID].ext`
      - 下载到 `data/files/materials/images/YYYY/MM/` 目录
      - 同时生成缩略图（宽 400px，等比缩放，质量 80%）
      - 下载完成后校验文件完整性（文件大小 > 0，可正常打开）
  - 输出: 本地存储路径和文件元数据

Step 8: 写入数据库与返回
  - 做什么: 将素材记录写入 `materials` 表和 `material_tags` 表
  - 怎么判断:
      - 为每张图片生成唯一 material_id（格式: MAT-YYYY-NNNNNN）
      - 写入 `materials` 表（type='image', copyright_status='free'）
      - 写入 `material_tags` 表（content_tags 关联）
      - 写入 `module_outputs` 表完整输出 JSON
      - 返回最终的 Output JSON
  - 输出: 最终的 Output JSON
```

## 6. 错误处理策略

| 错误类型 | 处理方式 | 重试策略 | 告警级别 |
|---------|---------|---------|---------|
| 输入 JSON Schema 校验失败 | 返回 status=failed，错误详情写入 `errors` 表，终止执行 | 不重试 | ERROR |
| Unsplash API 返回 401/403（认证失败） | 跳过该数据源，仅使用 Pexels；记录错误并告警 | 不重试 | CRITICAL |
| Pexels API 返回 401/403（认证失败） | 跳过该数据源，仅使用 Unsplash；记录错误并告警 | 不重试 | CRITICAL |
| API 速率限制（429 Too Many Requests） | 等待 Retry-After 头指定的秒数后重试 | 3次，指数退避 1s/2s/4s | WARN |
| API 搜索返回空结果 | 记录该关键词无结果，继续处理下一个关键词 | 不重试 | INFO |
| 图片下载超时（> 30s） | 记录该图片为 failed，继续下载下一张 | 3次，指数退避 1s/2s/4s | WARN |
| 图片文件损坏（下载完成但无法打开） | 删除损坏文件，重新下载；若仍失败则丢弃 | 重试 1 次 | ERROR |
| 磁盘空间不足（写入失败） | 清理 `data/files/materials/images/tmp/` 临时目录，告警 | 清理后重试 1 次 | CRITICAL |
| LLM 内容标签生成异常（500/429/超时） | 降级：使用图片文件名和 API 返回的描述作为 content_tags | 重试 1 次 | WARN |
| 色调分析 OpenCV 处理失败 | 降级：color_analysis 字段设为 null，不影响整体流程 | 不重试 | WARN |
| 数据库写入失败（连接断开/锁超时） | 暂存结果到本地 JSON 文件，发送告警 | 5次，间隔 2s | CRITICAL |
| 所有图片下载失败（status=failed） | 通知流水线引擎，写入 `pipeline_errors` 表 | 不重试 | CRITICAL |

### 降级策略

当模块完全不可用时（如两个 API 均不可达 + LLM 服务中断），启用以下降级方案：

1. **兜底素材库**: 使用 `data/files/materials/images/default/` 中的预置默认素材（分类别存储，如科技类、生活类、自然类），根据选题类别匹配
2. **人工介入**: 发送企业微信/钉钉通知给运营团队，附上选题 ID 和关键词，由人工手动搜集素材
3. **跳过本模块**: 在 `pipeline_config` 中设置 `image_collector.skip_on_failure = true`，流水线使用默认素材继续后续流程
