M-11 位置说明:
  上游: M-08（文案差异化生成模块）→ 传入文案草稿
        M-09（封面图生成模块）→ 传入封面图草稿
        M-10（视频初剪模块）→ 传入视频草稿
        M-13（分级审阅引擎模块）→ 传入审阅反馈
  下游: M-12（合规性自检模块）→ 精加工完成后触发合规检查
        M-16（定时分发模块）→ 定稿后进入分发队列
  反馈循环: M-12 → M-11（合规不通过时返回修改）
```

### 1.2 核心职责

作为内容生产流水线的最后一道工序，负责将 M-08/M-09/M-10 生成的文案、封面、视频初稿进行精加工处理。逐条读取 M-13 返回的审阅反馈，按反馈逐项修改文案内容、封面设计、视频剪辑。修改完成后调用 M-12 进行合规性自检，合规通过则标记为「定稿版」，生成版本变更日志，打包为「待发件（终审版）」写入分发队列。合规不通过则返回修订状态，等待重新修改。

### 1.3 触发时机

- **自动触发**: M-13（分级审阅引擎）返回审阅反馈后，由流水线引擎自动调用
- **人工触发**: 运营人员在审阅后台完成审阅并点击「提交修改」后触发精加工
- **批量触发**: 支持批量处理多个内容草稿的精加工任务

| `change_log` | 写入 `content_drafts` 表的 `change_log` 字段（JSONB），追加版本变更记录 |
| 修改后的文案 | 更新 `content_drafts` 表对应平台的 title/body/hashtags 字段 |
| 修改后的封面 | 更新 `cover_images` 表对应平台的 file_path 和 metadata |
| 修改后的视频 | 更新 `video_drafts` 表对应平台的 file_path 和 metadata |

## 4. 核心处理逻辑

### 4.1 处理步骤

```
Step 1: 输入校验与版本加载
  - 做什么: 校验输入格式，加载当前版本完整内容
  - 怎么判断:
      - 使用 jsonschema 库校验输入格式
      - 查询 content_drafts 表获取当前版本号（version_number）
      - 校验 review_feedback.items 非空
      - 校验 current_version 中的 draft_ids 有效
      - 如果 version_number 已达 max_edit_iterations，返回 status=failed
      - 加载完整内容: 文案文本、封面文件路径、视频文件路径
  - 输出: 当前版本完整内容快照（content_snapshot）、修改迭代计数

Step 2: 反馈解析与优先级排序
  - 做什么: 解析审阅反馈条目，按优先级排序
  - 怎么判断:
      - 按 priority 排序: critical → high → medium → low
      - 按 target 分组: copy → cover → video → metadata
      - 每个组内按 target_platform 细分
      - 检测冲突: 同一 target + 同一 platform 的多条反馈是否冲突
      - 冲突处理: 保留优先级更高的，标注冲突信息
      - 去重: 相同 description + target + platform 的条目去重
  - 输出: 排序后的修改任务队列（edit_queue）

Step 3: 文案修改执行
  - 做什么: 逐条执行 target='copy' 的修改
  - 怎么判断:
      - 对每条 copy 修改:
        - action='modify': 使用 LLM 根据 description + context 修改文案
          - Prompt: "根据以下反馈修改文案：{description}。原始文本：{original_text}。建议修改：{suggested_change}。保持其他内容不变，仅修改指定部分。"
          - 如果 suggested_change 存在，优先采用；否则由 LLM 自主修改
        - action='replace': 直接替换为 context.suggested_change
        - action='remove': 删除 context.section 指定的段落
        - action='add': 在指定位置插入 context.suggested_change
        - action='reorder': 调整段落顺序
      - 修改后校验:
        - 字数不超平台限制
        - 关键信息未丢失（使用 diff 对比修改前后）
      - 记录 before_snapshot / after_snapshot
  - 输出: 修改后的文案 + 变更记录

Step 4: 封面修改执行
  - 做什么: 逐条执行 target='cover' 的修改
  - 怎么判断:
      - 对每条 cover 修改:
        - action='modify': 调用封面生成模块的局部修改能力:
          - 文字颜色/字体/位置调整
          - 装饰元素调整
          - Logo 位置/大小调整
        - action='replace': 重新调用 M-09 生成新封面（传入修改后的参数）
        - action='remove': 移除特定元素（如移除水印）
        - action='retry': 使用不同的底图候选重新生成
      - 修改后校验:
        - 图片文件存在
        - 尺寸符合平台规格
        - 文件大小在合理范围内
      - 记录 before_snapshot（原始文件路径）/ after_snapshot（新文件路径）
  - 输出: 修改后的封面文件 + 变更记录

Step 5: 视频修改执行
  - 做什么: 逐条执行 target='video' 的修改
  - 怎么判断:
      - 对每条 video 修改:
        - action='modify': 使用 FFmpeg 进行局部修改:
          - 替换/添加片段（开头 Logo 动画、结尾 CTA）
          - 调整音频（BGM 音量、配音速度）
          - 字幕样式调整
        - action='replace': 重新调用 M-10 生成新视频（传入修改参数）
        - action='remove': 移除指定片段
        - action='retry': 使用不同编辑参数重新剪辑
      - 修改后校验:
        - 视频文件存在且可播放
        - 分辨率、帧率、码率符合平台规格
        - 时长在目标范围内
      - 记录 before_snapshot / after_snapshot
  - 输出: 修改后的视频文件 + 变更记录

Step 6: 合规性自检
  - 做什么: 调用 M-12 对修改后的内容进行合规检查
  - 怎么判断:
      - 调用 M-12 API: POST /api/modules/M-12/check
      - 传入参数: 修改后的 copy_text（所有平台）、cover_path（所有平台）、video_path（所有平台）
      - 等待 M-12 返回合规报告
      - 根据合规结果决定后续流程:
        - overall_status='pass': 进入 Step 7
        - overall_status='risk_detected': 检查风险等级
          - 无 critical/high 风险: 添加 warning 后继续
          - 有 critical/high 风险: 返回 revision_required，附带风险详情
        - overall_status='fail': 返回 compliance_failed
  - 输出: 合规检查结果（compliance_result）

Step 7: 版本变更日志生成
  - 做什么: 生成版本变更日志
  - 怎么判断:
      - 汇总所有 applied/skipped/failed 的变更记录
      - 生成 markdown 格式的变更日志:
        ```markdown
        ## v2 变更日志 (2026-07-12)
        ### 文案修改 (2 项)
        - [douyin] 第二段语气改为口语化 ✅
        - [x] 最后一句改为问句 ✅
        ### 封面修改 (1 项)
        - [all] 标题颜色改为品牌橙色 #FF6B35 ✅
        ### 视频修改 (1 项)
        - [bilibili] 开头加入品牌 Logo 动画 ✅
        ```
      - 写入 content_drafts 表的 change_log 字段
      - 更新 version_number = current + 1
  - 输出: 变更日志 + 更新后的版本信息

Step 8: 打包与入队
  - 做什么: 将定稿内容打包为待发件，写入分发队列
  - 怎么判断:
      - 如果 status='finalized':
        - 生成 package_id（格式: PKG-YYYY-NNNNNN）
        - 为每个目标平台打包:
          - 最终文案文本
          - 最终封面图路径（压缩版）
          - 最终视频路径（如有）
          - 话题标签列表
        - 写入 publish_queue 表
        - 更新 content_drafts 表 status='final'
        - 更新 cover_images 表 status='final'
        - 更新 video_drafts 表 status='final'
      - 如果 status='revision_required':
        - 更新 content_drafts 表 status='revision_required'
        - 通知 M-13 进行下一轮审阅
      - 如果 status='compliance_failed':
        - 更新 content_drafts 表 status='compliance_failed'
        - 附加合规失败详情，通知运营人员
  - 输出: publish_package（如 finalized）或错误状态

Step 9: 结果汇总与写入
  - 做什么: 汇总处理结果，写入数据库
  - 怎么判断:
      - 写入 module_outputs 表
      - 更新 content_drafts 表的状态和版本信息
      - 生成执行摘要
  - 输出: 最终的 Output JSON
```

### 4.2 修改类型处理矩阵

| Action | copy 处理方式 | cover 处理方式 | video 处理方式 |
|--------|-------------|--------------|--------------|
| **modify** | LLM 局部修改文本 | Pillow 局部调整元素 | FFmpeg 局部调整 |
| **replace** | 直接替换文本 | 重新调用 M-09 生成 | 重新调用 M-10 生成 |
| **remove** | 删除指定段落 | 移除指定元素 | 移除指定片段 |
| **add** | 插入新段落 | 添加新元素 | 添加新片段 |
| **reorder** | 调整段落顺序 | — | 调整片段顺序 |
| **retry** | 重新生成文案 | 换底图重新生成 | 换参数重新剪辑 |

## 6. 错误处理策略

| 错误类型 | 处理方式 | 重试策略 | 告警级别 |
|---------|---------|---------|---------|
| 输入 JSON Schema 校验失败 | 返回 status=failed，终止执行 | 不重试 | ERROR |
| content_draft_id 无效 | 返回 status=failed | 不重试 | ERROR |
| review_feedback.items 为空 | 跳过修改，直接进入合规检查和打包 | 不重试 | INFO |
| 版本迭代次数超过 max_edit_iterations | 返回 status=failed，错误: "超过最大修改迭代次数" | 不重试 | CRITICAL |
| 文案修改 LLM 调用失败 | 使用 context.suggested_change 直接替换（如有）；否则标记该条目为 failed | 2 次 | WARN |
| 封面修改失败（M-09 不可用） | 保留原始封面，标记该条目为 failed | 1 次重试 | WARN |
| 视频修改失败（FFmpeg 执行错误） | 保留原始视频，标记该条目为 failed | 1 次降分辨率重试 | WARN |
| M-12 合规检查超时（> 30s） | 降级: 跳过合规检查，标记 status=finalized 但添加 warning: "合规检查超时，建议人工复核" | 不重试 | WARN |
| M-12 合规检查服务不可用（5xx） | 标记 status=finalized，添加 warning: "合规检查服务不可用" | 不重试 | CRITICAL |
| 数据库写入失败 | 暂存到本地 JSON 文件 `data/m11_backup_{task_id}.json` | 5 次，间隔 2s | CRITICAL |
| 发布队列写入失败 | 标记 status=finalized 但 publish_package 为空，通知人工手动入队 | 3 次 | CRITICAL |
| 部分平台修改失败 | 继续处理其他平台，status=partial（最终为 revision_required） | 不重试（失败平台） | WARN |

### 降级策略

1. **LLM 不可用**: 文案修改降级为直接使用 suggested_change 替换；如无 suggested_change，标记失败并保留原文
2. **M-12 完全不可用**: 跳过合规检查，内容标记为「待人工合规审核」
3. **M-09/M-10 不可用**: 封面/视频修改无法重新生成，仅保留已有的局部修改
4. **发布队列不可用**: 内容标记为 finalized 但停留在 content_drafts 表，待队列恢复后自动入队

### 验收标准
- [ ] 输入校验通过: 无效 draft_id 应返回 status=failed
- [ ] 核心逻辑无异常: dry-run 和执行模式均无 Python Exception
- [ ] 输出格式匹配 Schema: 包含 version_info、change_log、compliance_result
- [ ] 反馈条目正确排序: critical → high → medium → low
- [ ] 文案修改成功应用: change_log 中 copy 条目 before/after 有差异
- [ ] 封面修改成功应用: 封面文件已更新，颜色/位置已修改
- [ ] 视频修改成功应用: 视频文件已更新，片段已添加/移除
- [ ] 合规检查已执行: compliance_result 包含有效评分和风险列表
- [ ] 合规通过后正确定稿: status=finalized, publish_package 非空
- [ ] 合规不通过正确返回: status=revision_required 或 compliance_failed
- [ ] 版本号正确递增: version_number = 原 + 1
- [ ] 变更日志完整: 包含所有条目的处理状态和快照
- [ ] 发布队列正确写入: publish_queue 表有对应记录
- [ ] 修改迭代限制生效: 超过 max_edit_iterations 返回 failed
- [ ] 错误场景有降级处理: LLM/M-12 不可用时能正确降级
- [ ] 日志输出完整: 包含 Step 1-9 的执行记录和耗时统计

### 注意事项
- 本模块是内容生产流水线的最后一道工序，修改质量直接影响发布内容
- 合规检查是强制步骤，不可跳过（除非配置 skip_compliance=true 用于测试）
- LLM 辅助修改仅用于文案，封面和视频修改依赖 M-09/M-10 的重新生成能力
- 修改迭代次数限制用于防止无限修改循环，默认 3 次
- 变更日志记录每个条目的 before/after 快照，用于后续 M-25 分析
- package_id 格式为 PKG-YYYY-NNNNNN，由数据库序列生成
- 发布队列中的内容由 M-16（定时分发模块）按 scheduled_at 时间消费
- 如果合规检查超时，内容仍可发布但需人工复核
