# M-24 下期计划草案生成

---

## 1. 模块定位

| 属性 | 值 |
|------|-----|
| **模块编号** | M-24 |
| **模块名称** | 下期计划草案生成 |
| **所属分类** | 数据分析 |
| **优先级** | P2（重要模块，数据分析链路的终点，连接策略与执行） |
| **触发方式** | 定时任务（每日 14:00 UTC+8，在所有上游分析模块完成后执行）；支持手动触发 |
| **在系统中的位置** | 作为数据分析链路的汇聚节点，综合 M-20（清洗数据）、M-21（效果分析）、M-22（语义分析）、M-23（热点追踪）的分析结果，结合策略偏好库，生成下一轮内容发布计划草案，输出供 M-13（审核⑤）使用 |

### 1.1 核心职责

1. **综合分析汇聚**：从 M-20、M-21、M-22、M-23 拉取最新分析结果，进行交叉综合分析。
2. **内容计划生成**：综合各维度分析，生成"下轮内容主题计划（草案）"，包含建议平台、热点方向、内容形式、爆款参考特征。
3. **答疑脚本大纲**：基于评论痛点分析，生成"答疑脚本大纲"草案，回应粉丝关切。
4. **热点 × 痛点交叉匹配**：将热点话题与用户痛点进行交叉匹配，生成 3-5 个优先选题。
5. **策略偏好库应用**：加载策略偏好库，根据历史策略有效性调整计划权重。
6. **草案输出**：生成结构化的计划草案文档（JSON + Markdown），提交至 M-13 进行人工审核。

### 1.2 上下游关系

```
M-20(数据清洗与聚合) ──┐
M-21(多维度效果分析) ──┤
                        ├──→ M-24(本模块) → M-13(审核⑤/人工审核)
M-22(评论区语义挖掘) ──┤
M-23(跨平台热点追踪) ──┘
```

- **上游**：M-20、M-21、M-22、M-23 四个分析模块
- **下游**：M-13（审核环节⑤），人工审核通过后进入 M-14 内容生产队列

### 4.2 热点 × 痛点交叉匹配算法

```
Function cross_match_hot_pain(hot_topics, pain_points):
    candidates = []

    FOR each hot in hot_topics:
        FOR each pain in pain_points:
            # 语义匹配度
            semantic_match = calculate_semantic_match(
                hot.keyword,
                pain.keyword,
                hot.description,
                pain.representative_comment
            )

            # 协同得分
            hot_score = hot.composite_score
            pain_severity = SEVERITY_WEIGHT[pain.severity]
            # critical=1.0, major=0.7, minor=0.4

            synergy_score = (
                semantic_match * 0.40 +
                hot_score * 0.30 +
                pain_severity * 0.30
            )

            IF synergy_score > 0.5:  # 最低阈值
                candidates.append({
                    "hot_topic": hot.keyword,
                    "pain_point": pain.keyword,
                    "synergy_score": synergy_score,
                    "hot_score": hot_score,
                    "pain_severity": pain.severity,
                    "suggested_angle": generate_cross_angle(hot, pain)
                })

    # 去重：语义相似度 > 0.8 的选题合并
    deduped = deduplicate_by_semantic_similarity(candidates, threshold=0.8)

    # 排序取 TOP 3-5
    sorted_candidates = sort_by(deduped, "synergy_score", descending=True)
    return sorted_candidates[:5]
```

### 4.3 内容配比计算

```
Function calculate_content_mix(strategy, total_days, constraints):
    total_slots = 0
    FOR day in range(total_days):
        IF day.date NOT in blackout_dates:
            total_slots += min(
                constraints.max_daily_posts,
                remaining_capacity
            )

    # 应用策略配比
    hot_chase_count = round(total_slots * strategy.hot_chase_ratio)
    original_count = round(total_slots * strategy.original_ratio)
    qa_response_count = round(total_slots * strategy.qa_response_ratio)
    series_count = total_slots - hot_chase_count - original_count - qa_response_count

    # 应用内容类型限制
    hot_chase_count = min(hot_chase_count, constraints.content_type_limits.get("hot_chase", 999))
    original_count = min(original_count, constraints.content_type_limits.get("original", 999))
    # ... etc

    RETURN {
        "hot_chase": hot_chase_count,
        "original": original_count,
        "qa_response": qa_response_count,
        "series_continuation": series_count,
        "total": total_slots
    }
```

```
Function generate_qa_script(pain_points, high_value_comments, content_context):
    sections = []

    # 按类别分组痛点

    # 按严重程度排序组

        4. 控制在 200 字以内

    # 生成核心信息点
    key_messages = extract_key_messages(sections)

    # 生成行动号召
    call_to_action = generate_cta(content_context)

        "sections": sections,
        "key_messages": key_messages,
        "call_to_action": call_to_action
```

## 7. 自学习机制

### 7.1 选题效果反馈

```
计划执行后，追踪每个 slot 的实际表现：
- 从 M-18/M-20 获取实际 engagement_rate
- 与 expected_engagement_rate 对比
- 计算预测准确度 MAPE

每周分析：
- IF 某种选题来源的实际表现持续低于预期：
  → 降低该来源的 content_mix_strategy 权重
- IF 某种选题来源的实际表现持续高于预期：
  → 提升该来源的 content_mix_strategy 权重
```

### 7.2 交叉匹配模型优化

```
记录每次热点 × 痛点交叉匹配的结果质量：
- 运营人员对 priority_topics 的采纳率
- 采纳后的内容表现

- IF 采纳率高（> 60%）且内容表现好：
  → 提升交叉匹配的 synergy_score 权重
- IF 采纳率低（< 20%）：
  → 调整语义匹配模型或降低匹配阈值
```

### 7.3 策略偏好库动态更新

```
每月评估策略有效性：
- 对比应用策略的内容 vs 未应用策略的内容的表现
- IF 策略有效（表现提升 > 10%）：
  → 策略 effectiveness_score +5
- IF 策略无效（表现无提升或下降）：
  → 策略 effectiveness_score -5
- IF effectiveness_score < 30：
  → 标记策略为待废弃，降低权重
```

```
你作为 M-24 下期计划草案生成模块的 Hermes Agent，请生成紧急计划草案：

触发条件：检测到突发热点或重大舆情事件

1. 加载最新 M-23 热点数据（优先处理 P0_immediate 话题）
2. 跳过完整上游分析，仅加载必要数据
3. 快速生成 24-48 小时内的紧急内容计划：
   - 优先填充 1-2 个紧急热点内容
   - 调整后续计划为新内容腾出空间
4. 输出简化版草案，标记为 "urgent"
5. 立即通知 M-13 进行快速审核
```

### 9.3 计划修订指令

```
你作为 M-24 下期计划草案生成模块的 Hermes Agent，请根据审核反馈修订计划草案：

输入参数：
- draft_id: 被驳回的草案 ID
- review_comments: M-13 的审核意见

执行流程：
1. 加载原始草案
2. 解析审核意见中的修改要求
3. 对受影响的 slot 进行重新生成或调整
4. 保持未受影响的部分不变
5. 更新计划草案，状态改为 "revised"
6. 重新提交 M-13 审核
```
