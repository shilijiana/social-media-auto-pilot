```
M-01 位置说明:
  上游: topics 表（选题计划表） → 读取原始关键词
  下游: M-02（图文素材采集）、M-03（视频素材采集）、M-04（参考文案与竞品采集）
  自学习: 向 M-25 写入关键词-内容表现关联数据
```

### 1.2 核心职责

读取选题计划表中的原始关键词，通过 LLM 生成同义词组、相关词组和长尾词组，并根据各平台特征生成平台适配的关键词变体。输出带类型标签的扩展关键词列表，写入 `keywords` 表，供下游素材采集模块使用。

### 1.3 触发时机

- **自动触发**: 选题计划确认后（`topics` 表 status 变为 `confirmed`），由流水线引擎自动调用
- **人工触发**: 运营人员在选题管理后台对单条选题执行「重新扩展关键词」
- **定时触发**: 每周日 02:00 UTC 执行全量关键词重评估（配合 M-23 热点追踪数据更新）

## 4. 核心处理逻辑

### 4.1 处理步骤

```
Step 1: 输入校验
  - 做什么: 校验 task_id 格式、topic_id 格式、source_keywords 非空且不超过 20 个、target_platforms 合法
  - 怎么判断: 使用 jsonschema 库校验，对比 2.1 中的 Schema
  - 输出: 校验通过 → 进入 Step 2; 校验失败 → 返回 status=failed，记录错误详情

Step 2: 加载偏好规则
  - 做什么: 如果输入包含 preference_rule_ids，从 M-25 偏好库加载关键词偏好规则
  - 怎么判断:
      - 调用 M-25 偏好查询接口，传入 preference_rule_ids
      - 若命中规则（置信度 >= 70%），将规则注入 LLM prompt 作为上下文约束
      - 若无偏好规则，使用 config/keyword_expansion_rules.yaml 中的默认规则
  - 输出: 规则上下文对象 `rule_context`

Step 3: LLM 同义词生成
  - 做什么: 调用 LLM 为每个原始关键词生成同义词组
  - 怎么判断:
      - 使用 Prompt: `prompts/keyword_synonym_generation.md`
      - 每个原始关键词请求 LLM 生成最多 `synonym_count` 个同义词
      - LLM 返回的同义词需附带语义相似度评分（0-1）
      - 过滤掉相似度 < `min_relevance_score` 的结果
  - 输出: `synonym_groups` 对象（每个原始关键词对应一个同义词组）

Step 4: LLM 相关词生成
  - 做什么: 调用 LLM 为每个原始关键词生成相关词
  - 怎么判断:
      - 使用 Prompt: `prompts/keyword_related_generation.md`
      - 相关词定义为：与原始关键词属于同一主题域但不是同义词的词
      - 每个原始关键词请求 LLM 生成最多 `related_count` 个相关词
      - 过滤掉关联度 < `min_relevance_score` 的结果
  - 输出: 每个原始关键词的 `related` 类型扩展词列表

Step 5: LLM 长尾词生成
  - 做什么: 调用 LLM 为每个原始关键词生成长尾关键词
  - 怎么判断:
      - 使用 Prompt: `prompts/keyword_long_tail_generation.md`
      - 长尾词定义为：包含原始关键词的更长、更具体的搜索短语
      - 每个原始关键词请求 LLM 生成最多 `long_tail_count` 个长尾词
      - 长尾词需考虑用户搜索意图（信息型、导航型、交易型）
  - 输出: 每个原始关键词的 `long_tail` 类型扩展词列表

Step 6: 平台适配变体生成
  - 做什么: 将 Step 3-5 生成的关键词，按平台特征生成平台特定变体
  - 怎么判断:
      - 读取 config/platform_keyword_profiles.yaml 中的平台关键词规则
      - 对每个目标平台，调用 LLM 进行关键词风格转换
      - 使用 Prompt: `prompts/keyword_platform_adaptation.md`
  - 输出: 每个 (原始关键词, 平台) 组合的扩展词列表

Step 7: 去重与质量过滤
  - 做什么: 对生成的所有扩展词进行去重和质量过滤
  - 怎么判断:
      - 同义词组内去重：同一 synonym_group 中的词，保留 representative 优先
      - 跨平台去重：同一关键词在不同平台出现时，保留平台特定版本
      - 质量过滤：移除包含敏感词、无效字符、过短（< 2 字）或过长（> 30 字）的扩展词
      - 数量上限：每个 (原始关键词, 平台) 组合最多保留 50 个扩展词
  - 输出: 过滤后的扩展词列表

Step 8: 写入数据库与返回
  - 做什么: 将扩展关键词写入 `keywords` 表和 `keyword_synonym_groups` 表
  - 怎么判断:
      - 写入前检查 `keywords` 表是否已有同一 topic_id + platform 的关键词记录
      - 若已有记录，对比差异并记录变更日志到 `keyword_change_logs` 表
      - 写入 `module_outputs` 表完整输出 JSON
      - 返回最终的 Output JSON
  - 输出: 最终的 Output JSON
```

### 4.2 平台适配

| 平台 | 差异处理 |
|------|---------|
| douyin | 关键词偏好：短促有力（2-5字）、口语化、带感叹/疑问语气、动词结构、热梗化表达。示例：「AI有多强」「这波操作」「绝了」 |

## 6. 错误处理策略

| 错误类型 | 处理方式 | 重试策略 | 告警级别 |
|---------|---------|---------|---------|
| 输入 JSON Schema 校验失败 | 返回 status=failed，错误详情写入 `errors` 表，终止执行 | 不重试 | ERROR |
| `topics` 表查询失败（表不存在/连接断开） | 返回 status=failed，记录错误日志，通知流水线引擎 | 不重试 | CRITICAL |
| `topics` 表中对应 topic_id 无关键词数据 | 返回 status=failed，记录「选题无关键词」错误 | 不重试 | WARN |
| LLM 同义词生成返回异常（500/429/超时） | 降级：使用内置同义词词典进行规则匹配（`config/synonym_dictionary.yaml`） | 重试 1 次 | WARN |
| LLM 相关词生成返回异常 | 降级：使用 M-23 热点关键词作为相关词补充 | 重试 1 次 | WARN |
| LLM 长尾词生成返回异常 | 降级：使用模板拼接生成基础长尾词（原始词 + 平台特征后缀） | 重试 1 次 | WARN |
| LLM 平台适配返回异常 | 降级：直接使用通用扩展词，不进行平台适配 | 重试 1 次 | WARN |
| 偏好规则查询超时（> 5s） | 降级：使用 `config/keyword_expansion_rules.yaml` 中的默认规则 | 不重试 | WARN |
| `keywords` 表写入失败（连接断开/锁超时） | 暂存结果到本地 JSON 文件，发送告警 | 5次，间隔 2s | CRITICAL |
| 扩展词生成数量为 0（全部被过滤） | 返回 status=partial，保留原始关键词作为最小兜底，记录告警 | 不重试 | WARN |
| 全部原始关键词扩展失败（status=failed） | 通知流水线引擎终止下游模块，写入 `pipeline_errors` 表 | 不重试 | CRITICAL |

### 降级策略

当模块完全不可用时（如 LLM 服务中断 + 内置词典损坏 + 热点数据不可达），启用以下降级方案：

1. **兜底关键词**: 将原始关键词直接作为扩展结果返回（type=original），确保下游模块至少可以基于原始关键词工作
2. **人工介入**: 发送企业微信/钉钉通知给运营团队，附上选题 ID 和原始关键词，由人工在后台手动扩展
3. **跳过本模块**: 在 `pipeline_config` 中设置 `keyword_expander.skip_on_failure = true`，流水线使用原始关键词继续后续流程

## 7. 自学习触发条件

### 7.1 学习数据收集

本模块向 `keywords` 表写入以下字段，供 M-25 定期收集分析：

| 数据字段 | 说明 | 收集时机 |
|---------|------|---------|
| `keyword` | 扩展后的关键词 | 每次关键词扩展后立即写入 |
| `source_keyword` | 原始关键词 | 每次关键词扩展后立即写入 |
| `type` | 扩展类型（synonym/related/long_tail） | 每次关键词扩展后立即写入 |
| `platform` | 目标平台 | 每次关键词扩展后立即写入 |
| `relevance_score` | LLM 评估的关联度 | 每次关键词扩展后立即写入 |
| `used_by_module` | 被下游模块使用的记录（M-02/M-03/M-04） | 下游模块查询 keywords 表后回写 |
| `content_performance_score` | 使用该关键词的内容最终表现评分 | 内容发布 7 天后，从 M-18 效果数据关联计算 |

### 7.2 偏好库调用

- **调取偏好类型**: 关键词风格偏好
- **置信度阈值**: >= 70%
- **无偏好时默认行为**: 使用 `config/keyword_expansion_rules.yaml` 中的 `default_template_set`，基于关键词类别进行规则匹配

### 7.3 阶段演进行为

| 阶段 | 行为差异 |
|------|---------|
| P0 (人工主导) | 关键词扩展完全由运营人员在后台手动输入。模块仅负责格式校验和写入 `keywords` 表，不参与生成。LLM 不参与任何生成步骤。 |
| P1 (人机协同) | 模块自动生成扩展关键词推荐列表，运营人员审核后确认。LLM 参与同义词和相关词生成，但平台适配由人工选择。`synonym_count` 建议设为 8，`related_count` 建议设为 8。 |
| P2 (系统主导) | 模块自动生成并写入扩展关键词，仅在关联度 < 0.6 或平台适配评分 < 60 时通知人工复核。LLM 参与全部生成步骤。偏好库置信度 >= 70% 时使用偏好规则。 |
