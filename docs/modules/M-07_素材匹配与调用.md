```
M-07 位置说明:
  上游: M-06（素材入库与版权追踪模块）→ 提供已入库的素材库（含版权状态和优先级）
        M-13（分级审阅引擎模块）→ 审阅①通过后传入已批准的话题计划
  下游: M-08（文案差异化生成模块）→ 使用匹配的素材生成平台差异化文案
        M-09（封面图生成模块）→ 使用匹配的素材生成封面图
        M-10（视频初剪模块）→ 使用匹配的素材进行视频初剪
  自学习: M-25 收集匹配成功率、短缺模式数据，优化匹配策略
```

### 1.2 核心职责

根据已通过审阅①的话题计划（来自 `topics` 表），按话题 ID 从素材库（`materials` 表）中智能匹配所需素材。优先使用通用素材池（M-15）中的公共素材，其次从版权已确认的专用素材中筛选。当素材覆盖不足时，自动生成「待补充清单」并通过企业微信/钉钉通知运营人员。输出完整的匹配素材列表和短缺报告（如有），供下游内容生产模块（M-08/M-09/M-10）使用。

### 1.3 触发时机

- **自动触发**: M-13 审阅①（计划表审阅）返回 pass 且 mode_annotation 不为 "rejected" 时，由流水线引擎自动调用。传入话题计划 ID 和审阅结果
- **人工触发**: 运营人员在素材管理后台对某个话题手动点击「重新匹配素材」
- **定时触发**: 每日 03:00 UTC 扫描所有已批准但未完成匹配的话题计划，自动触发匹配（处理积压话题）

### 2.3 数据来源

| 字段 | 来源 |
|------|------|
| `task_id` | 流水线引擎生成 |
| `topic_plan_id` | `topics` 表主键，由 M-24（下期计划草案生成）或人工创建 |
| `review_result` | M-13（分级审阅引擎）审阅① 输出，从 `review_tasks` 表读取 |
| `match_options` | 配置文件 `config/material_matching.yaml` 加载，支持运行时覆盖 |
| 话题的 required_material_types | `topics` 表的 `required_materials` 字段（JSONB），定义该话题需要的素材类型和数量 |
| 话题的 tags | `topics` 表的 `tags` 字段（数组），用于素材标签匹配 |
| 素材库数据 | `materials` 表，由 M-06 入库维护 |
| 通用素材池数据 | M-15（通用素材池管理模块）管理的共享素材池 |

## 4. 核心处理逻辑

### 4.1 处理步骤

```
Step 1: 输入校验与话题加载
  - 做什么: 校验 task_id 格式、topic_plan_id 格式，确认审阅结果为 pass
  - 怎么判断:
      - 使用 jsonschema 库校验输入格式
      - 查询 topics 表: SELECT * FROM topics WHERE topic_plan_id = '<输入值>'
      - 确认 topic 状态为 'reviewed_approved'（审阅①通过）
      - 如果 review_result.decision != 'pass'，返回 status=failed，错误信息: "话题计划未通过审阅①，无法进入素材匹配"
      - 提取话题的 required_materials（素材需求列表）和 tags（话题标签）
  - 输出: 话题完整信息（topic_info）、素材需求清单（required_materials）、话题标签列表（topic_tags）

Step 2: 话题标签扩展
  - 做什么: 对话题标签进行语义扩展，生成更丰富的搜索标签集
  - 怎么判断:
      - 从 topic_tags 提取原始标签列表
      - 调用 LLM（gemini-2.5-flash）对标签进行语义扩展:
        - 同义词扩展: "AI技术" → ["人工智能", "机器学习", "智能技术"]
        - 上下位扩展: "编程开发" → ["软件开发", "代码", "Python编程", "Web开发"]
        - 关联扩展: "AI技术" → ["大模型", "深度学习", "神经网络"]
      - 每个原始标签生成 3-8 个扩展标签
      - 扩展标签带有 similarity_to_original 评分（0-1）
      - 合并原始标签 + 扩展标签，去重
  - 输出: 扩展后的标签集 expanded_tags（含原始标签和扩展标签）

Step 3: 通用素材池优先查询（M-15）
  - 做什么: 优先从通用素材池中查询匹配素材
  - 怎么判断:
      - 如果 match_options.prefer_common_pool = false，跳过此步骤
      - 查询 M-15 通用素材池: 通过 common_pool_materials 表或 M-15 API 接口
      - 使用 expanded_tags 进行标签匹配（hybrid 模式: 精确匹配优先，语义匹配补充）
      - 筛选条件:
        - is_copyright_confirmed = true（或 match_options.require_copyright_confirmed = false）
        - quality_score >= match_options.min_quality_score
        - 不在近期已使用列表中（30 天内使用过的素材降低优先级，避免重复使用）
      - 按 relevance_score 降序排列，取前 max_materials_per_type 个
      - 标记 match_source = "common_pool"
  - 输出: common_pool_matches（按素材类型分组）

Step 4: 专用素材库补充查询
  - 做什么: 从专用素材库（materials 表）中查询补充素材
  - 怎么判断:
      - 对于 Step 3 中未满足的素材需求，从 materials 表查询
      - 使用相同的 expanded_tags 进行标签匹配
      - 筛选条件同上，额外条件: material_id NOT IN (已匹配的素材 ID 列表)
      - 按 priority_flag 降序、relevance_score 降序排列
      - 取前 (required_count - 已匹配数量) 个
      - 标记 match_source = "dedicated_library"
  - 输出: library_matches（按素材类型分组）

Step 5: 降级匹配（未确认版权素材）
  - 做什么: 当版权已确认素材仍不足时，考虑使用未确认版权的素材作为降级方案
  - 怎么判断:
      - 如果 match_options.include_unconfirmed_as_fallback = false，跳过此步骤
      - 对于仍未满足的素材需求，从 materials 表查询 is_copyright_confirmed = false 的素材
      - 使用相同的 expanded_tags 进行标签匹配
      - 取前 (required_count - 已匹配数量) 个
      - 标记 match_source = "unconfirmed_fallback"
      - 每个降级素材添加 warning: "版权未确认，建议人工审核后使用"
  - 输出: fallback_matches（按素材类型分组）

Step 6: 关联度评分计算
  - 做什么: 对每个匹配到的素材计算综合关联度评分
  - 怎么判断:
      - relevance_score 计算公式:
        - 标签匹配分 (50%): 匹配到的标签数量 / 话题标签总数 * 100
        - 语义相似分 (25%): LLM 语义相似度评分 (0-100)
        - 质量加权分 (15%): quality_score * 0.15
        - 新鲜度分 (10%): 未使用过的素材 +10，7天内使用过的 -5，30天内的 -2
      - relevance_score = sum of above weighted components
      - 最终得分 0-100
  - 输出: 每个素材的 relevance_score

Step 7: 覆盖率计算与短缺分析
  - 做什么: 计算各类素材的覆盖率，生成短缺清单
  - 怎么判断:
      - 按素材类型分组统计:
        - coverage = matched_count / required_count
        - 如果 matched_count < required_count → 该类型存在短缺
      - overall_coverage = SUM(每种类型的 matched_count) / SUM(每种类型的 required_count)
      - 短缺项优先级判定:
        - critical: 核心素材类型（如话题主图）完全缺失
        - high: 重要素材类型匹配率 < 50%
        - medium: 匹配率 50%-80%
        - low: 匹配率 >= 80% 但未达到 min_coverage_threshold
      - 为每个短缺项生成 suggested_keywords（基于话题标签和扩展标签）
      - 生成 recommendations（运营建议）
      - 如果 overall_coverage < match_options.min_coverage_threshold:
        - 生成 shortage_list
        - 触发通知（企业微信/钉钉）
        - status = "partial"
      - 如果 overall_coverage >= match_options.min_coverage_threshold:
        - status = "completed"
        - shortage_list 为空
  - 输出: coverage_report（含 shortage_list 和 recommendations）

Step 8: 结果写入与通知
  - 做什么: 将匹配结果写入数据库，发送短缺通知
  - 怎么判断:
      - 写入 material_matches 表（匹配记录）
      - 写入 module_outputs 表（模块输出）
      - 如果有短缺:
        - 写入 material_shortages 表
        - 通过 notifications 表创建通知记录
        - 调用企业微信/钉钉 Webhook 发送通知（包含 shortage_list 摘要和操作链接）
      - 写入 match_statistics 表（供 M-25 自学习）
  - 输出: 最终的 Output JSON
```

### 4.2 标签匹配策略

| 策略 | 描述 | 适用场景 | 性能 |
|------|------|---------|------|
| `exact` | 仅匹配完全一致的标签名 | 标签体系规范、标签数量充足时 | 最快（数据库索引直接查询） |
| `semantic` | 使用向量嵌入计算标签语义相似度，匹配 similarity >= 0.75 的标签 | 标签体系松散、跨语言场景 | 较慢（需向量检索） |
| `hybrid` | 优先使用 exact 匹配，未命中时使用 semantic 补充 | 推荐默认策略，兼顾准确性和召回率 | 适中 |

## 6. 错误处理策略

| 错误类型 | 处理方式 | 重试策略 | 告警级别 |
|---------|---------|---------|---------|
| 输入 JSON Schema 校验失败 | 返回 status=failed，错误详情写入 `errors` 表，终止执行 | 不重试 | ERROR |
| topic_plan_id 不存在 | 返回 status=failed，错误信息: "话题计划不存在" | 不重试 | ERROR |
| 话题计划未通过审阅① | 返回 status=failed，错误信息: "话题计划审阅状态不符" | 不重试 | WARN |
| topics 表查询超时（> 5s） | 重试查询，使用备库读取 | 3次，间隔 1s | WARN |
| materials 表查询超时（> 10s） | 降级：仅使用通用素材池（M-15）的缓存数据 | 2次，间隔 2s | WARN |
| LLM 标签扩展返回异常（500/429/超时） | 降级：仅使用原始标签进行精确匹配（exact 模式） | 重试 1 次 | WARN |
| M-15 通用素材池不可用 | 跳过通用素材池，直接查询专用素材库 | 不重试 | WARN |
| pgvector 向量检索超时 | 降级：切换为 exact 模式（纯数据库标签匹配） | 不重试 | WARN |
| 数据库写入失败（连接断开/锁超时） | 暂存结果到本地 JSON 文件 `data/m07_backup_{task_id}.json`，发送告警 | 5次，间隔 2s | CRITICAL |
| 企业微信/钉钉通知发送失败 | 记录到 `notification_retry_queue` 表，定时重试 | 3次，间隔 5min | WARN |
| 素材库完全为空（materials 表无记录） | 返回 status=failed，生成完整 shortage_list（所有 required_materials 均为短缺），通知运营人员 | 不重试 | CRITICAL |

### 降级策略

当模块完全不可用时（如数据库离线 + LLM 不可达 + M-15 不可用），启用以下降级方案：

1. **人工匹配**: 发送企业微信/钉钉通知给运营团队，附上话题详情和素材需求清单，由人工在素材管理后台手动选择素材并关联到话题
2. **缓存复用**: 如果 Redis 中缓存了该话题的历史匹配结果（7 天内），使用缓存结果作为临时输出，标记 `match_source = "cached_fallback"`
3. **跳过本模块**: 在 `pipeline_config` 中设置 `material_matcher.skip_on_failure = true`，下游模块（M-08/M-09/M-10）使用话题标签直接从素材库随机选取素材（quality_score 降序）

### 验收标准
- [ ] 输入校验通过: 不存在的 topic_plan_id 应返回 status=failed
- [ ] 审阅状态校验: 未通过审阅①的话题不应进入匹配流程
- [ ] 核心逻辑无异常: dry-run 模式下所有 Step 执行无 Python Exception
- [ ] 输出格式匹配 Schema: 包含 matched_materials、coverage_report、execution_metadata
- [ ] 通用素材池优先: 当 prefer_common_pool=true 时，common_pool 素材排在前面
- [ ] 覆盖率计算正确: by_material_type 中的 matched_count 与实际匹配数量一致
- [ ] 短缺清单完整: 每个未满足的素材类型都有对应的 shortage_list 条目
- [ ] 短缺清单含建议关键词: 每个 shortage item 都有 suggested_keywords
- [ ] 标签扩展正确: 每个原始标签产生 3-8 个扩展标签
- [ ] 版权优先级正确: copyright_confirmed 素材优先于 unconfirmed 素材
- [ ] 素材去重正确: matched_materials 中无重复 material_id
- [ ] 错误场景有降级处理: 模拟 LLM 不可用，确认降级到 exact 模式
- [ ] 日志输出完整: 检查日志包含 Step 1-8 的执行记录和每步耗时统计
- [ ] 通知触发正确: 有短缺时检查 notifications 表有对应记录

### 注意事项
- dry-run 模式不会写入数据库，仅输出到本地 JSON 文件
- 话题计划必须已通过 M-13 审阅①（decision='pass'）才能触发匹配
- 标签扩展依赖 LLM，首次运行前确保 `prompts/tag_expansion.md` 存在且 API Key 配置正确
- 通用素材池（M-15）是一个独立模块，如果尚未实现，可暂时跳过 Step 3，直接从专用素材库匹配
- 短缺清单会触发企业微信/钉钉通知，测试时注意避免频繁通知
- 匹配结果中的素材按 relevance_score 降序排列，下游模块应按此顺序使用
- 如果素材库中完全没有符合条件的数据，模块返回 status=failed 并生成完整 shortage_list
- 降级匹配（unconfirmed_fallback）仅当 include_unconfirmed_as_fallback=true 时才启用
- 素材新鲜度评分基于 usage_count 和 last_used_at，避免重复使用近期素材
- 首次运行前确保数据库迁移脚本已执行: `python -m db.migrate up`
