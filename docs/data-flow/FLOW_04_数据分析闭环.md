# FLOW_04: 数据分析闭环

> 闭环路径: M-20 → M-21 → M-22 → M-23 → M-24
> 闭环目标: 对已发布内容的各项数据进行清洗、分析和挖掘，生成多维度分析报告和下一期计划草案
> 目标读者: Hermes Agent

## 2. 各模块角色与数据转换

### M-20 — 数据清洗与聚合 (Data Cleaning & Aggregation)

| 属性 | 说明 |
|------|------|
| **角色** | 数据分析闭环的**数据入口**，承担数据质量保障职责 |
| **优先级** | P2 |
| **触发方式** | 定时任务（每日 09:00 UTC+8）或手动触发 |

**输入数据**:
- `PerformanceData`（来自 M-18 效果数据采集）
  - `content_id`、`platform`、`views`、`likes`、`comments`、`shares`、`favorites`
  - `completion_rate`、`engagement_rate`、`net_followers`、`watch_time`
  - `collected_at`、`collection_node`（24h / 72h / 7d / 30d）
- `AccountHealth`（来自 M-19 账号健康度监控）
  - `follower_count`、`violation_count`、`health_score`、`shadow_ban_flag`

**数据转换过程**:

```
原始数据 (raw) ──────────────────────────────────────────────► 清洗后数据 (clean)
│                                                                    │
├─ 缺失值检测 ───────────────────────────────────────────────────────┤
│  时间序列缺失 → 线性插值                                           │
│  指标缺失     → 同平台同类内容均值填充                              │
│  关键字段缺失 → 标记为 incomplete，降权处理                         │
│                                                                    │
├─ 异常值检测 ───────────────────────────────────────────────────────┤
│  IQR 方法: Q1-1.5×IQR < x < Q3+1.5×IQR → 正常范围                 │
│  3-Sigma:   |z-score| > 3 → 极端异常，标记 outlier_flag            │
│                                                                    │
├─ 数据标准化 ───────────────────────────────────────────────────────┤
│  views → z-score / min-max                                         │
│  engagement_rate → 保留原始比例                                    │
│                                                                    │
├─ 维度聚合 ─────────────────────────────────────────────────────────┤
│  按 platform → 各平台汇总指标                                      │
│  按 date     → 每日趋势                                            │
│  按 content_type → 内容类型对比                                    │
│  按 collection_node → 时间节点对比                                 │
│                                                                    │
└─ 基准对比 ─────────────────────────────────────────────────────────┤
   与 preference_rules 中的历史基准比较                                │
   偏差 > 2σ → 标记为 anomaly                                        │
```

**输出数据**:
- `CleanedDataset` 结构（写入 `analytics_reports` 表）:
  - `clean_metrics`: 清洗后的指标对象
  - `outlier_flags`: 异常标记列表
  - `agg_views`: 各维度聚合视图
  - `benchmark_diffs`: 基准偏差记录

**关键接口点**:
- 上游: M-18 的 `performance_metrics` 表, M-19 的 `account_health` 表
- 下游: M-21 消费 `analytics_reports` 表

### M-21 — 多维度效果分析 (Multi-Dimensional Performance Analysis)

| 属性 | 说明 |
|------|------|
| **角色** | 数据分析闭环的**核心分析引擎**，将清洗后数据转化为可行动洞察 |
| **优先级** | P2 |
| **触发方式** | 紧接 M-20 完成后自动触发 |

**输入数据**:
- `CleanedDataset`（来自 M-20）
- `preference_rules`（来自 M-29 学习成果应用，用于对比基准）

**六大分析维度**:

| 维度 | 分析内容 | 输出指标 |
|------|---------|---------|
| 平台维度 | 各平台播放量、互动率、粉丝增长横向对比 | `platform_rankings`, `platform_share` |
| 时间维度 | 日/周/月趋势、最佳发布时间窗口 | `best_posting_hours`, `growth_trend` |
| 内容维度 | 按主题/标签/类型聚类，识别高表现特征 | `top_performing_topics`, `content_type_performance` |
| 受众维度 | 粉丝画像、活跃时段、地域分布 | `audience_insights`, `peak_activity_hours` |
| 竞品维度 | 与竞品数据横向对比（需 M-04 数据） | `competitive_position`, `market_share_est` |
| 增长维度 | 粉丝增长率、传播系数、内容产出效率 | `viral_coefficient`, `growth_acceleration` |

**数据转换过程**:

```
CleanedDataset ──────────────────────────────────────────────► AnalyticsReport (partial)
│                                                                    │
├─ 横向对比 (Horizontal) ────────────────────────────────────────────┤
│  同一时间段内不同平台的表现对比                                     │
│  douyin vs xiaohongshu vs bilibili vs twitter                      │
│  → horizontal_comparison: {平台: {指标对象}}                       │
│                                                                    │
├─ 纵向对比 (Vertical) ──────────────────────────────────────────────┤
│  同一平台当前周期 vs 上一周期的表现对比                             │
│  → vertical_comparison: {指标: {current, previous, change_rate}}   │
│                                                                    │
├─ 相关性分析 ───────────────────────────────────────────────────────┤
│  发布时间 ↔ 互动率                                                 │
│  内容长度 ↔ 完播率                                                 │
│  标签数量 ↔ 曝光量                                                 │
│  → correlation_matrix                                              │
│                                                                    │
└─ 高表现识别 ───────────────────────────────────────────────────────┤
  互动率 > 均值+1σ → high_performer                                  │
  完播率 > 70%    → quality_content                                  │
  → high_performers: [CTN-xxx, ...]                                  │
```

**输出数据**:
- `AnalyticsReport`（部分字段）:
  - `metrics_summary`: 指标汇总
  - `horizontal_comparison`: 平台间横向对比
  - `vertical_comparison`: 时间纵向对比
  - `hotspot_analysis.high_performers`: 高表现内容列表

**关键接口点**:
- 上游: M-20 的 `analytics_reports` 表
- 下游: M-24 汇总分析结果；M-22/M-23 提供补充分析

### M-22 — 评论区语义挖掘与隐私脱敏 (Comment Semantic Mining & Privacy Sanitization)

| 属性 | 说明 |
|------|------|
| **角色** | 从用户评论中提取情感倾向、话题热点和用户反馈，同时保护用户隐私 |
| **优先级** | P1 |
| **触发方式** | 事件驱动（新评论到达时）或定时批量处理 |

**输入数据**:
- 各平台评论数据（通过 M-18 效果数据采集关联获取）
- 评论内容: `comment_text`, `comment_time`, `user_id`（脱敏前）, `platform`

**NLP 处理管道**:

```
原始评论文本 ──────────────────────────────────────────► CommentInsight
│                                                              │
├─ Step 1: 文本预处理 ────────────────────────────────────────┤
│  - 去除 HTML 标签、表情符号规范化                            │
│  - 中文分词（jieba）                                        │
│  - 停用词过滤                                               │
│                                                              │
├─ Step 2: 情感分析 ──────────────────────────────────────────┤
│  - 正面/负面/中性 三分类                                     │
│  - 情感强度评分 (0-1)                                       │
│  → sentiment: {label, score}                                │
│                                                              │
├─ Step 3: 关键词提取 ────────────────────────────────────────┤
│  - TF-IDF 关键词提取                                        │
│  - TextRank 关键短语提取                                    │
│  → keywords: [{word, weight}]                               │
│                                                              │
├─ Step 4: 主题聚类 ──────────────────────────────────────────┤
│  - LDA 主题模型                                             │
│  - 聚类主题标签生成                                         │
│  → topics: [{topic_id, label, keywords, comment_count}]     │
│                                                              │
├─ Step 5: 隐私脱敏 ──────────────────────────────────────────┤
│  - 手机号 → 138****1234                                     │
│  - 身份证 → 320***********1234                              │
│  - 邮箱   → us***@example.com                               │
│  - 实名   → 张**                                            │
│  - IP地址 → 192.168.***.***                                 │
│  → anonymized_text                                          │
│                                                              │
└─ Step 6: 用户意图识别 ──────────────────────────────────────┤
   - 提问/建议/投诉/赞赏 分类                                  │
   - 高频需求提取                                             │
   → user_intents: [{type, content, frequency}]               │
```

**输出数据**:
- `CommentInsight` 结构（写入 `comment_analysis` 表）:
  - `content_id`: 关联内容ID
  - `platform`: 平台
  - `sentiment`: 情感分析结果
  - `keywords`: 提取关键词
  - `topics`: 主题聚类
  - `anonymized_text`: 脱敏后文本
  - `user_intents`: 用户意图
  - `analyzed_at`: 分析时间

**关键接口点**:
- 上游: M-18 的评论数据
- 下游: M-24 汇总评论洞察

### M-23 — 跨平台热点追踪 (Cross-Platform Hotspot Tracking)

| 属性 | 说明 |
|------|------|
| **角色** | 跨平台监测和追踪热点话题，为选题提供趋势依据 |
| **优先级** | P2 |
| **触发方式** | 定时任务（每 4 小时）或事件驱动 |

**输入数据**:
- 各平台热搜榜 API 数据
- 行业 RSS/新闻源
- M-04 竞品采集数据（如有）
- M-22 评论中挖掘的潜在热点

**热点追踪流程**:

```
多源数据 ──────────────────────────────────────────────► HotTopicList
│                                                              │
├─ 数据源 ────────────────────────────────────────────────────┤
│  - 抖音热搜榜 API                                           │
│  - 小红书热门话题 API                                       │
│  - B站热门/每周必看 API                                     │
│  - X(Twitter) Trending API                                  │
│  - 行业资讯 RSS                                             │
│                                                              │
├─ 热点识别 ──────────────────────────────────────────────────┤
│  - 关键词频次统计                                           │
│  - 多平台共现检测（同一话题在多个平台出现 → 跨平台热点）    │
│  - 话题生命周期判断（上升期/高峰期/衰退期）                 │
│  → topics: [{topic, platform, phase, first_seen}]           │
│                                                              │
├─ 热度评分 ──────────────────────────────────────────────────┤
│  heat_score = w1×搜索量 + w2×讨论量 + w3×增长速度           │
│             + w4×跨平台系数 + w5×时效衰减                   │
│  → heat_score: 0-100                                        │
│                                                              │
├─ 趋势预测 ──────────────────────────────────────────────────┤
│  - 时间序列分析（ARIMA）                                    │
│  - 预测未来 24/48/72 小时热度走势                           │
│  → trend: rising / stable / declining                       │
│  → predicted_peak_at: ISO 8601                              │
│                                                              │
└─ 相关性匹配 ────────────────────────────────────────────────┤
   - 热点与账号历史内容主题的匹配度                            │
   - 推荐可追热点列表                                         │
   → recommended: [{topic, relevance_score, urgency}]         │
```

**输出数据**:
- `HotTopicList` 结构（写入 `hot_topics` 表）:
  - `topics`: 热点话题列表
  - `heat_scores`: 热度评分
  - `trends`: 趋势预测
  - `recommended`: 推荐追热点列表
  - `tracked_at`: 追踪时间

**关键接口点**:
- 上游: 平台 API、外部数据源
- 下游: M-24 汇总热点信息

### M-24 — 下期计划草案生成 (Next-Cycle Plan Generation)

| 属性 | 说明 |
|------|------|
| **角色** | 数据分析闭环的**终点模块**，综合所有分析结果生成可执行的下期计划 |
| **优先级** | P2 |
| **触发方式** | 所有上游分析模块（M-21/M-22/M-23）完成后自动触发 |

**输入数据**:
- `AnalyticsReport`（来自 M-21 多维度效果分析）
- `CommentInsight`（来自 M-22 评论区语义挖掘）
- `HotTopicList`（来自 M-23 跨平台热点追踪）
- `preference_rules`（来自 M-29 学习成果应用）

**计划生成流程**:

```
综合分析输入 ─────────────────────────────────────────────► PlanDraft
│                                                              │
├─ 1. 选题推荐引擎 ───────────────────────────────────────────┤
│  综合评分 = α×热点热度 + β×历史表现 + γ×用户偏好           │
│           + δ×竞争程度 + ε×内容匹配度                       │
│  → topic_suggestions: [{topic, score, reasoning}]           │
│                                                              │
├─ 2. 发布时间策略 ──────────────────────────────────────────┤
│  基于最佳发布时间窗口 + 热点时效性                           │
│  → schedule: [{content_type, platform, suggested_time}]     │
│                                                              │
├─ 3. 内容方向建议 ──────────────────────────────────────────┤
│  高互动主题/风格推荐 + 受众偏好反馈                         │
│  → content_direction: [{style, topics, expected_impact}]    │
│                                                              │
├─ 4. 平台权重调整 ──────────────────────────────────────────┤
│  基于各平台近期表现 + 账号健康度                             │
│  → platform_allocation: {douyin: 0.4, xiaohongshu: 0.3, ...}│
│                                                              │
├─ 5. 风险预警 ──────────────────────────────────────────────┤
│  账号健康趋势 + 合规风险 + 热点风险                         │
│  → risk_alerts: [{type, severity, description, mitigation}] │
│                                                              │
└─ 6. 资源分配 ──────────────────────────────────────────────┤
   各平台内容数量/类型配比 + 素材需求预估                     │
   → resource_plan: {total_posts, per_platform, material_needs}│
```

**输出数据**:
- `PlanDraft` 结构（写入 `plan_drafts` 表）:
  - `topic_suggestions`: 选题建议列表
  - `schedule`: 发布时间策略
  - `content_direction`: 内容方向建议
  - `platform_allocation`: 平台权重分配
  - `risk_alerts`: 风险预警
  - `resource_plan`: 资源分配计划
  - `generated_at`: 生成时间

**关键接口点**:
- 上游: M-21/M-22/M-23 的综合分析结果
- 下游: 反馈至 M-01 关键词扩展，触发新一轮素材采集闭环

## 3. 调试顺序

数据分析闭环的调试应严格按以下顺序进行，确保上游模块数据就绪后再调试下游：

```
调试阶段 1: M-20 数据清洗与聚合
  ├── 前置条件: M-18 已有 performance_metrics 数据, M-19 已有 account_health 数据
  ├── 验证点:
  │   ├── 缺失值检测逻辑是否正确（人为制造缺失数据验证）
  │   ├── IQR 异常值检测阈值是否合理（查看 outlier_flags 输出）
  │   ├── 维度聚合结果是否完整（检查 agg_views 各维度）
  │   └── 基准对比偏差计算是否正确
  └── 成功标准: analytics_reports 表中有清洗后的数据记录

调试阶段 2: M-21 多维度效果分析
  ├── 前置条件: M-20 已产出 CleanedDataset
  ├── 验证点:
  │   ├── 六大分析维度是否均有输出
  │   ├── 横向对比数据与原始数据一致性
  │   ├── 纵向对比 change_rate 计算正确
  │   └── high_performers 识别逻辑合理
  └── 成功标准: AnalyticsReport 中 metrics_summary 和 comparisons 字段完整

调试阶段 3: M-22 评论区语义挖掘与隐私脱敏
  ├── 前置条件: 有评论数据可供分析
  ├── 验证点:
  │   ├── 情感分析准确率（抽样人工校验）
  │   ├── 关键词提取是否过滤了无意义词
  │   ├── 隐私脱敏是否完整（手机号/邮箱/身份证/实名）
  │   └── 脱敏后文本可读性
  └── 成功标准: comment_analysis 表记录完整且隐私信息已脱敏

调试阶段 4: M-23 跨平台热点追踪
  ├── 前置条件: 平台 API 可正常访问
  ├── 验证点:
  │   ├── 各平台热点数据是否成功获取
  │   ├── 跨平台共现检测是否准确
  │   ├── 热度评分公式权重是否合理
  │   └── 趋势预测方向与实际走势对比
  └── 成功标准: hot_topics 表有各平台热点记录

调试阶段 5: M-24 下期计划草案生成
  ├── 前置条件: M-21/M-22/M-23 均已产出结果
  ├── 验证点:
  │   ├── 选题推荐排序是否合理（检查综合评分各因子权重）
  │   ├── 发布时间策略是否基于最佳窗口数据
  │   ├── 风险预警是否覆盖关键风险维度
  │   └── 计划草案是否可被 M-01 消费（格式兼容性）
  └── 成功标准: plan_drafts 表记录完整且可触发新一轮采集闭环
```

## 4. 验收标准

### 4.1 数据质量

| 验收项 | 标准 | 验证方式 |
|--------|------|---------|
| 缺失值处理率 | 100% 缺失值被标记或填充 | 检查 outlier_flags 中 incomplete 标记 |
| 异常值检出率 | IQR 方法检出率 > 95% | 人工注入已知异常值验证 |
| 数据聚合完整性 | 所有维度聚合无遗漏 | 检查 agg_views 覆盖全部预期维度 |
| 隐私脱敏覆盖率 | 100% PII 数据被脱敏 | 正则扫描脱敏后文本 |

### 4.2 分析质量

| 验收项 | 标准 | 验证方式 |
|--------|------|---------|
| 情感分析准确率 | > 85% | 抽样 200 条评论人工校验 |
| 热点识别召回率 | > 80%（与已知热点对比） | 对比平台官方热搜榜 |
| 选题推荐相关性 | 推荐选题中 > 60% 与账号定位匹配 | 人工评审 |
| 计划可执行性 | 100% 计划项有明确执行步骤 | 格式校验 |

### 4.3 系统性能

| 验收项 | 标准 | 验证方式 |
|--------|------|---------|
| M-20 清洗耗时 | < 60s（10000条记录） | 性能测试 |
| M-21 分析耗时 | < 120s | 性能测试 |
| M-22 评论处理 | < 5s/100条 | 性能测试 |
| M-23 热点追踪 | < 30s（4平台） | 性能测试 |
| M-24 计划生成 | < 30s | 性能测试 |
