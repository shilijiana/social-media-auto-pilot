# FLOW_05: 自学习闭环

> 闭环路径: M-25 → M-26 → M-27 → M-28 → M-29 → M-30
> 闭环目标: 从运营数据中持续学习，提炼用户偏好和运营规则，管理规则生命周期，并将学习成果反馈至内容生产和分发环节
> 目标读者: Hermes Agent

---

### 3.1 状态流转图

### 3.2 状态转换条件

| 转换 | 触发条件 | 自动化 |
|------|---------|--------|
| created → active | confidence >= 50 且通过验证期（至少3次应用） | 半自动（需人工确认） |
| created → deleted | confidence < 30 或人工否决 | 自动 |
| active → degraded | confidence < 40 或连续3次应用成功率 < 30% | 自动 |
| active → paused | 30天内无应用记录 | 自动 |
| active → deleted | 人工主动删除 | 手动 |
| degraded → active | 新数据支持，confidence 回升至 >= 50 | 自动 |
| degraded → deleted | confidence < 20 持续 7 天 | 自动 |
| paused → active | 人工恢复 | 手动 |
| paused → deleted | paused 状态超过 90 天 | 自动 |

### 3.3 置信度衰减公式

```
confidence(t) = confidence(0) × e^(-λt)

其中:
  λ = 衰减系数（默认 0.01/天）
  t = 距离上次验证的天数

每次验证通过:
  confidence_new = confidence_old × (1 - α) + α × 验证评分
  
  其中 α = 学习率（默认 0.3）
```

## 4. 调试顺序

自学习闭环的调试应严格按以下顺序进行：

```
调试阶段 1: M-25 学习数据收集
  ├── 前置条件: M-13 有审阅记录, M-18 有效果数据
  ├── 验证点:
  │   ├── 审阅事件是否全部被收集（对比 M-13 审阅任务数与 M-25 记录数）
  │   ├── 六大分类归档是否正确（抽样检查标签准确性）
  │   ├── 事件数据 event_data 是否完整（必填字段检查）
  │   └── 学习记录 ID 格式是否符合规范
  └── 成功标准: learning_records 表中记录数与审阅事件数一致

调试阶段 2: M-26 偏好提炼与规则生成
  ├── 前置条件: M-25 有学习记录
  ├── 验证点:
  │   ├── 置信度计算公式是否正确（用已知数据手工验算）
  │   ├── 三大偏好库是否均有规则产出
  │   ├── 规则内容是否可读（自然语言描述是否准确）
  │   └── 重复规则是否被去重
  └── 成功标准: preference_rules 表中有新规则生成

调试阶段 3: M-27 规则时效性管理
  ├── 前置条件: M-26 有规则数据
  ├── 验证点:
  │   ├── 置信度衰减是否按公式执行
  │   ├── 自动降级条件是否正确触发（模拟低置信度场景）
  │   ├── 自动删除是否正确执行（模拟极低置信度场景）
  │   └── 状态转换日志是否完整记录
  └── 成功标准: 过期规则被正确降级/暂停/删除

调试阶段 4: M-28 初始偏好模板导入
  ├── 前置条件: 有模板文件或内置模板
  ├── 验证点:
  │   ├── 模板解析是否正确（JSON/CSV 格式校验）
  │   ├── 导入的规则 confidence 是否设为 50
  │   ├── human_labeled 是否设为 true
  │   └── 重复导入是否被幂等处理
  └── 成功标准: preference_rules 表中有模板规则记录

调试阶段 5: M-29 学习成果应用
  ├── 前置条件: M-26/M-28 有 active 状态规则
  ├── 验证点:
  │   ├── 规则是否按优先级正确分发
  │   ├── 各目标模块是否正确接收规则注入
  │   ├── LLM prompt 是否正确拼接偏好规则
  │   └── 规则应用是否可追溯（applied_count 递增）
  └── 成功标准: 各业务模块能读取并应用偏好规则

调试阶段 6: M-30 学习成效量化与报告
  ├── 前置条件: M-25~M-29 有完整运行数据
  ├── 验证点:
  │   ├── 各项评估指标计算是否正确
  │   ├── 异常检测是否覆盖全部 5 种异常类型
  │   ├── Top10 置信度规则是否准确
  │   └── 报告格式是否符合 LearningReport Schema
  └── 成功标准: learning_reports 表有完整的周期性报告
```

## 5. 验收标准

### 5.1 数据完整性

| 验收项 | 标准 | 验证方式 |
|--------|------|---------|
| 学习记录覆盖率 | 100% 审阅事件被记录 | 对比 M-13 与 M-25 记录数 |
| 分类准确率 | > 95% 记录分类正确 | 抽样 100 条人工校验 |
| 规则去重率 | 重复规则不生成 | 规则内容相似度检测 |

### 5.2 学习质量

| 验收项 | 标准 | 验证方式 |
|--------|------|---------|
| 人工修改率下降 | 连续4周呈下降趋势 | 趋势分析 |
| 一次通过率提升 | 连续4周呈上升趋势 | 趋势分析 |
| 自动通过率提升 | 目标 > 40% | 统计计算 |
| 规则命中率 | > 60% 内容至少命中1条规则 | 规则应用日志 |

### 5.3 规则生命周期

| 验收项 | 标准 | 验证方式 |
|--------|------|---------|
| 置信度衰减 | 每天按公式衰减 | 查询规则表 confidence 字段变化 |
| 自动降级 | 条件触发后 1h 内执行 | 模拟低置信度场景 |
| 自动删除 | degraded+低置信度 7天后执行 | 时间模拟 |
| 状态日志 | 每次状态变更均有日志 | 检查 rule_lifecycle_log 表 |

### 5.4 系统性能

| 验收项 | 标准 | 验证方式 |
|--------|------|---------|
| M-25 数据收集 | < 10s（100条审阅记录） | 性能测试 |
| M-26 规则生成 | < 60s | 性能测试 |
| M-27 时效检查 | < 30s（全量规则） | 性能测试 |
| M-28 模板导入 | < 5s（100条规则模板） | 性能测试 |
| M-29 规则分发 | < 10s | 性能测试 |
| M-30 报告生成 | < 30s | 性能测试 |
